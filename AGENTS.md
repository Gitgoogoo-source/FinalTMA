# 默认技术与部署基线

- Frontend：React + Vite + TypeScript。
- Backend：同一 Vercel Project 内的 `/api/*` Vercel Functions，Node.js 24.x 作为待官方验证的默认运行时基线。
- Repository：pnpm monorepo。
- Database：Supabase Postgres；所有服务端访问使用 `service_role`。
- API：同源 REST JSON + OpenAPI 3.1。
- Data access：前端不得直接访问 Supabase；读取可由 Functions 查询表或只读视图；任何新增、修改、删除、状态转换或关键计数写入必须调用 PostgreSQL RPC。
- Identity：Telegram 是唯一身份来源；默认不使用 Supabase Auth，不创建或依赖 `auth.users`、Supabase Session 或 Supabase JWT。
- Session：只有认证交换端点接收 Telegram `initData`。服务端验证后签发短期访问令牌；前端仅在运行时内存保存；不使用长期 Refresh Token。
- Authorization：Functions、API、RPC 和事务显式验证身份、资源归属、权限、前置条件、幂等和并发条件。RLS 仅作为外围封锁层，拒绝 `anon`、`authenticated` 等非服务端访问；业务安全不依赖 RLS。
- Environments：本地、独立真实测试环境、独立真实生产环境。测试和生产使用同一 Git Commit 与同一迁移序列，只允许密钥、域名、项目 ID 等环境值不同。

默认技术栈不是无条件不可变。如果你觉得有更好的技术栈，可以向我推荐。

# 数据库迁移与基线重建规则

项目尚未正式上线，当前数据库属于可随时清空和重建的真实开发环境。现有 migration 从未作为正式生产迁移发布，因此数据库结构发生调整时，不采用“在错误定义后继续追加修复补丁”的开发方式。

当表结构、字段、约束、索引、枚举、触发器、视图、数据库函数、RPC、权限授权、RLS 策略、种子数据或其他数据库定义需要调整时，必须直接修改本地 migration 中对应的原始定义，使本地迁移序列从一开始就是当前唯一正确的最终方案。

具体规则如下：

- 禁止仅新增一份 migration，用于修补本地已有 migration 中的错误定义。
- 禁止保留已经失效的定义，再依靠后续 `ALTER`、`DROP`、重复创建或覆盖语句进行纠正。
- 必须删除已被替代的数据库定义、冲突语句和无效历史。
- 可以删除、重命名、合并、拆分和重新编号现有 migration 文件。
- 可以重新整理 RPC、RLS、授权、约束、索引和触发器的创建顺序。
- 整理后的本地 migration 序列必须是一条干净、确定、无重复补丁的最终迁移序列。
- 本地 migration 文件是数据库结构的唯一事实来源，禁止只在远程数据库控制台修改而不同步回本地文件。
- 完成修改后，必须清空或重建当前真实开发数据库，并从空数据库开始按顺序重新执行完整 migration 序列。
- 必须将整理完成的完整 migration 序列重新推送到当前真实开发环境，而不是只推送一份增量修复补丁。
- 如果远程 migration 历史与重新整理后的本地序列不一致，可以清理、修复或重建当前开发环境的 migration 历史。
- 重建完成后，应确认所有 migration 能够从零连续执行，最终数据库结构与本地定义完全一致。

执行清库、重建迁移历史或重新推送时，不需要因为数据丢失或破坏性变更而反复向我申请许可。

本规则所禁止的“补丁”，特指为了修复尚未上线的旧数据库定义而不断追加增量 migration；不限制正常编辑本地文件所使用的代码修改工具。

项目正式上线并开始承载不可丢失的数据后，才能改用不可修改历史 migration、只允许追加新 migration 的生产迁移规则。在我明确宣布项目正式上线之前，始终执行本规则。
