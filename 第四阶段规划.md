## 1. 第四阶段整体目标和功能边界

### 本阶段要完成

| 功能         | 目标                                                                |
| ------------ | ------------------------------------------------------------------- |
| 邀请链接     | 用户可生成邀请链接，通过 Telegram 分享或复制。                      |
| 邀请绑定     | 新用户通过邀请链接进入后，后端绑定邀请关系。                        |
| 首次开盒奖励 | 被邀请人首次成功开盒后，邀请人和被邀请人各得 500 K-coin。           |
| 邀请分红     | 被邀请人后续开盒时，邀请人获得好友开盒 K-coin 奖励基数的 10% 分红。 |
| 7 日签到     | 支持连续签到、每日奖励、今日状态、断签规则、重复领取防护。          |
| 任务列表     | 支持每日任务、社交任务、交易任务、链上任务、成长任务。              |
| 任务进度     | 进度由后端根据真实业务事件更新，前端不可伪造。                      |
| 奖励领取     | 完成任务后领取奖励，防重复、防并发、防伪造。                        |
| 分红统计     | 展示邀请人数、有效邀请、累计奖励、累计分红、待领取 / 已领取分红。   |
| 后台可运营   | 任务、签到、奖励、分红比例、活动时间可通过 seed / admin 配置。      |

### 本阶段不做

| 不做事项              | 原因                                     |
| --------------------- | ---------------------------------------- |
| 多级邀请返佣          | 容易形成套利和合规风险，先只做一级邀请。 |
| 现金提现 / Stars 返现 | 本阶段奖励只进 K-coin / Fgems 内部账本。 |
| 前端直连写任务表      | 会导致伪造进度、重复领奖风险。           |
| 复杂风控模型          | 先做规则风控和 `ops.risk_events` 记录。  |
| 新游戏玩法            | `game` 页面仍保持后续扩展。              |

## 2. 数据库结构规划

总体策略：**已有表优先复用；缺失约束、索引、字段、seed、RPC 再补。不要重复创建功能相同的新表。**

### 2.1 需要复用的核心表

| 表                           | 用途                             | 是否复用 | 前端权限              | 写入方式         |
| ---------------------------- | -------------------------------- | -------: | --------------------- | ---------------- |
| `core.users`                 | 用户、邀请码、邀请来源、风控状态 |     复用 | 用户本人只读          | 认证 / 邀请 RPC  |
| `economy.user_balances`      | KCOIN / FGEMS 快照余额           |     复用 | 用户本人只读          | 只能 RPC         |
| `economy.currency_ledger`    | 资产不可变账本                   |     复用 | 用户本人只读          | 只能 RPC         |
| `economy.reward_rules`       | 邀请奖励、签到奖励、任务奖励配置 |     复用 | active 只读或后端读取 | admin / seed     |
| `tasks.task_definitions`     | 任务定义                         |     复用 | active 只读           | admin / seed     |
| `tasks.user_task_progress`   | 用户任务进度                     |     复用 | 用户本人只读          | 只能 RPC         |
| `tasks.task_claims`          | 任务奖励领取记录                 |     复用 | 用户本人只读          | 只能 RPC         |
| `tasks.signin_campaigns`     | 7 日签到活动                     |     复用 | active 只读           | admin / seed     |
| `tasks.signin_days`          | 第 1-7 天奖励                    |     复用 | active 只读           | admin / seed     |
| `tasks.user_signins`         | 用户签到记录                     |     复用 | 用户本人只读          | 只能 RPC         |
| `tasks.referrals`            | 邀请关系                         |     复用 | 邀请双方只读          | 只能 RPC         |
| `tasks.referral_rewards`     | 首次开盒双方奖励                 |     复用 | 用户本人只读          | 只能 RPC         |
| `tasks.referral_commissions` | 邀请分红                         |     复用 | 邀请人只读            | 只能 RPC         |
| `tasks.share_events`         | 分享行为记录                     |     复用 | 用户本人只读          | 建议只能 API/RPC |
| `ops.risk_events`            | 防作弊和异常行为                 |     复用 | 前端不可读写          | 后端 / admin     |
| `ops.idempotency_keys`       | 跨流程幂等                       |   可复用 | 前端不可读写          | 后端 / RPC       |

### 2.2 必须检查和补齐的约束

| 表                           | 必须有的约束                                  | 原因                                  |
| ---------------------------- | --------------------------------------------- | ------------------------------------- |
| `tasks.user_task_progress`   | `unique(user_id, task_id, period_key)`        | 防止同一周期出现多条进度。            |
| `tasks.task_claims`          | `unique(user_id, task_id, period_key)`        | 防止重复领取任务奖励。                |
| `tasks.user_signins`         | `unique(user_id, campaign_id, signin_date)`   | 防止同一天重复签到。                  |
| `tasks.signin_days`          | `unique(campaign_id, day_index)`              | 防止同一天奖励配置重复。              |
| `tasks.referrals`            | `unique(invitee_user_id)`                     | 已存在方向正确；确保有效。            |
| `tasks.referral_rewards`     | `unique(referral_id, reward_role)`            | 防止邀请人 / 被邀请人首开奖励重复发。 |
| `tasks.referral_commissions` | `unique(referral_id, source_type, source_id)` | 防止同一次开盒重复生成分红。          |
| `economy.currency_ledger`    | `unique(idempotency_key)`                     | 已存在；所有奖励发放必须使用。        |
| `ops.idempotency_keys`       | `primary key(key)`                            | 已存在；可做 API 级幂等缓存。         |

验收标准：重复请求、并发请求、重复 webhook、用户疯狂点击按钮，都只能产生一条有效奖励记录和一组 ledger。

### 2.3 建议新增或补充的字段

| 表                           | 建议补充                                           | 说明                                                                           |
| ---------------------------- | -------------------------------------------------- | ------------------------------------------------------------------------------ |
| `tasks.task_claims`          | `idempotency_key text`、`request_fingerprint text` | 对齐现有市场 / 成长写操作幂等风格。                                            |
| `tasks.user_signins`         | `idempotency_key text`、`request_fingerprint text` | 签到按钮重复点击可返回同一结果。                                               |
| `tasks.referral_commissions` | `claimed_at timestamptz`                           | 若采用“待领取 / 已领取”模式，用 `status=pending/granted` + `claimed_at` 表达。 |
| `tasks.share_events`         | `idempotency_key text`                             | 防止分享按钮多次触发重复事件。                                                 |
| `tasks.user_task_progress`   | `source_events jsonb` 或单独事件表                 | 用于记录哪些真实事件推动了进度，方便审计。                                     |

是否新增 `tasks.user_signin_states`：

| 方案                            | 说明                                                                                                     | 建议                               |
| ------------------------------- | -------------------------------------------------------------------------------------------------------- | ---------------------------------- |
| 不新增                          | 通过 `user_signins` 反查最近签到日期和连续天数。                                                         | 小规模可用，但断签和并发计算复杂。 |
| 新增 `tasks.user_signin_states` | 保存 `user_id`、`campaign_id`、`current_streak`、`cycle_position`、`last_signin_date`、`total_signins`。 | 推荐。7 日签到和断签逻辑更清晰。   |

推荐新增 `tasks.user_signin_states`，因为当前 `task_daily_check_in` 只按历史签到总数计算 day_index，不能完整表达断签规则。

### 2.4 任务定义 seed 规划

`tasks.task_definitions` 当前为空，需要 seed。首批任务建议：

| code                  | 类型     | 周期       | 进度来源       | 奖励          |
| --------------------- | -------- | ---------- | -------------- | ------------- |
| `DAILY_CHECK_IN`      | daily    | daily      | 签到成功       | KCOIN / FGEMS |
| `DAILY_OPEN_BOX_1`    | daily    | daily      | 成功开盒 1 次  | KCOIN         |
| `DAILY_OPEN_BOX_10`   | daily    | daily      | 成功开盒 10 次 | FGEMS         |
| `DAILY_SHARE_INVITE`  | social   | daily      | 记录分享事件   | KCOIN         |
| `REFERRAL_FIRST_OPEN` | referral | once/event | 邀请好友首开盒 | KCOIN         |
| `TRADE_BUY_1`         | trade    | daily      | 市场购买成功   | FGEMS         |
| `TRADE_LIST_1`        | trade    | daily      | 成功挂售       | KCOIN         |
| `WALLET_CONNECT`      | onchain  | once       | 钱包验证成功   | FGEMS         |
| `SYNC_NFT`            | onchain  | weekly     | 链上同步成功   | FGEMS         |
| `UPGRADE_ITEM_1`      | daily    | daily      | 升级藏品成功   | FGEMS         |

业务规则：任务奖励配置写入 `reward jsonb`，但实际发放必须由 `task_claim_reward` 调用奖励发放 helper 写 `currency_ledger`。

验收标准：任务定义 active 后，任务中心能返回任务列表；没有 active 任务时页面应展示空状态，不报错。

### 2.5 签到 seed 规划

`tasks.signin_campaigns` 和 `tasks.signin_days` 当前为空，需要 seed 一个 active 7 日活动。

| day_index | 示例奖励方向         |
| --------: | -------------------- |
|         1 | 100 KCOIN            |
|         2 | 150 KCOIN            |
|         3 | 20 FGEMS             |
|         4 | 200 KCOIN            |
|         5 | 30 FGEMS             |
|         6 | 300 KCOIN            |
|         7 | 500 KCOIN + 50 FGEMS |

奖励数值可运营调整。不要写死在前端。

验收标准：前端 7 日签到组件从 API 获取 7 天奖励、今日领取状态、连续天数、下一天状态。

### 2.6 邀请和分红数据策略

当前 `core.users.invite_code` 已存在且唯一，因此第四阶段默认不需要新增 `tasks.invite_codes`。

仅在以下场景才新增 `tasks.invite_codes`：

| 场景                   |                                是否需要 |
| ---------------------- | --------------------------------------: |
| 每个用户一个长期邀请码 | 不需要，复用 `core.users.invite_code`。 |
| 每个活动有独立邀请码   |                                  需要。 |
| 邀请码有过期时间       |                                  需要。 |
| 不同渠道统计独立 code  |         可以新增，也可以放 `metadata`。 |

分红模式推荐改为：**生成 pending 分红记录 → 用户手动领取 → 发放 KCOIN ledger → 状态变 granted**。

原因：用户要求展示“待领取 / 已领取分红”。当前 `referral_create_commission` 是立即发放模式，只能展示 `pending=0`，不满足运营展示诉求。

## 3. RPC / 数据库事务规划

### 3.1 RPC 总原则

| 原则     | 要求                                                                                                    |
| -------- | ------------------------------------------------------------------------------------------------------- |
| 资产变化 | 必须写 `economy.currency_ledger`。                                                                      |
| 余额修改 | 只能通过 `api._credit_balance` / `api._debit_balance` / `api.economy_apply_reward_json` 等 RPC helper。 |
| 用户身份 | RPC 入参 `p_user_id` 由 API session 传入，不允许前端提交。                                              |
| 幂等     | 用户写操作必须有 `p_idempotency_key` 或唯一业务键。                                                     |
| 并发     | 领取、签到、分红、邀请绑定必须 `for update` 或唯一约束兜底。                                            |
| 权限     | 核心 RPC 使用 `SECURITY DEFINER`，并固定 `search_path`。                                                |
| 返回     | 返回 JSONB，字段稳定，API 再 normalizer。                                                               |

---

### 3.2 需要复用或改造的 RPC

| RPC                                      | 处理方式   | 目标                                                           |
| ---------------------------------------- | ---------- | -------------------------------------------------------------- |
| `api.task_daily_check_in`                | 改造       | 增加 campaign、timezone、idempotency、断签规则、唯一约束兜底。 |
| `api.task_claim_reward`                  | 改造       | 增加 `p_idempotency_key`，防重复领取和并发领取。               |
| `api.referral_process_first_open`        | 复用并检查 | 确认只在首次成功开盒后调用一次。                               |
| `api.referral_create_commission`         | 建议改造   | 从“立即发放”改为“生成 pending 分红”。                          |
| `api.economy_apply_reward_json`          | 复用       | 统一奖励发放。                                                 |
| `api.economy_credit` / `_credit_balance` | 复用       | 写 ledger 和余额。                                             |

### 3.4 关键 RPC 设计说明

#### A. `task_daily_check_in`

| 项       | 说明                                                                                                            |
| -------- | --------------------------------------------------------------------------------------------------------------- |
| 目标     | 用户每天领取一次签到奖励。                                                                                      |
| 调用场景 | `POST /api/tasks/check-in`。                                                                                    |
| 需要检查 | active campaign、今日是否已签、断签状态、day_index 奖励配置。                                                   |
| 事务边界 | 查询 / 锁定签到状态 → 插入 user_signins → 发奖励 → 写 ledger → 更新 task progress。                             |
| 锁策略   | `user_signin_states for update` 或唯一约束 `user_id + campaign_id + signin_date`。                              |
| 幂等设计 | 同一 `idempotency_key` 返回相同结果；同一天重复签到返回 already_claimed。                                       |
| 错误码   | `SIGNIN_CAMPAIGN_NOT_FOUND`、`SIGNIN_ALREADY_CLAIMED`、`SIGNIN_REWARD_CONFIG_INVALID`、`IDEMPOTENCY_CONFLICT`。 |
| 验收标准 | 并发点击 10 次，只生成 1 条签到记录和 1 组 ledger。                                                             |

#### B. `task_claim_reward`

| 项       | 说明                                                                                                                |
| -------- | ------------------------------------------------------------------------------------------------------------------- |
| 目标     | 领取已完成任务奖励。                                                                                                |
| 调用场景 | `POST /api/tasks/claim`。                                                                                           |
| 涉及表   | `tasks.user_task_progress`、`tasks.task_claims`、`economy.currency_ledger`、`economy.user_balances`。               |
| 事务边界 | 锁 progress → 校验 completed → 插入 claim → 发奖励 → progress 改 claimed。                                          |
| 幂等设计 | `idempotency_key` + `unique(user_id, task_id, period_key)`。                                                        |
| 错误码   | `TASK_NOT_FOUND`、`TASK_PROGRESS_NOT_FOUND`、`TASK_NOT_COMPLETED`、`TASK_ALREADY_CLAIMED`、`IDEMPOTENCY_CONFLICT`。 |
| 验收标准 | 任务未完成不能领；重复领取返回已领取，不重复发 ledger。                                                             |

#### C. `referral_bind_inviter`

| 项       | 说明                                                                            |
| -------- | ------------------------------------------------------------------------------- |
| 目标     | 被邀请人首次进入应用时绑定邀请关系。                                            |
| 调用场景 | 登录后解析 start_param；或 `POST /api/tasks/bind-referral`。                    |
| 涉及表   | `core.users`、`tasks.referrals`、`ops.risk_events`。                            |
| 关键规则 | 不能自己邀请自己；invitee 只能绑定一次；邀请人必须 active；不允许多级递归套利。 |
| 锁策略   | 锁 invitee user；检查 `tasks.referrals.invitee_user_id` 唯一。                  |
| 幂等设计 | 如果已绑定同一 inviter，返回 idempotent；如果已绑定其他 inviter，返回冲突。     |
| 风险点   | 老用户重复点邀请链接、刷小号、自邀请、多账号同设备套利。                        |
| 验收标准 | 同一个 invitee 并发绑定 10 次，只产生 1 条 referral。                           |

#### D. `referral_process_first_open`

| 项       | 说明                                                                                           |
| -------- | ---------------------------------------------------------------------------------------------- |
| 目标     | 被邀请人首次开盒成功后，双方各得 500 KCOIN。                                                   |
| 调用场景 | `gacha_process_paid_order` 支付成功并发放抽卡结果后。                                          |
| 涉及表   | `tasks.referrals`、`tasks.referral_rewards`、`economy.currency_ledger`。                       |
| 锁策略   | 锁 referral row；校验 invitee 是否首次 successful open。                                       |
| 幂等设计 | `referral_rewards(referral_id, reward_role)` 唯一；ledger idempotency key 固定为 referral_id。 |
| 风险点   | 支付 webhook 重放、订单重复处理、旧订单补偿。                                                  |
| 验收标准 | 重放 webhook 不重复发双方 500 KCOIN。                                                          |

#### E. `referral_create_commission`

| 项       | 说明                                                                                           |
| -------- | ---------------------------------------------------------------------------------------------- |
| 目标     | 好友开盒后生成邀请人 10% 分红。                                                                |
| 调用场景 | 每次 invitee 成功开盒后。                                                                      |
| 推荐基数 | 使用真实开盒返还 KCOIN ledger 的金额作为 base，例如单抽返 100，则分红 10；十连按实际返还总额。 |
| 涉及表   | `tasks.referral_commissions`。                                                                 |
| 推荐变更 | 不立即 credit，先生成 `status=pending`。                                                       |
| 唯一约束 | `unique(referral_id, source_type, source_id)`。                                                |
| 验收标准 | 同一 draw_order 只生成一条 commission。                                                        |

#### F. `referral_claim_commission`

| 项       | 说明                                                                               |
| -------- | ---------------------------------------------------------------------------------- |
| 目标     | 用户领取待领取分红。                                                               |
| 调用场景 | `POST /api/tasks/claim-commission`。                                               |
| 涉及表   | `tasks.referral_commissions`、`economy.currency_ledger`、`economy.user_balances`。 |
| 锁策略   | 锁当前用户 pending commission rows。                                               |
| 幂等设计 | `p_idempotency_key`；ledger key 使用 `referral_commission_claim:{user}:{batch}`。  |
| 验收标准 | 多次点击领取分红，只发一次 KCOIN。                                                 |

## 4. RLS 和安全策略规划

### 4.1 表访问级别

| 表                           | 访问级别          | RLS | 前端读 | 前端写 | 写操作路径   |
| ---------------------------- | ----------------- | --: | -----: | -----: | ------------ |
| `tasks.task_definitions`     | 公开 active 只读  |  是 |     可 |     否 | admin / seed |
| `tasks.task_periods`         | 公开 active 只读  |  是 |     可 |     否 | admin / cron |
| `tasks.user_task_progress`   | 用户本人只读      |  是 |     可 |     否 | RPC          |
| `tasks.task_claims`          | 用户本人只读      |  是 |     可 |     否 | RPC          |
| `tasks.signin_campaigns`     | 公开 active 只读  |  是 |     可 |     否 | admin / seed |
| `tasks.signin_days`          | 公开 active 只读  |  是 |     可 |     否 | admin / seed |
| `tasks.user_signins`         | 用户本人只读      |  是 |     可 |     否 | RPC          |
| `tasks.referrals`            | 邀请双方只读      |  是 |     可 |     否 | RPC          |
| `tasks.referral_rewards`     | 用户本人只读      |  是 |     可 |     否 | RPC          |
| `tasks.referral_commissions` | 邀请人只读        |  是 |     可 |     否 | RPC          |
| `tasks.share_events`         | 用户本人只读      |  是 |     可 | 建议否 | API + RPC    |
| `economy.user_balances`      | 用户本人只读      |  是 |     可 |     否 | RPC          |
| `economy.currency_ledger`    | 用户本人只读      |  是 |     可 |     否 | RPC          |
| `ops.risk_events`            | 后端 / 管理员专用 |  是 |     否 |     否 | 后端 / admin |
| `ops.idempotency_keys`       | 后端专用          |  是 |     否 |     否 | RPC / API    |

---

### 4.2 必须只能走 RPC 的操作

| 操作         | 原因                                 |
| ------------ | ------------------------------------ |
| 签到         | 涉及连续天数、奖励、ledger、防重复。 |
| 任务进度更新 | 不能让前端伪造进度。                 |
| 任务奖励领取 | 涉及 ledger、防重复、并发锁。        |
| 邀请关系绑定 | 涉及防作弊、自邀请、重复绑定。       |
| 首开盒奖励   | 必须由真实支付 / 开盒结果触发。      |
| 分红生成     | 必须由真实好友开盒结果触发。         |
| 分红领取     | 涉及 ledger、防重复、并发锁。        |
| 风控事件写入 | 不能暴露给前端。                     |

---

### 4.3 RPC 权限要求

| RPC                           | `SECURITY DEFINER` | 必须检查                                         |
| ----------------------------- | -----------------: | ------------------------------------------------ |
| `task_daily_check_in`         |                 是 | `p_user_id` 有效、用户 active、campaign active。 |
| `task_claim_reward`           |                 是 | progress 属于 `p_user_id`，任务已完成。          |
| `task_record_progress`        |                 是 | 只能由后端调用；不能暴露给普通前端 RPC。         |
| `referral_bind_inviter`       |                 是 | invitee 是当前 session 用户；邀请人合法。        |
| `referral_process_first_open` |                 是 | draw_order 属于 invitee 且已成功开盒。           |
| `referral_create_commission`  |                 是 | source_id 是真实开盒订单，不能伪造。             |
| `referral_claim_commission`   |                 是 | commission 属于当前 inviter。                    |
| `get_user_task_center`        |       是或 Invoker | 返回数据只能是当前用户相关数据。                 |

---

### 4.4 不能返回给前端的字段

| 字段                                                | 原因                                    |
| --------------------------------------------------- | --------------------------------------- |
| `core.users.risk_score`                             | 风控敏感。                              |
| `core.users.metadata` 中的风控字段                  | 避免被绕过。                            |
| `ops.risk_events.detail`                            | 防作弊规则不能暴露。                    |
| `currency_ledger.metadata` 中的内部 request context | 可能含内部来源。                        |
| `tasks.referrals.invitee_user_id` 的完整 UUID       | 邀请记录可以展示昵称，不必暴露完整 ID。 |
| `payments.*` 原始 webhook                           | 支付敏感。                              |

---

### 4.5 RLS 验证方式

验收必须包括：

1. A 用户不能读取 B 用户的 `user_task_progress`。
2. A 用户不能读取 B 用户的 `task_claims`。
3. A 用户不能读取 B 用户的 `user_signins`。
4. 邀请关系只有 inviter 和 invitee 能看到。
5. 分红只有 inviter 能看到。
6. 前端角色不能 insert/update/delete `currency_ledger`。
7. 前端角色不能写 `task_claims`、`referrals`、`referral_commissions`。
8. `ops.risk_events` 和 `ops.idempotency_keys` 对 anon/authenticated 均不可访问。
9. 所有写操作用 API + service role + RPC 完成。

## 7. 邀请系统开发流程

### 7.1 邀请链接生成

| 项       | 内容                                                                      |
| -------- | ------------------------------------------------------------------------- |
| 目标     | 用户点击“立即邀请”后获得可分享链接。                                      |
| 复用结构 | `core.users.invite_code`。                                                |
| API      | `POST /api/tasks/referral-link`。                                         |
| 前端     | `InviteCampaignCard`、`useReferralLink`、`useInviteShare`。               |
| 业务规则 | 每个用户稳定邀请码；邀请码不暴露内部 user_id。                            |
| 风险点   | 链接被篡改、空 code、过期活动。                                           |
| 验收标准 | 返回 `inviteUrl`、`startPayload`、`shareText`，复制和 Telegram 分享可用。 |

### 7.2 邀请关系绑定

| 项        | 内容                                                                                     |
| --------- | ---------------------------------------------------------------------------------------- |
| 目标      | 新用户通过邀请链接进入后，绑定 inviter。                                                 |
| 需要检查  | `api/auth/telegram.ts` 是否已解析 Telegram start param；登录完成后能否调用 RPC。         |
| 新增/修改 | `api/tasks/bind-referral.ts`，或在 auth 成功后内部调用 `referral_bind_inviter`。         |
| 涉及表    | `core.users`、`tasks.referrals`、`ops.risk_events`。                                     |
| 关键规则  | 不能自己邀请自己；一个 invitee 只能绑定一个 inviter；inviter 必须 active；不做多级邀请。 |
| 风险点    | 老用户点链接重绑；同设备刷号；恶意构造 start payload。                                   |
| 验收标准  | 新用户首次进入可绑定；重复绑定同一 inviter 返回幂等；绑定不同 inviter 返回冲突。         |

### 7.3 首次开盒奖励

| 项       | 内容                                                                                          |
| -------- | --------------------------------------------------------------------------------------------- |
| 目标     | 被邀请人首次成功开盒后，双方各得 500 KCOIN。                                                  |
| 需要检查 | `api.gacha_process_paid_order` 是否已调用 `referral_process_first_open`。                     |
| 修改位置 | Supabase `api.gacha_process_paid_order` RPC，而不是前端。                                     |
| 涉及表   | `gacha.draw_orders`、`tasks.referrals`、`tasks.referral_rewards`、`economy.currency_ledger`。 |
| 关键规则 | 只有成功支付并成功开盒后触发；首次开盒只触发一次。                                            |
| 风险点   | webhook 重放、订单补偿重复执行、referral 状态不一致。                                         |
| 验收标准 | 邀请人和被邀请人各产生 1 条 referral_reward 和 1 条 KCOIN ledger。                            |

### 7.4 邀请分红

| 项       | 内容                                                                             |
| -------- | -------------------------------------------------------------------------------- |
| 目标     | 好友后续开盒后，邀请人获得 10% KCOIN 分红。                                      |
| 需要检查 | 当前 `referral_create_commission` 是立即发放模式。                               |
| 推荐修改 | 改为生成 `pending` commission；用户手动 claim 后写 ledger。                      |
| 分红基数 | 使用好友开盒真实 KCOIN 返还金额作为 base。                                       |
| 涉及表   | `tasks.referral_commissions`、`economy.currency_ledger`。                        |
| 风险点   | 同一订单重复分红；分红基数不一致；状态 pending/granted 混乱。                    |
| 验收标准 | 同一 draw_order 只生成一条 commission；领取后 ledger 金额等于 pending 分红总额。 |

| ------------- | ---------------------------------------- |
| 用户今天未签  | 今日可领取。                             |
| 用户今天已签  | 返回 already_claimed，不重复发奖励。     |
| 昨天已签      | streak + 1。                             |
| 昨天未签      | streak 重置为 1。                        |
| streak 到 7   | 第 7 天后下一次回到第 1 天，开始新周期。 |
| campaign 结束 | 不允许签到。                             |

当前 `task_daily_check_in` 只按历史 claimed 次数计算 day_index，不足以完整支持断签。建议新增 `tasks.user_signin_states`，或重写 RPC 通过最近签到日期计算 streak。

---

### 8.3 签到领取事务

| 步骤 | 说明                                    |
| ---: | --------------------------------------- |
|    1 | API 从 session 获取 user_id。           |
|    2 | 校验 body 和 idempotency key。          |
|    3 | RPC 查询 active campaign。              |
|    4 | 锁定用户签到状态。                      |
|    5 | 检查今日是否已签到。                    |
|    6 | 计算 day_index 和 streak。              |
|    7 | 插入 `tasks.user_signins`。             |
|    8 | 调用奖励 helper，写 `currency_ledger`。 |
|    9 | 更新任务进度：`DAILY_CHECK_IN`。        |
|   10 | 返回签到结果和新余额。                  |

验收标准：并发请求只产生一次签到、一次奖励、一次 ledger。

## 9. 任务列表和奖励领取开发流程

### 9.1 任务进度来源

| 任务类型     | 真实事件来源            | 触发位置                      |
| ------------ | ----------------------- | ----------------------------- |
| 开盒任务     | 成功开盒结果            | `gacha_process_paid_order`    |
| 邀请任务     | referral 首开成功       | `referral_process_first_open` |
| 分享任务     | 分享事件记录            | `referral_record_share_event` |
| 交易购买任务 | 市场购买成功            | `market_buy_listing`          |
| 挂售任务     | 创建挂单成功            | `market_create_listing`       |
| 升级任务     | 藏品升级成功            | `inventory_upgrade_item`      |
| 合成任务     | 藏品合成成功或尝试      | `inventory_evolve_item`       |
| 分解任务     | 分解成功                | `inventory_decompose_item`    |
| 图鉴任务     | 图鉴里程碑领取 / 新发现 | album RPC / trigger           |
| 钱包任务     | 钱包验证成功            | wallet RPC                    |
| 链上任务     | NFT 同步 / Mint 成功    | onchain RPC                   |

核心规则：**前端不能提交“我完成了任务”。只能由后端在真实业务 RPC 成功后记录进度。**

---

### 9.2 任务进度更新流程

| 项       | 内容                                                                         |
| -------- | ---------------------------------------------------------------------------- |
| 目标     | 业务事件成功后自动更新相关任务。                                             |
| 新增 RPC | `api.task_record_progress`。                                                 |
| 涉及表   | `tasks.task_definitions`、`tasks.task_periods`、`tasks.user_task_progress`。 |
| 关键规则 | 根据 action_type 匹配 active task；按 period_key upsert progress。           |
| 幂等     | 同一 `source_type + source_id + task_id` 只能加一次进度。                    |
| 风险点   | webhook 重放导致进度重复增加。                                               |
| 验收标准 | 同一开盒订单重复处理，任务进度只增加一次。                                   |

如需要完整审计，建议新增 `tasks.task_progress_events`：

| 字段          | 用途                                |
| ------------- | ----------------------------------- |
| `user_id`     | 用户                                |
| `task_id`     | 任务                                |
| `period_key`  | 周期                                |
| `source_type` | gacha / market / inventory / wallet |
| `source_id`   | 业务事件 ID                         |
| `amount`      | 进度增量                            |
| `metadata`    | 事件详情                            |

---

### 9.3 任务领取流程

| 步骤 | 说明                                   |
| ---: | -------------------------------------- |
|    1 | 前端展示任务 status。                  |
|    2 | 用户点击领取。                         |
|    3 | API 校验 session 和 idempotency key。  |
|    4 | RPC 锁定 progress row。                |
|    5 | 校验 progress.status = completed。     |
|    6 | 插入 `task_claims`。                   |
|    7 | 调用奖励 helper 写 `currency_ledger`。 |
|    8 | progress.status 改为 claimed。         |
|    9 | 返回奖励结果。                         |
|   10 | 前端刷新资产和任务列表。               |

验收标准：未完成任务不能领；已领取任务重复点击返回 claimed，不重复发奖励。

## 10. 分红统计开发流程

### 10.1 统计口径

| 指标         | 计算来源                                                                            |
| ------------ | ----------------------------------------------------------------------------------- |
| 邀请人数     | `tasks.referrals` where `inviter_user_id = user_id`。                               |
| 有效邀请人数 | referral status in `qualified/rewarded` 或 `first_open_order_id is not null`。      |
| 首开人数     | `first_open_order_id is not null`。                                                 |
| 累计邀请奖励 | `tasks.referral_rewards` where `user_id = inviter` and status granted。             |
| 待领取分红   | `tasks.referral_commissions` where `inviter_user_id = user_id` and status pending。 |
| 已领取分红   | `tasks.referral_commissions` where status granted。                                 |
| 累计分红     | pending + granted 或仅 granted，页面需明确文案。                                    |
| 分红比例     | 当前默认 10%，建议从配置或 RPC 返回。                                               |

---

### 10.2 分红领取流程

| 步骤 | 说明                                            |
| ---: | ----------------------------------------------- |
|    1 | 用户进入任务页，加载分红统计。                  |
|    2 | 若 `pendingCommissionKcoin > 0`，显示领取按钮。 |
|    3 | 用户点击领取。                                  |
|    4 | API 传 idempotency key。                        |
|    5 | RPC 锁定该用户 pending commission rows。        |
|    6 | 汇总金额。                                      |
|    7 | 写一条或多条 `currency_ledger`。                |
|    8 | commission 状态改为 granted，写 `claimed_at`。  |
|    9 | 返回领取金额和余额。                            |
|   10 | 前端刷新资产、统计、分红历史。                  |

验收标准：并发领取只能成功一次，后续请求返回 0 pending 或 idempotent result。

### 11.5 Ledger 对账测试

必须满足：

5. `user_balances.available_amount` 与 ledger 汇总一致。
6. 无任何奖励只改 `user_balances` 不写 ledger。

## 12. 最终验收标准

第四阶段完成后必须满足：

| 类别     | 标准                                                               |
| -------- | ------------------------------------------------------------------ |
| 页面     | `/tasks` 不再是占位页，完整展示邀请、签到、任务、分红。            |
| 邀请链接 | 用户可生成、复制、Telegram 分享邀请链接。                          |
| 邀请绑定 | 新用户通过链接进入后，后端绑定邀请关系；自邀请和重复绑定被拒绝。   |
| 首开奖励 | 被邀请人首次成功开盒后，双方各得 500 KCOIN，且写入 ledger。        |
| 分红     | 好友开盒后生成 10% 分红；支持待领取 / 已领取统计；领取写 ledger。  |
| 签到     | 支持 7 日签到、连续签到、断签规则、今日状态、重复领取防护。        |
| 任务列表 | 支持每日、社交、交易、链上等任务展示。                             |
| 任务进度 | 只能由后端真实业务事件更新，前端不能伪造。                         |
| 奖励领取 | 未完成不能领，已领取不能重复发，所有奖励有 ledger。                |
| 幂等     | 签到、领取任务、绑定邀请、首开奖励、生成分红、领取分红均具备幂等。 |
| 并发     | 多次点击、并发请求、webhook 重放不会重复发奖励。                   |
| RLS      | 用户只能看到自己的任务、签到、邀请、分红、ledger；不能写核心表。   |
| 风控     | 自邀请、重复绑定、异常分享、刷号行为写入 `ops.risk_events`。       |
| 后台运营 | 任务定义、签到奖励、邀请奖励、分红比例可通过 seed/admin 配置。     |
| 测试     | DB、RPC、API、前端、并发、RLS、ledger 对账测试通过。               |
