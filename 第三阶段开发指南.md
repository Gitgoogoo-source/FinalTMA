# 第三阶段：成长系统开发教学指南

## 0. 当前项目基线判断

你现在的项目已经具备第三阶段的基础骨架，但第三阶段还没有完整接通。

当前仓库是 `pnpm` monorepo，包含 `apps/*`、`packages/*`、`contracts`，根目录已经准备了 `dev`、`build`、`typecheck`、`lint`、`test`、`test:db`、`db:types:linked`、`ops:backfill-album` 等脚本，适合按“数据库 → API → 前端 → 测试 → 发布”的顺序推进第三阶段。

前端应用已经使用 React、Vite、React Router、TanStack Query、Zustand、TMA SDK、TON Connect、Zod 等技术栈。第三阶段不需要换技术栈，应该沿用当前架构继续扩展。

当前前端路由已有 `/box`、`/collection`、`/trade`、`/game`、`/tasks`，但还没有 `/album` 图鉴路由。因此第三阶段需要新增独立图鉴页面，或者把图鉴作为 `collection` 的子页面；更推荐新增 `/album`，因为图鉴、图鉴奖励、排行榜是独立系统。

当前 `CollectionPage` 已经可以读取用户库存、维护选中藏品、展示空状态 / 加载状态 / 错误状态，并渲染 `CharacterHero` 和 `CharacterGrid`。这部分可以作为升级、合成、分解 UI 的基础，不需要推倒重做。

当前 `useInventory` 已经通过 TanStack Query 获取库存，并且只在用户已登录时启用查询；`collection.api.ts` 已经把库存接口返回值标准化成前端可用的藏品对象；`collection.types.ts` 已经包含 `level`、`power`、`status`、`isUpgradeable`、`isEvolvable`、`isDecomposable`、`isMintable` 等第三阶段需要的字段。

当前 `CharacterHero` 已经展示藏品名称、稀有度、图片、等级、战力、编号；`CharacterGrid` 和 `CharacterThumb` 已经展示藏品列表、选中状态、等级、稀有度、战力。因此第三阶段应该在这些组件旁边增加详情面板和成长操作，不应该把藏品展示逻辑重复写一套。

当前前端 API 常量里 `inventory` 只有 `list`，没有 `detail`、`upgrade`、`evolve`、`decompose`；也没有 `album` 相关 endpoint。因此第三阶段的前端 API 常量、query key、hooks 都需要补齐。

当前 `api/inventory/list.ts` 已经按项目规范实现了：验证 query、要求 session、调用 Supabase RPC、统一转换返回值。但它目前明确拒绝了大量高级筛选条件，例如 keyword、rarity、type、only_duplicates、level、power、sort 等。第三阶段需要扩展这些筛选能力，或者新增专用接口给升级 / 合成 / 分解选择面板使用。

当前项目已经有成熟的后端写接口模式：以 `api/market/create-listing.ts` 为例，写操作会读取 session、解析 body、校验 schema、读取幂等键、调用 `api` schema 下的 RPC、标准化返回值、映射 RPC 错误、配置 rate limit。第三阶段所有升级、合成、分解、领取图鉴奖励接口都应该照这个模式实现。

当前 `packages/validation/src/inventory.schemas.ts` 已经预先定义了库存详情、升级、合成、分解、出售入口、库存活动记录等 schema；`packages/validation/src/album.schemas.ts` 已经定义了图鉴进度、图鉴册、图鉴物品、里程碑奖励、排行榜、发现记录等 schema。第三阶段应该复用这些 schema，不要在 API 文件里重新手写一套规则。

当前 RPC 封装文件已经写明核心原则：所有核心写操作走 RPC，RPC 内部做事务、锁、幂等、余额校验、库存校验；API 层不能相信前端传来的 `user_id`，必须从 session 中取用户 ID 后传给 RPC。第三阶段必须继续遵守这个原则。

# 1. 第三阶段总目标

第三阶段最终要完成 6 个闭环：

| 功能        | 最终目标                                                                           |
| ----------- | ---------------------------------------------------------------------------------- |
| 升级        | 用户选择藏品，消耗 Fgems，等级和战力提升，升级必定成功。                           |
| 合成 / 进化 | 用户消耗 3 份相同藏品 + K-coin，按成功率进化到下一形态；失败返还等级最高的主藏品。 |
| 分解        | 用户分解重复藏品，获得 Fgems；分解后藏品不可恢复。                                 |
| 图鉴        | 用户查看总收集进度、系列进度、稀有度进度、已收集 / 未收集藏品。                    |
| 图鉴奖励    | 用户达到里程碑后领取奖励；奖励由服务端判断。                                       |
| 排行榜      | 用户查看每周图鉴榜，包括排名、完成度、稀有藏品数、Mint 数量、总分。                |

第三阶段的开发顺序必须是：

```txt
数据库规则数据
  ↓
数据库 RPC / 查询视图
  ↓
Vercel API 接口
  ↓
前端 API client / hooks
  ↓
Collection 成长 UI
  ↓
Album 图鉴 UI
  ↓
Leaderboard 排行榜 UI
  ↓
测试与验收
  ↓
预发部署与正式发布
```

不要从 UI 开始写。因为升级、合成、分解、图鉴奖励、排行榜全部依赖数据库规则和服务端返回结果。

# 2. 第一步：建立第三阶段开发基线

## 要完成什么

建立一个稳定开发分支，确认当前第二阶段交易闭环没有被破坏，并确认数据库、API、前端可以正常跑起来。

## 怎么做

创建第三阶段分支：

```txt
stage-3-growth-system
```

然后完成以下检查：

```txt
1. 安装依赖
2. 生成数据库类型
3. 启动前端
4. 启动 Vercel API
5. 打开 Telegram Mini App 或本地 Web 页面
6. 进入 collection 页面
7. 确认当前用户能看到已有藏品
8. 确认 trade 页面仍然能正常展示
```

你当前根目录已经准备了 `db:types:linked`、`dev:web`、`dev:vercel`、`typecheck`、`lint`、`test`、`build` 等脚本，所以第三阶段应该继续用这些脚本做基础验证。

本地浏览器验收时，页面地址必须和 `VITE_API_BASE_URL` 的主机名保持一致。当前示例配置使用 `VITE_API_BASE_URL=http://localhost:3000/api`，所以本地验收应打开 `http://localhost:3000`，不要混用 `http://127.0.0.1:3000`。浏览器会把 `localhost` 和 `127.0.0.1` 当成不同站点，混用时 session cookie 可能无法按预期写入或携带，页面会表现为“登录已失效 / 正在重新认证”的循环。

## 验收标准

完成后必须满足：

```txt
pnpm typecheck 通过
pnpm lint 通过
pnpm build 通过
collection 页面能读取库存
trade 页面第二阶段功能不报错
浏览器控制台没有关键运行错误
API 请求能带上 session cookie
```

但关键规则数据目前是空的：

```txt
inventory.upgrade_rules        0 行
inventory.evolution_rules      0 行
inventory.decompose_rules      0 行
album.books                    0 行
album.book_items               0 行
album.milestones               0 行
album.score_rules              0 行
album.weekly_leaderboards      0 行
album.leaderboard_entries      0 行
```

同时已有：

```txt

catalog.collectible_media      48 行
album.user_discoveries         4 行
```

这说明第三阶段不是“缺表”，而是：

```txt
表结构已有
RPC 部分已有
规则数据为空
前端接口未接
图鉴页面未建
排行榜查询和刷新缺失
```

## 怎么做

先整理一份数据库检查清单：

```txt
1. 当前有哪些 active 藏品模板
2. 每个模板是否有 form
3. 每个 form 是否配置 next_form_id
4. 每个藏品是否允许 upgradeable / evolvable / decomposable
5. 用户是否有足够重复藏品用于合成 / 分解测试
6. 用户是否有 KCOIN / FGEMS 余额用于测试
7. 图鉴是否按 template 统计，还是按 form 统计
```

其中第 7 点必须先决定。

当前数据库的 `album.book_items` 只记录 `book_id + template_id`，`album.user_discoveries` 也只记录 `user_id + template_id`。这意味着当前图鉴模型是“按藏品模板点亮”，不是“按形态点亮”。

如果你的三段形态只是同一个藏品的外观成长，当前结构可以继续用。
如果你希望低阶、中阶、高阶都算独立图鉴进度，就需要在第三阶段先调整图鉴表，给 `album.book_items` 和 `album.user_discoveries` 增加 `form_id` 维度。

## 推荐决策

第三阶段建议先采用：

```txt
图鉴按 template 点亮
形态作为藏品详情里的成长状态展示
排行榜按 template 收集进度计算
```

这样可以少改数据库，先完成第三阶段闭环。

如果你后续要做“形态图鉴”，可以在第四阶段或后续版本升级为：

```txt
album.book_items: book_id + template_id + form_id
album.user_discoveries: user_id + template_id + form_id
```

## 验收标准

完成后必须得到一份明确结论：

```txt
图鉴统计维度：template 级别
升级规则：按 rarity + form_index + level
合成规则：按 from_template_id + from_form_id → to_template_id + to_form_id
分解规则：按 rarity + form_index + min_level
排行榜计算来源：album.user_discoveries + onchain.nft_items + score_rules
```

# 4. 第三步：补齐成长系统规则数据

## 要完成什么

让数据库中存在可执行的升级、合成、分解规则。否则当前 RPC 即使存在，也会因为找不到规则而失败。

当前数据库已有 `api.inventory_upgrade_item`、`api.inventory_evolve_item`、`api.inventory_decompose_item` 这三个 RPC，但它们都依赖规则表：

```txt
inventory.upgrade_rules
inventory.evolution_rules
inventory.decompose_rules
```

这些表目前为空，所以第三阶段第一批真实开发任务就是补规则数据。

---

## 4.1 升级规则

### 要完成什么

为所有可升级藏品配置升级规则。

`inventory.upgrade_rules` 的核心维度是：

```txt
rarity_code
form_index
from_level
to_level
cost_fgems
power_gain
active
```

### 怎么做

按稀有度配置不同升级成本：

```txt
common      成本低，战力增长低
rare        成本中，战力增长中
epic        成本高，战力增长高
legendary   成本最高，战力增长最高
```

按形态配置额外倍率：

```txt
form_index = 1  低阶形态
form_index = 2  中阶形态
form_index = 3  高阶形态
```

推荐第三阶段先做 1 级升 1 级，不要一开始做批量升 10 级。因为当前 `api.inventory_upgrade_item` RPC 本身就是按当前等级找一条升级规则，然后升到下一等级。

推荐初始规则：

```txt
common:
  Lv.1 → Lv.2: 10 FGEMS, +5 power
  Lv.2 → Lv.3: 15 FGEMS, +6 power
  Lv.3 → Lv.4: 20 FGEMS, +7 power

rare:
  Lv.1 → Lv.2: 20 FGEMS, +8 power
  Lv.2 → Lv.3: 30 FGEMS, +10 power
  Lv.3 → Lv.4: 40 FGEMS, +12 power

epic:
  Lv.1 → Lv.2: 40 FGEMS, +14 power
  Lv.2 → Lv.3: 60 FGEMS, +18 power
  Lv.3 → Lv.4: 80 FGEMS, +22 power

legendary:
  Lv.1 → Lv.2: 80 FGEMS, +25 power
  Lv.2 → Lv.3: 120 FGEMS, +32 power
  Lv.3 → Lv.4: 160 FGEMS, +40 power
```

第三阶段可以先支持到 Lv.10，商业化前再扩展到 Lv.100。

### 验收标准

任选一个用户的 available 藏品，必须能查到：

```txt
该藏品的 rarity_code
该藏品的 form_index
该藏品当前 level
对应 upgrade_rules 中存在 from_level = 当前 level 的 active 规则
```

验收结果：

```txt
inventory.upgrade_rules 不再是 0 行
任意 active 藏品至少能从 Lv.1 升到 Lv.2
没有配置 active=false 的规则被前端展示为可用
```

## 4.2 合成 / 进化规则

### 要完成什么

配置 3 份相同藏品进化到下一形态的规则。

`inventory.evolution_rules` 的核心维度是：

```txt
from_template_id
from_form_id
to_template_id
to_form_id
required_count
cost_kcoin
success_rate_bps
active
```

当前 RPC 逻辑已经要求：

```txt
必须正好 3 个 item id
必须是 3 个不同实例
必须属于同一个用户
必须都是 available
必须同 template
必须同 form
必须 template.evolvable = true
必须找到 active evolution_rule
```

### 怎么做

从 `catalog.collectible_forms.next_form_id` 生成进化规则。

如果当前设计是“同一个 template 有多个 form”，则：

```txt
from_template_id = 当前 template_id
from_form_id     = 当前 form_id
to_template_id   = 当前 template_id
to_form_id       = 当前 form 的 next_form_id
```

如果未来你改成“低阶、中阶、高阶是不同 template”，则：

```txt
from_template_id = 低阶 template
to_template_id   = 中阶 template
```

第三阶段建议先用同 template 不同 form 的方式，因为当前数据库已经有 `collectible_forms` 表。

推荐初始成功率：

```txt
低阶 → 中阶:
  common      80%
  rare        70%
  epic        60%
  legendary   50%

中阶 → 高阶:
  common      60%
  rare        50%
  epic        40%
  legendary   30%
```

推荐初始 K-coin 成本：

```txt
common      100 KCOIN
rare        300 KCOIN
epic        800 KCOIN
legendary   2000 KCOIN
```

失败规则不需要写在配置表里，因为当前 RPC 已经写死：

```txt
失败后返还 3 个藏品中等级最高、战力最高、获取时间最早的那个主藏品
其余两个消耗
KCOIN 不返还
```

### 验收标准

任选一个用户拥有 3 个相同 template + form 的 available 藏品，必须满足：

```txt
能查到 active evolution_rule
required_count = 3
cost_kcoin >= 0
success_rate_bps 在 0 到 10000 之间
to_form_id 指向有效 collectible_forms
```

验收结果：

```txt
inventory.evolution_rules 不再是 0 行
至少有一个藏品能从低阶进化到中阶
如果没有 3 个重复藏品，需要先通过测试种子数据补齐
```

## 4.3 分解规则

### 要完成什么

配置重复藏品分解后获得多少 Fgems。

`inventory.decompose_rules` 的核心维度是：

```txt
rarity_code
form_index
min_level
reward_fgems
active
```

当前 RPC 已经要求：

```txt
只能分解 available 藏品
只能分解当前用户自己的藏品
只能分解可 decomposable 的藏品
同 template + form 的 available 数量必须 >= 2
也就是只能分解重复藏品
```

### 怎么做

先按稀有度和形态配置基础分解收益：

```txt
common:
  form 1: 5 FGEMS
  form 2: 15 FGEMS
  form 3: 40 FGEMS

rare:
  form 1: 15 FGEMS
  form 2: 45 FGEMS
  form 3: 120 FGEMS

epic:
  form 1: 50 FGEMS
  form 2: 150 FGEMS
  form 3: 400 FGEMS

legendary:
  form 1: 150 FGEMS
  form 2: 450 FGEMS
  form 3: 1200 FGEMS
```

如果以后要按等级增加分解收益，可以增加：

```txt
min_level = 10
min_level = 20
min_level = 50
```

第三阶段建议先只做 `min_level = 1`。

### 验收标准

任选一个用户拥有 2 个相同 template + form 的 available 藏品，必须满足：

```txt
能查到 active decompose_rule
reward_fgems > 0
该藏品 template.decomposable = true
```

验收结果：

```txt
inventory.decompose_rules 不再是 0 行
只有重复藏品显示分解按钮
单件藏品不会显示为可分解
```

# 5. 第四步：补齐图鉴基础数据

让图鉴页面有“图鉴册、图鉴物品、里程碑奖励”。

## 5.1 建立图鉴册

### 怎么做

先创建 3 类图鉴册：

```txt
1. all 总图鉴
2. series 系列图鉴
3. rarity 稀有度图鉴
```

第三阶段建议先不做 event 限时图鉴，避免复杂度过高。

推荐图鉴册：

```txt
all_collectibles
series_<series_slug>
rarity_common
rarity_rare
rarity_epic
rarity_legendary
```

### 验收标准

```txt
album.books 至少有 1 个 all 图鉴
每个 active series 至少有 1 个 series 图鉴
每个 rarity 至少有 1 个 rarity 图鉴
所有 active book 都有明确 book_type
```

---

## 5.2 建立图鉴物品

### 怎么做

给每个图鉴册写入 `album.book_items`。

总图鉴：

```txt
包含所有 active collectible_templates
```

系列图鉴：

```txt
包含对应 series_id 下的 active collectible_templates
```

稀有度图鉴：

```txt
包含对应 rarity_code 下的 active collectible_templates
```

注意：如果当前图鉴按 template 统计，则 `album.book_items` 每个藏品只记录 template。不要在前端自己推断图鉴范围。

### 验收标准

```txt
每个 active book 的 book_items 数量 > 0
总图鉴 book_items 数量 = active collectible_templates 数量
series 图鉴数量 = 对应 series 下 active templates 数量
rarity 图鉴数量 = 对应 rarity 下 active templates 数量
```

---

## 5.3 建立图鉴里程碑

### 怎么做

给每个图鉴册配置里程碑。

总图鉴建议：

```txt
收集 1 个：100 KCOIN
收集 3 个：200 FGEMS
收集 6 个：500 KCOIN
收集 12 个：1000 KCOIN + 500 FGEMS
```

系列图鉴建议：

```txt
收集 1 个：50 FGEMS
收集 50%：150 FGEMS
收集 100%：500 KCOIN
```

稀有度图鉴建议：

```txt
common 收集进度：FGEMS 奖励
rare 收集进度：KCOIN + FGEMS
```

```txt
8. 调用 /api/inventory/evolve
9. 后端返回 success 或 failed
10. 成功：展示新形态
11. 失败：展示返还主藏品和消耗材料
12. invalidate inventory、assets、album
```

### 选择规则

前端可以帮用户自动选择 3 个，但最终以后端校验为准。

推荐自动选择逻辑：

```txt
1. 默认保留等级最高的作为主藏品
2. 如果用户点选了 3 个，则按后端规则重新计算主藏品
3. 前端明确提示：失败返还规则以后端返回为准
```

### 禁用条件

```txt
同款 available 数量 < 3
item.status != available
item.isEvolvable = false
没有 evolution_rule
已经是最高形态
KCOIN 不足
API 正在提交中
```

### 验收标准

成功路径：

```txt
旧 3 件藏品从列表中消失或显示为 consumed
新增目标形态藏品
KCOIN 减少
弹窗显示“合成成功”
图鉴进度可能更新
```

失败路径：

```txt
主藏品仍在库存中
其他两个材料消失或显示为 consumed
KCOIN 减少
弹窗显示“合成失败，已返还主藏品”
不会产生目标形态藏品
```

异常路径：

```txt
listed item 不能参与合成
locked item 不能参与合成
不同 template / form 不能合成
重复传同一个 item id 被拒绝
```



刷新：

```txt
inventory.root
me.assetsRoot
```

如果失败没有新发现，则不一定要刷新 album。但为了简单，可以统一刷新 `album.root`。

---

### 分解成功后

刷新：

```txt
inventory.root
me.assetsRoot
```

不要减少图鉴进度。图鉴基于 `album.user_discoveries`，不是当前库存。

---

### 图鉴奖励领取成功后

刷新：

```txt
album.root
me.assetsRoot
```

---

## 验收标准

```txt
操作成功后资产栏立即刷新
库存列表立即刷新
图鉴奖励领取后状态变成已领取
失败操作不会错误刷新成本地假成功
```

# 21. 最终验收标准

第三阶段完成时，必须可以完整跑通：

```txt
1. 用户进入藏品页
2. 选择藏品
3. 查看详情
4. 消耗 FGEMS 升级成功
5. 消耗 3 个重复藏品 + KCOIN 合成
6. 合成成功时获得新形态
7. 合成失败时返还主藏品
8. 分解重复藏品获得 FGEMS
9. 进入图鉴页
10. 查看总收集进度
11. 查看系列 / 稀有度图鉴
12. 达成里程碑后领取奖励
13. 领取后资产刷新
14. 查看每周图鉴排行榜
15. 出售或分解藏品后，图鉴已点亮状态不丢失
16. 所有资产变化都有 ledger
17. 所有库存变化都有 item_instance_events
18. 所有写操作都能防重复提交
```

按照这个顺序开发，第三阶段会形成完整闭环：**成长消耗 → 藏品状态变化 → 图鉴进度沉淀 → 奖励领取 → 排行榜竞争**。
