TMA 用户 session 链路说明：

1. `/api/auth/telegram` 只接收 Telegram WebApp 提供的原始 `initData`。
2. 后端通过 `verifyTelegramInitData(initData)` 校验 Bot Token 签名、`hash`、`auth_date` 和 `user`。
3. 后端调用 `api.auth_upsert_telegram_user(...)` 创建或刷新已验证用户。
4. 后端生成 opaque session token，数据库只保存 `session_token_hash`。
5. 后端调用 `api.auth_create_session(...)` 写入 `core.app_sessions`，并通过 `ops.telegram_init_data_consumptions` 防止同一 `initDataHash` 重放。
6. 登录响应只设置 HttpOnly Cookie，并只返回 session 元数据；浏览器前端不接收 Bearer token。

`core.app_sessions` 当前关键字段：

- `id`
- `user_id`
- `session_token_hash`
- `telegram_auth_date`
- `init_data_hash`
- `ip_hash`
- `user_agent`
- `device_id`
- `platform`
- `expires_at`
- `revoked_at`
- `last_seen_at`
- `created_at`

Bearer 约定：

- 浏览器前端主链路只使用 HttpOnly Cookie。
- `Authorization: Bearer ...` 只用于后端/测试客户端携带同一 opaque session token。
- 不要把 Bearer token 发行给浏览器前端，也不要把 token 存进前端 store/localStorage。

权限约定：

- `core.app_sessions` 和 `core.user_api_tokens` 是服务端内部表。
- `anon` / `authenticated` 不应直接读写这些表。
- Vercel Functions 通过 server-only Supabase secret/service role 读取 session。
