# 风控和安全支撑功能说明

本文只根据当前仓库里的真实代码整理。

这份文档说明的是项目里已经能从代码确认的风控和安全支撑能力。它不是接口说明，不写请求内容，不写密钥值，不写内部阈值，也不补充代码里没有确认到的结论。

## 1. 功能定位

风控和安全支撑不是一个单独页面，而是一组后端、数据库、定时任务和监控能力。

它主要负责这些事：

- 确认用户登录身份是真的。
- 保护登录会话，避免前端伪造用户身份。
- 限制接口被高频调用。
- 在开盒、交易、任务、钱包、Mint、支付等关键动作前做风险检查。
- 对异常行为做拦截、复核或记录。
- 对后台任务、支付回调、钱包验证、链上 Mint 等入口做安全校验。
- 把异常、风险事件、后台操作和系统错误记录下来，方便运营和开发排查。
- 通过功能开关在异常时暂停部分高风险功能。
- 通过数据库权限控制，限制前端直接改关键数据。

一句话说，这套功能的作用是：让用户正常使用业务，同时尽量挡住伪造登录、重复请求、刷奖励、异常交易、支付回调伪造、钱包证明重放、Mint 异常和后台误操作。

## 2. 整体结构

当前项目里的风控和安全支撑主要分成这些层：

| 层级 | 做什么 |
| --- | --- |
| 登录安全 | 校验 Telegram 登录数据，创建安全会话，防止登录凭证重复使用 |
| 会话安全 | 后端从安全 Cookie 或可信服务端令牌里识别用户，不相信前端传来的用户身份 |
| API 入口保护 | 统一处理请求方法、内容类型、请求大小、跨域、安全响应头、错误返回和请求编号 |
| 接口限流 | 按接口、用户、会话、钱包、Telegram 用户等维度限制高频访问 |
| 风控决策 | 对业务动作给出放行、挑战、人工复核或拒绝的结果 |
| 用户风控标记 | 对被限制、冻结、观察或需要人工复核的用户做标记 |
| 支付安全 | 校验支付环境、支付回调来源和支付订单状态，防止重复或伪造回调 |
| 钱包安全 | 校验钱包归属、签名、域名、网络和一次性证明，防止钱包证明被重复使用 |
| Mint 安全 | Mint 前确认藏品、钱包、用户和队列状态，失败后转入重试或人工处理 |
| 数据库权限 | 关键表和内部函数只允许后端服务角色操作，前端直连不能随意写 |
| 功能开关 | 支持关闭支付、市场、钱包、Mint、后台任务等高风险功能 |
| 定时任务保护 | 后台任务需要密钥校验、功能开关、任务锁和运行记录 |
| 监控和日志 | 记录接口错误、支付错误、Mint 错误、风险事件和后台运行结果 |
| 后台审计 | 记录高风险后台操作，支持风控事件查询和处理 |

## 3. 登录身份保护

用户从 Telegram Mini App 进入项目后，登录不是由前端自己认定的。

代码里能确认的流程是：

- 前端把 Telegram 提供的登录数据交给后端。
- 后端使用 Bot Token 相关校验逻辑验证这份登录数据。
- 后端检查登录数据是否缺失、是否重复、是否过期、是否来自未来时间、签名是否正确。
- 后端确认 Telegram 用户信息有效后，才会创建或刷新项目用户。
- 后端创建会话时，会记录登录凭证的使用情况，避免同一份登录凭证被重复拿来登录。
- 如果登录数据无效、过期或被重复使用，后端会拒绝继续登录。

这里的重点是：前端不能自己决定“这个人是谁”。真正的用户身份由后端验证 Telegram 登录数据后确认。

## 4. 会话安全

登录成功后，项目会创建服务端会话。

代码里能确认的会话保护包括：

- 浏览器拿到的是安全 Cookie。
- 数据库里保存的是会话令牌的哈希，不保存明文令牌。
- Cookie 默认带有安全属性，降低被脚本读取和跨站滥用的风险。
- 后端识别用户时，会从会话中查用户，不相信前端自己传来的用户身份。
- 会话会检查是否过期、是否被撤销、用户是否仍然有效。
- 退出登录时，后端会撤销当前会话，必要时也能撤销全部会话。
- 刷新登录状态时，后端会延长当前会话，而不是让前端自己改登录状态。
- 会话会记录用户设备、访问时间、IP 和浏览器信息的哈希，方便后续安全排查。

这里的重点是：业务接口的用户身份来自后端会话，不来自页面传值。

代码依据：

- `packages/server/src/auth/issueSession.ts`
- `api/auth/_sessionCookies.ts`
- `api/_shared/requireSession.ts`
- `api/auth/refresh.ts`
- `api/auth/logout.ts`
- `packages/server/src/auth/sessionConfig.ts`

## 5. API 入口统一保护

项目的 API 入口有统一包装逻辑。

代码里能确认的保护包括：

- 每次请求会带上请求编号，方便查日志。
- 会统一处理请求方法，不符合的请求会被拒绝。
- 会统一处理跨域。
- 会统一加安全响应头。
- 默认不让接口响应被浏览器缓存。
- 会统一限制请求内容类型和请求大小。
- 请求内容会先做格式校验，不符合要求就返回错误。
- 业务错误和系统错误会分开处理。
- 系统级错误不会把内部细节直接暴露给前端。
- 严重错误会写入日志或外部错误上报工具。
- 被限流的请求会记录运营事件。

这里的重点是：大多数 API 不需要各自重复写基础安全逻辑，而是先经过统一入口保护。

代码依据：

- `api/_shared/handler.ts`
- `api/_shared/parseBody.ts`
- `api/_shared/validate.ts`
- `api/_shared/errors.ts`
- `api/_shared/observability.ts`

## 6. 接口限流

项目已经实现接口限流。

代码里能确认的限流范围包括：

- 全局访问。
- 登录、刷新登录、退出登录。
- 开盒和开盒结果查询。
- Telegram 支付回调。
- 市场创建挂单、改价、取消挂单和购买。

- 钱包连接、钱包证明、钱包同步、Mint 和 Mint 状态查询。

限流可以按不同对象计算，例如用户、会话、IP、Telegram 用户、钱包和自定义对象。实际计数可以使用内存，也可以使用数据库里的共享限流记录。

代码里还做了敏感信息处理。限流和日志里不会直接保留令牌、密钥、签名、Cookie、登录原始数据这类敏感内容。

这里的重点是：高频请求不会只靠前端按钮禁用来控制，后端也会限制。

## 7. 风控决策机制

项目里有独立的风控决策逻辑。

代码里能确认的风控结果包括：

- 放行。
- 需要额外挑战。
- 进入人工复核。
- 拒绝操作。

风控会结合多类信号判断当前动作是否安全：

- 用户和会话是否存在。
- Telegram 登录数据是否异常。
- 请求来源信息是否缺失或可疑。
- 写操作是否缺少重复请求保护。
- 是否已经触发限流。
- 用户是否带有风控标记。
- 当前业务动作本身是否存在异常。

风控不是只看一个条件，而是综合身份、请求、用户状态和业务状态后给结果。

代码依据：

- `packages/server/src/security/riskControl.ts`
- `api/_shared/riskGuards.ts`
- `api/_shared/riskEvents.ts`

项目支持给用户打风控标记。

- 用户是否需要人工复核。

## 10. 业务异常识别

风控逻辑里已经能识别多种业务异常。

### 10.1 开盒相关

代码里会检查盒子是否可用、抽取数量是否异常、支付金额是否和订单匹配。

如果开盒动作不符合当前业务状态，会被拦截或转为复核。

### 10.2 支付相关

代码里会检查支付回调是否缺少支付信息、订单是否存在、回调是否重复、支付环境是否允许、发货功能是否打开。

支付成功但业务发货没有完成时，后台有补偿任务继续处理。

### 10.3 市场相关

代码里会检查用户是否自己买自己的挂单、挂单是否有效、余额是否足够、价格是否明显异常、数量是否有效。

如果价格偏离项目记录的参考范围，会触发风控信号。

### 10.4 库存相关

代码里会检查藏品是否被锁定、是否正在出售、是否正在 Mint、是否已经达到成长上限、是否满足进化要求、是否允许分解。

已经 Mint 的藏品进行分解时，会进入复核判断。

### 10.5 任务和签到相关

代码里会检查任务奖励是否重复领取、签到是否重复领取。

### 10.6 邀请相关

代码里会检查用户是否自己邀请自己，也会识别疑似同来源邀请。

自邀请会被拒绝，疑似刷邀请会被记录或复核。

### 10.7 钱包相关

代码里会检查钱包地址是否合法、钱包证明是否存在、钱包证明是否被重复使用、钱包地址是否被重复绑定、Mint 的藏品是否可以进入链上流程。

### 10.8 Mint 相关

代码里会检查用户、钱包、藏品、模板、锁定状态、队列状态和链上处理状态。

Mint 失败后，后台会按队列继续处理；超过可自动处理范围时，会转入人工处理并记录风险事件。

代码依据：

- `packages/server/src/security/riskControl.ts`
- `api/_shared/mintRiskEvents.ts`
- `api/cron/retry-mint-queue.ts`
- `packages/server/src/ton/mintGuards.ts`
- `packages/server/src/ton/mintQueue.ts`

## 11. 重复请求保护

项目对关键写操作有重复请求保护。

代码里能确认的做法是：

- 钱包连接、Mint、部分交易和支付类操作会使用重复请求记录。
- 同一次业务动作不会因为用户重复点击或网络重试而随意执行多遍。
- 后台会清理过期的重复请求记录。
- 数据库里内部重复请求表不允许前端直接访问。

这里的重点是：重复提交不应该导致重复扣款、重复发奖、重复 Mint 或重复改状态。

## 12. 支付安全支撑

项目里的支付安全主要保护两件事：

- 创建支付订单时，确认当前环境和功能开关允许创建。
- 收到 Telegram 支付回调时，确认回调来源和订单状态可信。

代码里能确认的支付保护包括：

- 生产环境不能使用开发支付模式。
- 缺少服务端支付配置时，支付创建会被拒绝。
- 支付创建和支付发货都受功能开关控制。
- Telegram 回调会校验回调密钥。
- 回调失败、密钥不对、格式异常、订单异常都会记录事件。
- 预支付确认和支付成功处理分开走。
- 支付成功后，是否发货由后端根据订单和功能开关决定。
- 支付发货失败后，后台补偿任务可以继续重试。

这里的重点是：支付结果不能只靠前端页面状态确认，必须由后端回调和订单状态确认。

代码依据：

- `packages/server/src/payments/paymentGuards.ts`
- `api/telegram/webhook.ts`
- `packages/server/src/payments/telegramStars.ts`
- `api/cron/retry-payments.ts`

## 13. 钱包安全支撑

钱包功能里，项目不是简单保存一个用户提交的钱包地址。

代码里能确认的安全保护包括：

- 用户连接钱包时，后端会生成一次性验证内容。
- 钱包证明必须由后端校验。
- 后端会校验钱包地址、签名、域名、网络和时间。
- 代码不会默认相信前端传来的钱包公开密钥。
- 一次性证明不能重复使用。
- 钱包证明异常会记录风险事件。
- 钱包同步功能受功能开关和服务端开关控制。
- 钱包记录不会保存私钥。

这里的重点是：钱包是否属于当前用户，由后端验证签名后确认，不由前端自己声明。

## 14. Mint 安全支撑

Mint 功能涉及用户资产和链上流程，所以代码里做了多层检查。

代码里能确认的保护包括：

- Mint 入口必须先有登录会话。
- Mint 功能受功能开关和服务端开关控制。
- Mint 前会走重复请求保护。
- Mint 前会走风控检查。
- 后端会确认钱包属于当前用户并且已经验证。
- 后端会确认藏品属于当前用户。
- 后端会确认藏品、模板、媒体和集合信息来自数据库，不接受前端随意提交链上字段。
- 后端会确认藏品没有被锁定、没有正在出售、没有已经 Mint 成功，也没有正在队列中处理。
- Mint 队列由后台任务继续推进。
- Mint 重试超过自动处理范围后，会转为人工处理并记录风险事件。

当前代码能确认 Mint 流程保护和队列处理。合约源码文件没有展开实现内容，所以本文不写合约安全结论。

## 15. 数据库权限和内部表保护

项目里有多份数据库权限加固脚本。

代码里能确认的保护包括：

- 前端直连角色主要只能读允许公开读取的数据。
- 关键写操作走后端 API 和数据库内部函数。
- 风险事件、限流记录、重复请求记录、后台审计、系统设置、功能开关、支付回调事件等内部表不允许前端直接读写。
- 内部函数默认撤销公开执行权限，只授予后端服务角色。
- 多个内部表开启行级安全，并对前端角色设置拒绝策略。
- 高风险后台变更会进入后台审计记录。

这里的重点是：项目不是只靠前端隐藏按钮来防止用户改数据，数据库权限层也做了限制。

## 21. 总结

当前项目的风控和安全支撑已经覆盖登录、会话、接口、限流、业务风控、支付回调、钱包证明、Mint 队列、数据库权限、功能开关、后台任务、监控日志和后台审计。

它的核心做法是：

- 入口先验身份。
- 会话由后端掌握。
- 关键写操作先做风控和重复请求保护。
- 支付、钱包和 Mint 不相信前端单方面声明。
- 内部数据不让前端直写。
- 异常会记录，严重问题可以通过功能开关先止损。

所以，这个功能不是单一模块，而是贯穿项目核心业务的一整套安全底座。
