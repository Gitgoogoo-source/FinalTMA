# 第一阶段开发步骤教学文档：基础闭环

本阶段目标是完成一个**能跑通的最小业务闭环**：

用户从 Telegram Mini App 进入 → 服务端验证 Telegram 登录 → 创建用户 → 展示头像和资产栏 → 展示盲盒和藏品 catalog → 用户点击基础开盒 → 后端生成抽卡结果 → 写入库存 → 写入 K-coin / Fgems ledger → 前端展示开盒结果和最新资产。

本阶段不做交易市场、任务、邀请、钱包、NFT Mint、后台管理、正式 Stars webhook。正式 Telegram Stars 支付可以在后续支付阶段接入；但本阶段数据库和订单结构必须先预留支付状态，避免后面重构。Telegram 官方明确要求 Mini App 的 `initData` 要在 bot server 侧验证，不能信任 `initDataUnsafe`；Vercel Functions 可以放在项目 `/api` 目录中运行服务端代码，当前 Vercel 新项目默认 Node.js 24.x；Supabase 暴露 schema 的表应启用 RLS，Postgres 函数也可以通过 Supabase RPC 调用。([Telegram][1])

---

## 1. 第一阶段最终验收标准

完成后，你应该能验证这些结果：

| 验收项         | 结果                                              |
| ----------- | ----------------------------------------------- |
| Telegram 登录 | 用户打开 TMA 后，后端能验证 `initData`，数据库生成用户记录和 session。 |
| 用户资产栏       | 页面顶部能展示头像、K-coin、Fgems、Stars 展示值、钱包入口占位。        |
| Catalog     | 数据库有藏品基础配置、稀有度、系列、形态、图片字段。                      |
| 库存表         | 用户开盒后生成真实 `item_instances`。                     |
| Ledger      | 所有 K-coin / Fgems 变化都有流水记录。                     |
| 基础开盒        | 用户选择盲盒后，后端生成订单、抽卡结果、更新保底、发放藏品。                  |
| 开盒返还        | 开盒后按规则返还 K-coin，例如每抽返还 100 K-coin。              |
| 结果展示        | 前端展示抽到的藏品、稀有度、数量、余额变化。                          |
| 可重复测试       | 新用户、已有用户、单抽、十连、保底、库存写入都能重复验证。                   |

## 2. 第一阶段不做的内容

第一阶段只做基础闭环，不要一开始把所有商业功能混进去。

| 暂不做                       | 原因                                                                                                                                         |
| ------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------ |
| 正式 Telegram Stars webhook | 支付链路复杂，先用开发支付状态跑通订单、抽卡、发奖事务。Telegram Stars 是 Telegram 生态内用于购买数字商品和服务的虚拟物品，正式接入时再处理 invoice、pre-checkout、successful payment。([Telegram][2]) |
| 交易市场                      | 第一阶段只需要库存和资产，交易会引入挂单锁、手续费、并发购买。                                                                                                            |
| 任务 / 邀请                   | 需要等开盒事件稳定后再计算任务进度和邀请奖励。                                                                                                                    |
| TON 钱包 / NFT              | 需要等库存实例稳定后，再做链上 Mint 映射。                                                                                                                   |
| 后台管理                      | 第一阶段先用 seed 数据配置盲盒、概率、藏品。                                                                                                                  |
| 复杂动画                      | 先保证业务闭环正确，再优化开盒动画。                                                                                                                         |

第一阶段的开盒支付建议使用 `DEV_PAID` 模式：用户点击开盒后，后端创建订单，并在开发环境直接把订单处理为已支付，然后执行抽卡事务。后续接入正式 Stars 时，只需要把 `DEV_PAID` 替换为 Telegram Stars webhook 触发处理。

## 4. 开发总顺序

按这个顺序做，不要先写页面：

```txt
第 1 步：初始化项目骨架
第 2 步：配置环境变量
第 3 步：设计并创建第一阶段数据库表
第 4 步：写 seed 数据
第 5 步：写约束、索引、RLS
第 6 步：写核心 RPC
第 7 步：写后端共享包 packages/server
第 8 步：写 Vercel API
第 9 步：写前端认证和 Provider
第 10 步：写资产栏
第 11 步：写开盒页
第 12 步：写结果弹窗
第 13 步：写最小藏品库存页
第 14 步：联调完整闭环
第 15 步：补 SQL 测试、API 测试、E2E 测试
第 16 步：部署到 Vercel，并用 Telegram Mini App 打开验证
```

核心原则：**先数据库，再 RPC，再 API，再前端。**
抽卡、发奖、库存、ledger 都必须在数据库事务内完成。前端只展示结果。

# 5. 详细开发步骤

## 第 1 步：初始化项目骨架

### 你要完成的文件

```txt
package.json
pnpm-workspace.yaml
tsconfig.base.json
.env.example
apps/web/package.json
apps/web/vite.config.ts
apps/web/tsconfig.json
api/_shared/*
packages/domain/*
packages/validation/*


从这里开始
packages/server/*
supabase/config.toml
```

### 你要做的事

先搭建 monorepo。根目录只负责管理工作区，不写业务页面。
`apps/web` 是 Telegram Mini App 前端。
`api` 是 Vercel Functions 后端入口。
`packages/server` 是后端共享工具。
`packages/validation` 是前后端共用参数校验。
`packages/domain` 是业务枚举和状态定义。
`supabase` 是数据库迁移、RPC、seed、测试。

### 这一阶段不要做什么

不要直接在页面里写假数据。
不要在前端计算抽卡结果。
不要先做复杂 UI 动画。
不要把 Supabase service role key 放进前端环境变量。

### 完成标准

本地能启动前端空页面。
API 下能创建一个最简单的健康检查函数。
项目能通过 TypeScript 检查。
`.env.example` 写清楚需要哪些环境变量。

## 第 2 步：配置环境变量

### 你要完成的文件

```txt
.env.example
apps/web/src/env.ts
packages/server/src/env.ts
```

### 前端环境变量

```txt
VITE_API_BASE_URL              # API 基础地址，本地一般是 http://localhost:3000
VITE_TG_BOT_USERNAME           # Telegram Bot 用户名
VITE_TMA_ENV                   # development / staging / production
```

前端只能放公开变量。任何 Bot token、Supabase service role、session secret 都不能出现在 `VITE_` 变量里。

### 后端环境变量

```txt
SUPABASE_URL                   # Supabase 项目地址
SUPABASE_SERVICE_ROLE_KEY      # Supabase service role，只能服务端使用
SUPABASE_ANON_KEY              # 可选，服务端部分只读场景使用
TELEGRAM_BOT_TOKEN             # Telegram Bot token，用于验证 initData
APP_SESSION_SECRET             # 自定义 session 签名密钥
DRAW_RANDOM_SECRET             # 抽卡随机审计用密钥
DEV_GACHA_PAYMENT_MODE         # true / false，第一阶段建议 true
```

### 完成标准

前端启动时能校验公开环境变量。
后端启动时能校验服务端环境变量。
缺少关键环境变量时，应用应该启动失败，而不是运行到一半才报错。

## 第 3 步：创建第一阶段数据库表

第一阶段数据库只做这 5 组：

```txt
core        # 用户、资料、session
economy     # K-coin / Fgems / ledger
catalog     # 藏品配置、稀有度、系列、形态、图片
gacha       # 盲盒、奖励池、保底、开盒订单、结果
inventory   # 用户藏品实例、库存事件
```

### 3.1 创建 `core` 用户结构

### 你要完成的文件

```txt
supabase/migrations/000001_extensions.sql
supabase/migrations/000002_core_users.sql
```

### 表结构责任

| 表                    | 责任                                            |
| -------------------- | --------------------------------------------- |
| `core.users`         | 用户主表；保存内部 user id、Telegram user id、状态、注册时间。   |
| `core.user_profiles` | 用户头像、昵称、语言、Telegram username。                 |
| `core.app_sessions`  | 应用 session；保存 session hash、user id、过期时间、设备信息。 |

### 必须字段

`core.users` 至少要有：

```txt
id
telegram_user_id
status
created_at
updated_at
```

`core.user_profiles` 至少要有：

```txt
user_id
display_name
username
avatar_url
language_code
created_at
updated_at
```

`core.app_sessions` 至少要有：

```txt
id
user_id
session_hash
expires_at
revoked_at
created_at
last_seen_at
```

### 关键约束

`telegram_user_id` 必须唯一。
`user_profiles.user_id` 必须唯一。
session 不能存明文 token，只存 hash。
用户状态要预留 `active`、`blocked`、`risk_limited`。

---

### 3.2 创建 `economy` 资产结构

### 你要完成的文件

```txt
supabase/migrations/000003_economy_ledger.sql
```

### 表结构责任

| 表                         | 责任                                |
| ------------------------- | --------------------------------- |
| `economy.currencies`      | 币种定义，例如 KCOIN、FGEMS。              |
| `economy.user_balances`   | 用户当前余额。                           |
| `economy.currency_ledger` | 不可变资产流水，所有 K-coin / Fgems 变化都写这里。 |

### 必须币种

```txt
KCOIN      # 游戏积分，用于市场购买、合成等
FGEMS      # 游戏材料积分，用于升级、分解获得
```

`Stars` 第一阶段只作为展示或订单预留，不建议放进用户可自由修改余额中。正式 Stars 支付以后续 Telegram 支付回调为准。

### Ledger 必须记录的信息

每一笔流水至少要记录：

```txt
user_id
currency_code
amount
direction              # credit / debit
reason                 # open_box_rebate / admin_grant / upgrade_cost / dev_seed
source_type            # gacha_order / inventory_item / system
source_id
balance_before
balance_after
created_at
```

### 关键约束

余额不能为负。
`user_balances` 的 `user_id + currency_code` 必须唯一。
`currency_ledger` 不允许业务层更新和删除。
所有资产变动都必须先写 ledger，再更新余额，或者在同一事务内一起完成。

---

### 3.3 创建 `catalog` 藏品配置结构

### 你要完成的文件

```txt
supabase/migrations/000004_catalog_collectibles.sql
```

### 表结构责任

| 表                               | 责任                                |
| ------------------------------- | --------------------------------- |
| `catalog.rarities`              | 稀有度配置：Common、Rare、Epic、Legendary。 |
| `catalog.series`                | 藏品系列。                             |
| `catalog.collectible_templates` | 藏品模板，例如角色名称、基础战力、稀有度。             |
| `catalog.collectible_forms`     | 形态配置：低阶、中阶、高阶。                    |
| `catalog.collectible_media`     | 图片资源：大图、小图、头像图。                   |

### 为什么要区分 template 和 instance

`catalog.collectible_templates` 是配置。
`inventory.item_instances` 是用户实际拥有的一份藏品。

例如：

```txt
catalog.collectible_templates:
- 月冕守门人
- 玫影蘑歌者

inventory.item_instances:
- 用户 A 拥有的第 1 个 月冕守门人
- 用户 A 拥有的第 2 个 月冕守门人
- 用户 B 拥有的第 1 个 玫影蘑歌者
```

这样后面才能支持升级、挂售、合成、Mint、编号、链上 NFT。

### 3.4 创建 `gacha` 开盒结构

### 你要完成的文件

```txt
supabase/migrations/000005_gacha_boxes.sql
supabase/migrations/000006_gacha_orders_results.sql
```

### 表结构责任

| 表                          | 责任                  |
| -------------------------- | ------------------- |
| `gacha.blind_boxes`        | 三档盲盒：价格、状态、库存、活动时间。 |
| `gacha.box_price_rules`    | 单抽价格、十连 9 折。        |
| `gacha.drop_pool_versions` | 奖励池版本。              |
| `gacha.drop_pool_items`    | 奖励池内容：藏品、权重、概率、库存。  |
| `gacha.pity_rules`         | 保底规则。               |

### 6.7 `gacha_process_dev_paid_order.sql`

### 责任

第一阶段的核心抽卡 RPC。它模拟“订单已支付”，并在一个事务中完成抽卡、发奖、库存、ledger、保底。

### 输入

```txt
order_id
user_id
```

### 事务内执行顺序

```txt
1. 锁定 draw_orders
2. 校验订单属于当前 user
3. 校验订单未处理
4. 将订单状态改为 processing
5. 锁定 blind_boxes 库存
6. 锁定 user_pity_states
7. 根据订单绑定的 drop_pool_version 获取奖励池
8. 按 draw_count 循环抽卡
9. 每一抽判断是否触发保底
10. 选出奖励 collectible_template
11. 创建 inventory.item_instances
12. 写 inventory.item_instance_events
13. 写 gacha.draw_results
14. 更新 user_pity_states
15. 扣减盲盒库存
16. 按每抽 100 KCOIN 写 economy.currency_ledger
17. 更新 economy.user_balances
18. 将订单状态改为 completed
19. 写 draw_audit
20. 返回完整抽卡结果
```

### 保底处理建议

每一抽之后更新保底计数：

```txt
如果抽到目标稀有度或更高：
    pity_count 重置为 0

如果没有抽到：
    pity_count + 1

如果 pity_count 达到阈值：
    当前抽强制从目标稀有度以上奖励池中抽
    pity_count 重置为 0
```

### K-coin 返还建议

建议第一阶段定义为：

```txt
每抽返还 100 KCOIN
单抽返还 100 KCOIN
十连返还 1000 KCOIN
```

如果你想改成“每笔订单返还 100 KCOIN”，不要写死在 RPC 里，应写入 `economy.reward_rules` 或系统配置。第一阶段可以先固定为每抽 100，后续再配置化。

### 完成标准

订单只能处理一次。
抽卡结果只能生成一次。
十连生成 10 条 `draw_results`。
每个结果都生成一条 `item_instances`。
K-coin 返还有 ledger。
保底进度正确变化。
库存不足时事务整体失败，不产生半成品数据。

# 7. 第一阶段关键错误码

你应该提前设计错误码，不要直接把数据库错误暴露给前端。

| 错误码                       | 场景                      |
| ------------------------- | ----------------------- |
| `AUTH_INIT_DATA_INVALID`  | Telegram initData 验证失败。 |
| `AUTH_SESSION_EXPIRED`    | session 过期。             |
| `USER_BLOCKED`            | 用户被封禁。                  |
| `BOX_NOT_FOUND`           | 盲盒不存在。                  |
| `BOX_NOT_ACTIVE`          | 盲盒未开始、已结束、暂停或售罄。        |
| `BOX_STOCK_NOT_ENOUGH`    | 库存不足。                   |
| `DRAW_COUNT_INVALID`      | 抽数不是 1 或 10。            |
| `ORDER_ALREADY_PROCESSED` | 订单已处理。                  |
| `ORDER_NOT_FOUND`         | 订单不存在或不属于当前用户。          |
| `DROP_POOL_EMPTY`         | 奖励池为空。                  |
| `BALANCE_LEDGER_FAILED`   | 资产流水写入失败。               |
| `INVENTORY_CREATE_FAILED` | 库存写入失败。                 |
| `IDEMPOTENCY_CONFLICT`    | 幂等 key 冲突。              |

# 11. 最容易出错的地方

| 问题                      | 正确做法                                               |
| ----------------------- | -------------------------------------------------- |
| 前端用 `initDataUnsafe` 登录 | 只能把 raw `initData` 发给后端验证。                         |
| 前端计算中奖结果                | 抽卡必须由数据库 RPC 完成。                                   |
| 开盒成功后前端直接加余额            | 前端必须刷新后端资产，不能自己改余额。                                |
| 十连只写一条结果                | 十连必须写 10 条 `draw_results` 和 10 个 `item_instances`。 |
| 没有 ledger               | 所有 K-coin / Fgems 变化必须写 `currency_ledger`。         |
| 没有幂等 key                | 用户重复点击会重复抽卡和发奖。                                    |
| 保底存在全局状态                | 保底必须按 `user_id + box_id` 独立记录。                     |
| 抽卡概率直接存在前端              | 概率展示可以给前端，真实抽取必须使用后端奖励池版本。                         |
| 奖励池没有版本                 | 后续运营改概率后无法解释历史订单。                                  |
| service role 暴露到前端      | service role 只能存在 Vercel API 环境变量中。                |
