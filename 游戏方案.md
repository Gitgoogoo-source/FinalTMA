# PokePets 大世界 RPG 小游戏方案

## 一、游戏定位

### 游戏暂定名称

**PokePets：无尽大陆**

### 一句话玩法

玩家选择自己拥有的三只藏品作为队伍，在由多个地图区块机械拼接而成的大陆上移动，击败地图中的全部巡游怪物和区域 Boss，解锁下一张地图，直到挑战最终传奇区域。

### 产品定位

这不是完整的 Pokémon RPG，而是一个：

* 有地图移动感
* 有宠物遭遇感
* 有区域推进感
* 有大世界连续感
* 战斗规则极简
* 不产生新宠物
* 不改变真实资产
* 不新增后端业务

的轻量前端 RPG。

核心体验不是复杂战斗，而是：

> 使用自己已有的藏品，不断探索更高等级地图，挑战战斗力更高的怪物。

---

# 二、必须明确的技术边界

玩家的真实藏品和战斗力目前来自现有库存数据。

项目现有库存条目已经包含：

* 藏品名称
* 图片
* 稀有度
* 进化阶段
* 战斗力
* 可用数量

因此，可以直接复用现有的 `inventory.list` 查询，不需要新增 API、数据库表或 RPC。

但需要准确区分：

* **无需新增后端**
* 不等于完全没有 API 请求

要读取玩家真实藏品，就必须复用已有库存查询。

如果要求绝对零 API，则只能使用固定试玩宠物，无法读取玩家真实藏品。

本方案默认：

> 复用现有只读库存接口，但不新增任何后端功能。

---

# 三、整体游戏流程

```text
进入“游戏”页面
        ↓
点击“大世界冒险”
        ↓
自动读取玩家可用藏品
        ↓
自动选择战斗力最高的3个藏品
        ↓
玩家可调整队伍和出战顺序
        ↓
进入第1张地图
        ↓
自由移动，碰触巡游怪物
        ↓
自动进入战斗
        ↓
击败地图内全部普通怪物
        ↓
Boss区域开放
        ↓
击败地图Boss
        ↓
通往下一地图的出口开放
        ↓
进入下一地图
        ↓
不断循环，直到最终传奇地图
```

每次战斗结束后，玩家三只宠物全部恢复满血。

地图内部不存在持续损耗，因此游戏重点是：

* 探索移动
* 找到怪物
* 判断当前战斗力是否足够
* 组建更强的三宠队伍
* 不断解锁更高地图

# 四、地图系统设计

## 4.1 只开发一个地图模型

整个游戏只开发一套固定地图结构。

建议地图尺寸：

```text
18列 × 28行
```

或者使用固定像素坐标：

```text
1080 × 1620
```

地图包含以下固定位置：

* 玩家出生点
* 地图出口
* 三条怪物巡逻路线
* 一个 Boss 区域
* 固定墙体和障碍物
* 固定道路
* 固定边界碰撞
* 固定入口和出口连接点

所有地图共享：

* 完全相同的碰撞区域
* 完全相同的道路结构
* 完全相同的怪物路线
* 完全相同的 Boss 位置
* 完全相同的出口位置

只替换：

* 地图背景图
* 环境装饰图
* 天气或粒子效果
* 怪物藏品模板
* Boss藏品模板
* 地图名称
* 背景音乐，可暂不开发

---

## 4.2 地图皮肤必须遵守同一结构

地图背景图不能完全自由绘制。

所有皮肤必须保证以下位置一致：

* 道路位置一致
* 山体、河流、树木等阻挡区域一致
* 出生点一致
* 怪物巡逻区一致
* Boss区域一致
* 出口一致

例如同一个障碍区域：

| 地图皮肤 | 障碍表现 |
| ---- | ---- |
| 草原地图 | 树林   |
| 雪原地图 | 冰山   |
| 沙漠地图 | 岩石   |
| 火山地图 | 岩浆   |
| 遗迹地图 | 破损建筑 |

视觉不同，但实际碰撞模型相同。

这是整个低成本地图系统能够成立的关键。

# 五、大世界拼接机制

## 5.1 地图不是独立关卡弹窗

玩家击败 Boss 后，不进入传统的“关卡选择页面”。

而是直接从当前地图出口移动出去，经过一个短暂的过渡动画，进入下一张地图。

视觉表现：

```text
地图01 → 地图02 → 地图03
                      ↓
地图06 ← 地图05 ← 地图04
↓
地图07 → 地图08 → 地图09
                      ↓
地图12 ← 地图11 ← 地图10
```

地图在世界地图上采用蛇形拼接。

这让玩家感觉自己在持续向大陆深处移动，而不是逐关点击。

---

## 5.2 地图坐标

每张地图拥有世界坐标：

```ts
interface WorldMapCoordinate {
  worldX: number;
  worldY: number;
}
```

例如：

```text
地图01：0,0
地图02：1,0
地图03：2,0
地图04：2,1
地图05：1,1
地图06：0,1
地图07：0,2
```

玩家离开当前地图时：

1. 当前地图向相反方向滑出。
2. 下一张地图滑入。
3. 玩家从对应入口出现。
4. 世界小地图点亮新区域。

虽然实际上只是替换背景图和配置数据，但视觉上会像在一个完整大陆中移动。

---

## 5.3 世界小地图

游戏顶部显示简化世界地图：

```text
●—●—●
      │
●—●—●
│
●—○—○
```

状态含义：

* 实心点：已完成
* 高亮点：当前地图
* 空心点：尚未进入
* 锁定点：尚未开放

世界小地图非常重要。

它负责让机械重复的小地图产生“大陆推进感”。

# 六、单张地图结构

每张地图建议包含：

* 3只普通巡游怪物
* 1只地图 Boss
* 1个出口
* 3条固定巡逻路线

## 地图流程

```text
玩家进入地图
   ↓
3只普通怪物在固定路线移动
   ↓
玩家可以自由选择攻击顺序
   ↓
击败全部3只普通怪物
   ↓
Boss屏障解除
   ↓
击败Boss
   ↓
地图出口开放
   ↓
进入下一地图
```

地图顶部显示：

```text
暮色丘陵
地图 03 / 12

普通怪物：2 / 3
Boss：未开放
```

击败全部普通怪物后：

```text
普通怪物：3 / 3
Boss：可以挑战
```

---

# 七、巡游怪物机制

## 7.1 地图怪物展示

怪物直接使用现有藏品模板图片。

建议使用缩略图资源，而不是详情大图。

项目现有藏品已经包含缩略图和详情图路径。

怪物在地图中的视觉尺寸：

```text
48 × 48
或
56 × 56
```

增加简单效果：

* 上下浮动
* 左右翻转
* 移动时轻微缩放
* 稀有怪物外圈光效

不需要制作行走帧动画。

---

## 7.2 固定巡逻路线

每个地图模型预设三条路线：

```ts
const patrolRoutes = {
  routeA: [
    { x: 3, y: 5 },
    { x: 8, y: 5 },
    { x: 8, y: 9 },
    { x: 3, y: 9 },
  ],

  routeB: [
    { x: 11, y: 8 },
    { x: 15, y: 8 },
    { x: 15, y: 14 },
    { x: 11, y: 14 },
  ],

  routeC: [
    { x: 4, y: 18 },
    { x: 12, y: 18 },
    { x: 12, y: 23 },
    { x: 4, y: 23 },
  ],
};
```

怪物循环移动。

建议速度：

```text
每500～700毫秒移动一个格子
```

玩家与怪物发生碰撞后：

1. 地图暂停。
2. 怪物停止移动。
3. 打开战斗界面。
4. 战斗结束后返回原地图。
5. 胜利则删除该怪物。
6. 失败则怪物继续存在。

---

# 八、Boss机制

Boss固定站在地图出口前方。

在普通怪物未全部被击败时：

* Boss周围有屏障
* 玩家无法进入Boss区域
* 点击或靠近时显示：

```text
击败本区域全部巡游怪物后，才能挑战守卫。
```

Boss基础属性继续使用藏品战斗力公式。

为了体现Boss身份，只增加一个简单修正：

```text
Boss最大生命值 = 基础生命值 × 1.5
Boss攻击力 = 基础攻击力
```

不增加：

* Boss技能
* 多阶段机制
* 属性攻击
* 状态异常

Boss只是更耐打，不引入额外系统。

---

# 九、玩家队伍机制

## 9.1 自动读取玩家藏品

进入小游戏时，读取玩家当前可用藏品。

只允许以下状态的藏品进入游戏：

```text
available > 0
```

不允许使用：

* 出售中的藏品
* 交易中的藏品
* 远征中的藏品
* Mint中的藏品

现有库存模型已经区分这些数量。

---

## 9.2 自动选择三只

系统默认选择玩家可用藏品中战斗力最高的三个单位。

如果同一模板有多个可用数量，可以重复选择，但不能超过该模板的可用数量。

例如：

```text
赤曜火狐 ×2
玄岳巨龟 ×1
```

可以组成：

```text
赤曜火狐
赤曜火狐
玄岳巨龟
```

玩家进入地图前可以手动调整。

---

## 9.3 不足三只的情况

如果玩家只有0～2只可用藏品：

```text
至少需要3只可用藏品才能进入大世界冒险。
```

按钮：

```text
前往开盒
前往藏品
```

不使用系统赠送的试玩宠物，避免产生资产归属误解。

---

# 十、战斗数值

严格按照你的公式。

## 玩家宠物

```ts
maxHp = 50 + combatPower;
attack = Math.floor(15 + combatPower / 3);
```

## 普通敌人

```ts
maxHp = 50 + combatPower;
attack = Math.floor(15 + combatPower / 3);
```

## Boss

```ts
maxHp = Math.floor((50 + combatPower) * 1.5);
attack = Math.floor(15 + combatPower / 3);
```

例如：

|   战斗力 |    HP | 攻击力 |
| ----: | ----: | --: |
|   100 |   150 |  48 |
|   300 |   350 | 115 |
|   600 |   650 | 215 |
| 1,000 | 1,050 | 348 |
| 2,400 | 2,450 | 815 |

这套公式有一个明显特点：

> 同等战斗力的战斗，大约三至四轮就能结束。

这是适合小游戏的，因为战斗不会拖得太久。

---

# 十一、战斗方式

## 推荐使用自动战斗

由于宠物只有：

* HP
* 攻击力

没有技能、属性、防御或状态，手动设置“攻击按钮”没有实际决策意义。

如果每回合只能点击一次攻击，玩家只是在机械点击。

因此建议：

> 碰到怪物后，自动进行回合制战斗。

---

## 11.1 战斗流程

玩家三只宠物按照队伍顺序出战。

```text
第1只宠物上场
↓
双方同时攻击
↓
任意一方HP归零
↓
玩家宠物倒下则第2只自动上场
↓
敌人倒下则战斗胜利
```

普通怪物每场只有一只。

玩家最多可以连续使用三只宠物。

---

## 11.2 同时攻击

每轮双方同时造成伤害：

```ts
playerHp -= enemyAttack;
enemyHp -= playerAttack;
```

每轮间隔：

```text
0.7～1秒
```

战斗通常持续：

```text
3～8秒
```

这样可以快速回到地图探索。

---

## 11.3 同时倒下

如果玩家宠物和敌人同时归零：

* 敌人判定被击败
* 玩家进入下一只宠物
* 如果这是玩家最后一只宠物，则仍判定玩家胜利

这样可以减少因为结算顺序产生的争议。

---

## 11.4 自动恢复满血

每次战斗结束后：

```ts
team.forEach((pet) => {
  pet.currentHp = pet.maxHp;
});
```

无论胜利或失败，下一次战斗开始前全部恢复满血。

因此不需要：

* 治疗道具
* 恢复点
* 营地
* 药水
* 生命管理
* 战斗外状态保存

这会显著降低开发复杂度。

---

# 十二、战斗失败

如果玩家三只宠物全部倒下，而敌人仍有生命：

1. 关闭战斗界面。
2. 玩家返回当前地图出生点。
3. 玩家宠物全部恢复满血。
4. 已击败的地图怪物保持已击败状态。
5. 当前失败怪物继续存在。
6. 玩家可以重新挑战或退出更换队伍。

提示：

```text
当前队伍战斗力不足。

建议：
· 更换战斗力更高的藏品
· 进化现有藏品
· 调整出战顺序
```

不要重置整张地图，否则重复移动会造成明显挫败。

# 十四、每张地图如何选择怪物

每张地图配置：

* 3只普通怪物
* 1只Boss

## 普通怪物

从该地图战斗力区间中选择三只不同模板。

例如地图01：

```text
怪物A：战斗力101
怪物B：战斗力105
怪物C：战斗力109
```

## Boss

选择本地图区间内战斗力最高的模板。

例如：

```text
Boss战斗力：110
Boss HP倍率：1.5
```

这样不需要单独设计怪物等级。

怪物难度只由正式模板的战斗力决定。

# 十五、地图解锁和进度保存

因为第一版没有真实奖励，可以将进度保存在前端 `localStorage`。

保存内容：

```ts
interface WorldRpgProgress {
  selectedTeam: SelectedPet[];
  currentMapId: string;
  currentPlayerPosition: Position;
  defeatedEnemyIds: string[];
  defeatedBossIds: string[];
  unlockedMapIds: string[];
  completedMapIds: string[];
  highestCompletedMap: number;
  updatedAt: string;
}
```

重新进入游戏时显示：

```text
继续冒险
当前地图：黑曜火山
进度：地图 09 / 12

重新开始
```

需要明确：

* 本地进度可以被清除
* 不跨设备同步
* 不代表真实资产
* 不用于发放奖励

因为没有经济奖励，所以前端保存被修改也不会造成资产风险。

# 十六、游戏页面嵌入方式

当前游戏页面由远征和转盘组成。

建议增加第三个入口，并放在最上方：

```tsx
<div className="game-stack">
  <WorldRpgPanel />
  <ExpeditionPanel />
  <WheelPanel />
</div>
```

未进入时显示卡片：

```text
无尽大陆

使用你的3只藏品探索连续大地图，
击败巡游怪物和区域Boss。

当前进度：地图 05 / 12

[继续冒险]
```

首次进入显示：

```text
[开始冒险]
```

---

## 进入游戏后的页面状态

点击后仍然停留在 `/game`。

但将远征和转盘隐藏，只显示：

```text
地图名称
世界小地图
剩余怪物
游戏地图
方向控制
退出按钮
```

推荐结构：

```tsx
export function GamePage() {
  const [worldMode, setWorldMode] = useState(false);

  return worldMode ? (
    <WorldRpgGame onExit={() => setWorldMode(false)} />
  ) : (
    <main className="page game-page">
      <WorldRpgPanel onStart={() => setWorldMode(true)} />
      <ExpeditionPanel />
      <WheelPanel />
    </main>
  );
}
```

不需要新增独立路由。

# 十七、移动控制

手机端建议提供两种移动方式。

## 第一版必须有

左下角虚拟方向键：

```text
    ↑
←       →
    ↓
```

每次按下移动一个格子。

支持长按连续移动。

## 桌面调试支持

同时支持：

```text
W A S D
方向键
```

## 不建议第一版做

* 点击地图自动寻路
* 摇杆模拟
* 八方向移动
* 平滑物理移动
* 复杂碰撞

使用四方向格子移动最稳定。


## 地图中央

显示：

## 地图底部

```text

```

* 图片

# 二十四、最关键的设计结论

这个游戏的“大世界感”不来自地图本身有多复杂，而来自四件事：

1. 地图之间没有传统关卡选择页面。
2. 玩家从当前地图边缘直接进入下一地图。
3. 世界小地图持续显示已探索区域。
4. 已击败的怪物和已完成地图保持完成状态。

因此即使所有地图使用相同结构，只要：

* 背景皮肤明显不同
* 怪物种类不同
* 入口出口方向交替
* 地图过渡连续
* 世界坐标持续推进

玩家仍然会感觉自己正在穿越一片完整大陆。

## 最终版本定义

第一版定为：

```text
12张地图
1套地图碰撞模型
12张背景皮肤
每张地图3只巡游怪物
每张地图1只Boss
玩家使用3只真实藏品
只读取战斗力
自动战斗
每战满血恢复
从普通逐步提升到传说
无收服
无技能
无等级
无真实奖励
不新增后端
```

这套方案已经足够形成一个具有 Pokémon 式探索感、但开发复杂度显著更低的定制大世界 RPG 小游戏。
