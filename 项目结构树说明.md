下面方案按界面和功能拆成：**资产、开盒、交易、藏品、图鉴、任务、钱包、支付、链上 NFT、后台运营**。核心原则保持不变：**前端只展示和发起请求；抽卡、支付确认、库存、交易、积分、保底、任务奖励、合成、分解、Mint 全部以后端与数据库事务为准。**

## 1. 技术栈定稿

Vercel 当前文档显示 Node.js 24.x 是默认可用版本，且 Node.js Runtime 可在项目 `/api` 目录中构建 TypeScript / JavaScript Functions；Supabase 适合承担 Postgres 数据库、RLS、Storage、RPC；Telegram Mini App 的 `initData` 必须在服务端验证，不能信任 `initDataUnsafe`；Telegram Stars 数字商品支付需要走 Bot Payments API，并使用 `XTR`；TON Connect 是 TON 钱包连接标准协议，可连接地址、请求签名和发起交易，同时不会接触用户私钥。([Vercel][1])

| 层级         | 选型                                             | 说明                                                                              |
| ---------- | ---------------------------------------------- | ------------------------------------------------------------------------------- |
| 前端         | React + Vite + TypeScript                      | Telegram Mini App 本质是 WebView，Vite 更轻，适合小游戏、交易市场、抽卡动画。                          |
| 前端状态       | TanStack Query + Zustand                       | Query 负责服务端数据；Zustand 只存 UI 临时状态，比如当前 tab、筛选条件、弹窗状态。                            |
| TMA 能力     | `@tma.js/sdk-react` / Telegram WebApp JS       | 读取主题、平台、initData、分享、返回按钮、主按钮等。                                                  |
| UI         | Tailwind CSS + 自研组件库                           | 你的界面是高拟物卡片风格，建议自研 `Card / BottomNav / AssetPill / Sheet / Modal`。               |
| 后端 API     | Vercel Functions，Node.js 24，部署在 `/api`         | 所有请求先验证 Telegram session，再调用 Supabase RPC。                                      |
| 数据库        | Supabase Postgres + RLS + RPC + Storage        | 所有核心业务用 Postgres 事务和 RPC 封装。                                                    |
| 支付         | Telegram Stars Bot Payments API                | 开盒支付用 Stars，成功回调后才发放藏品和 K-coin。                                                 |
| 钱包         | TON Connect                                    | 只保存公开地址和验证结果，不保存私钥。                                                             |
| NFT 合约     | 新项目建议 Tolk / FunC；Tact 仅作为可选或遗留方案              | TON 官方文档目前提示 Tact 正在被弃用，官方智能合约语言是 Tolk；如果你坚持用 Tact，建议把合约层隔离，方便后续替换。([TON文档][2]) |
| TON NFT 标准 | TEP-62 NFT Collection / Item，TEP-66 Royalty 可选 | TON NFT 标准文档定义了 NFT Item 转移、归属、Collection get 方法和 royalty 参数。([TON文档][3])       |
| 后台管理       | 独立 `apps/admin`                                | 配置盲盒、概率、库存、藏品、任务、活动 banner、市场手续费、风控。                                            |
| 测试         | Vitest + Playwright + pgTAP / SQL tests        | 前端交互、API、数据库事务、抽卡概率、市场并发都要测。                                                    |
| 监控         | Vercel Logs + Supabase Logs + Sentry           | 重点监控支付回调、抽卡、交易、钱包同步、RPC 失败。                                                     |

补充：Telegram 对 Mini App 的区块链能力有 TON / TON Connect 相关要求；因此链上 NFT、钱包连接、Mint、同步都应围绕 TON Connect 和 TON NFT 设计。([Telegram][4])

## 2. 总体架构

```txt
Telegram Mini App 前端
    ↓
Vercel API Functions
    ↓
Supabase RPC / Postgres Transaction
    ↓
Ledger / Inventory / Market / Gacha / Task / NFT Tables

Telegram Stars Payment Webhook
    ↓
Vercel /api/telegram/webhook
    ↓
验证订单、去重、处理 successful_payment
    ↓
RPC：发放抽卡结果 + 返还 K-coin + 写入 ledger

TON Connect
    ↓
用户钱包签名 / 交易
    ↓
Vercel API 验证 ton_proof / 查询链上状态
    ↓
Supabase 同步钱包、NFT、Mint 队列、链上交易
```

关键边界：

前端不计算中奖结果，不扣余额，不改库存，不判断任务完成，不直接改市场挂单。
Vercel API 不直接拼业务结果，只做认证、参数校验、调用 RPC、处理外部回调。
Supabase RPC 是核心业务层，负责事务、锁、幂等、余额、库存、保底、奖励、审计。

## 5. 后端 API 职责说明

| API 模块                 | 负责事项                                                                   |
| ---------------------- | ---------------------------------------------------------------------- |
| `api/auth`             | 校验 Telegram `initData`，创建用户，发行短期 session。                              |
| `api/me`               | 聚合首屏资产、头像、钱包状态、红点、任务提醒。                                                |
| `api/boxes`            | 查询盲盒、创建开盒订单、查询开盒结果。                                                    |
| `api/telegram/webhook` | 接收 Telegram Stars 支付回调，处理 `pre_checkout_query` 和 `successful_payment`。 |
| `api/market`           | 市场列表、商品详情、购买、出售、改价、下架。                                                 |
| `api/inventory`        | 库存、升级、合成、分解、库存操作历史。                                                    |
| `api/album`            | 图鉴进度、图鉴奖励、排行榜。                                                         |
| `api/tasks`            | 任务列表、签到、任务奖励领取、邀请统计。                                                   |
| `api/wallet`           | TON 地址保存、ton_proof 验证、链上 NFT 同步、Mint 队列。                               |
| `api/admin`            | 后台配置，必须加管理员权限、审计日志、二次确认。                                               |
| `api/cron`             | 过期挂单、榜单刷新、Mint 重试、市场统计。                                                |

后端 API 的统一规则：

每个写操作都必须带 `idempotency_key`。
每个 API 从 session 取 `user_id`，不相信 body 里的 `user_id`。
每个核心写操作调用一个数据库 RPC。
支付、购买、出售、合成、分解、任务领取必须在数据库事务中完成。
所有外部回调先落库，再处理，避免丢事件。

## 6. Supabase 数据库结构规划

建议把核心表按 schema 拆分，不要把所有表都堆在 `public`。如果前端需要直接读取某些公开数据，可以暴露只读 view；核心交易表仍由 API / RPC 访问。

### 6.1 `core`：用户、会话、钱包基础

| 表                    | 责任                                                |
| -------------------- | ------------------------------------------------- |
| `core.users`         | 用户主表，绑定 Telegram user id、状态、风控状态、注册来源。            |
| `core.user_profiles` | 头像、昵称、语言、地区、展示信息。                                 |
| `core.app_sessions`  | TMA session，记录过期时间、设备、IP hash、Telegram auth_date。 |
| `core.user_wallets`  | TON 钱包公开地址、连接状态、最后验证时间。                           |
| `core.wallet_proofs` | ton_proof challenge、签名、验证结果、防重放。                  |
| `core.user_flags`    | 封禁、限购、提现限制、风控标签。                                  |
| `core.notifications` | 用户操作结果、系统通知、红点消息。                                 |

关键约束：

`core.users.telegram_user_id` 唯一。
`core.user_wallets.user_id + ton_address` 唯一。
session 过期后必须重新验证 Telegram initData。

### 6.2 `economy`：K-coin、Fgems、余额、流水

| 表                             | 责任                                   |
| ----------------------------- | ------------------------------------ |
| `economy.currencies`          | 币种定义：`KCOIN`、`FGEMS`、`STAR_DISPLAY`。 |
| `economy.user_balances`       | 用户当前可用余额和锁定余额。                       |
| `economy.currency_ledger`     | 不可变流水表，所有资产变化都写这里。                   |
| `economy.balance_locks`       | 市场购买、挂单、活动等造成的锁定余额。                  |
| `economy.reward_rules`        | 开盒返 100 K-coin、任务奖励、分解奖励等通用奖励规则。     |
| `economy.fee_rules`           | 平台手续费，例如市场 5%。                       |
| `economy.reconciliation_runs` | 对账任务记录。                              |

设计要点：

`user_balances` 是查询缓存，`currency_ledger` 是真实账本。
任何 K-coin / Fgems 变化必须有 ledger。
市场手续费、任务奖励、开盒返利都要写 `source_type` 和 `source_id`，方便对账。
如果顶部 Stars 是 Telegram 官方 Stars，不能把前端显示值当作真实平台余额；真实扣款以 Telegram invoice 和 `successful_payment` 为准。

### 6.4 `gacha`：盲盒、奖励池、概率、保底、开盒订单

| 表                          | 责任                                  |
| -------------------------- | ----------------------------------- |
| `gacha.blind_boxes`        | 三档盲盒：价格、状态、库存、活动时间。                 |
| `gacha.box_price_rules`    | 单抽价格、十连 9 折、活动折扣。                   |
| `gacha.drop_pool_versions` | 奖励池版本，概率变更后创建新版本，历史结果可追溯。           |
| `gacha.drop_pool_items`    | 每个奖励项：藏品模板、权重、概率、库存、是否限定。           |
| `gacha.pity_rules`         | 每款盲盒独立保底规则：次数、目标稀有度、重置条件。           |
| `gacha.user_pity_states`   | 用户在每款盲盒上的当前保底进度。                    |
| `gacha.draw_orders`        | 开盒订单：单抽 / 十连、支付状态、Stars 金额。         |
| `gacha.draw_results`       | 每次抽卡结果：奖励、是否命中保底、生成的 item instance。 |
| `gacha.draw_audit`         | 抽卡审计：池版本、随机种子 hash、规则快照。            |

设计要点：

每款盲盒独立保底：`user_id + box_id` 唯一。
盲盒状态包括：`not_started`、`active`、`ended`、`sold_out`、`paused`。
奖励池必须有版本号，防止运营改概率后历史订单无法解释。
抽卡结果必须在支付成功后生成，不能在支付前发放。
十连订单应生成 10 条 `draw_results`，不是一条模糊结果。

### 6.5 `inventory`：用户藏品实例、锁定、升级、合成、分解

| 表                                | 责任                            |
| -------------------------------- | ----------------------------- |
| `inventory.item_instances`       | 用户真实拥有的每一份藏品。                 |
| `inventory.inventory_locks`      | 挂售、合成、Mint 等操作锁定藏品，防止重复使用。    |
| `inventory.item_instance_events` | 获得、升级、合成、失败返还、分解、挂售、下架、转移记录。  |
| `inventory.upgrade_rules`        | 不同等级 / 稀有度升级消耗 Fgems 和战力增长。   |
| `inventory.upgrade_logs`         | 升级记录。                         |
| `inventory.evolution_rules`      | 3 份相同藏品 + K-coin 合成到下一形态，成功率。 |
| `inventory.evolution_attempts`   | 合成尝试记录，成功 / 失败 / 返还主藏品。       |
| `inventory.decompose_rules`      | 分解规则：稀有度、等级、形态对应 Fgems 奖励。    |
| `inventory.decompose_logs`       | 分解记录。                         |

设计要点：

建议每一份藏品都是一条 `item_instances`，即使是 Common，也不要只做数量字段。这样能支持挂售、升级、合成、Mint、编号、链上 NFT。
藏品状态建议包括：`available`、`locked`、`listed`、`consumed`、`decomposed`、`minting`、`minted`、`transferred`。
合成失败时返还等级最高的主藏品，其余两份和 K-coin 消耗不返还，这个规则必须写在 RPC 里。
挂售中的藏品不能升级、合成、分解、Mint。

### 6.6 `market`：购买、出售、报价管理、手续费、价格健康

| 表                           | 责任                              |

| `market.orders`             | 成交订单：买家、卖家、价格、手续费、到账金额。         |
| `market.order_items`        | 成交的具体藏品实例。                      |
| `market.listing_events`     | 上架、改价、下架、成交、过期事件。               |
| `market.price_snapshots`    | 地板价、最近成交、均价、价格区间。               |
| `market.depth_snapshots`    | 市场深度：某藏品不同价位挂单数量。               |
| `market.price_health_rules` | 价格健康判断：过高、过低、正常。                |
| `market.fee_settlements`    | 平台手续费结算。                        |

设计要点：

上架时锁定 `item_instances`。
购买时同时完成：买家扣 K-coin、卖家到账、平台手续费入账、藏品转移、挂单状态更新。
并发购买同一个挂单时，数据库必须锁住 listing 和 listing_items。
改价只改 active listing；已成交或下架不可改价。
下架后释放藏品锁，状态回到 available。

### 6.7 `payments`：Telegram Stars 支付

| 表                                  | 责任                                                      |
| ---------------------------------- | ------------------------------------------------------- |
| `payments.star_orders`             | Stars 支付订单：业务类型、金额、状态、payload。                          |
| `payments.star_invoices`           | invoice 信息，关联 Telegram invoice payload。                 |
| `payments.star_payments`           | `successful_payment` 记录，含 `telegram_payment_charge_id`。 |
| `payments.telegram_webhook_events` | 原始 webhook 事件，防止丢失和重复。                                  |
| `payments.star_refunds`            | 退款记录。                                                   |
| `payments.payment_disputes`        | 用户支付争议、客服处理状态。                                          |

设计要点：

`telegram_payment_charge_id` 必须唯一。
只有收到 `successful_payment` 才能发放开盒结果。Telegram 文档也明确指出，不能仅凭 `pre_checkout_query` 就交付数字商品。([Telegram][5])
支付 webhook 要先落库，再调用 RPC 处理。
开盒订单、支付记录、抽卡结果必须能一一对应。

### 6.8 `tasks`：邀请、签到、每日任务、社交任务

| 表                            | 责任                           |
| ---------------------------- | ---------------------------- |
| `tasks.task_definitions`     | 任务模板：每日、社交、交易、链上任务。          |
| `tasks.task_periods`         | 周期任务的周期 key，例如每日、每周。         |
| `tasks.user_task_progress`   | 用户任务进度。                      |
| `tasks.task_claims`          | 奖励领取记录，防重复领取。                |
| `tasks.signin_campaigns`     | 7 日签到活动配置。                   |
| `tasks.user_signins`         | 用户签到记录。                      |
| `tasks.referrals`            | 邀请关系，invitee 只能绑定一个 inviter。 |
| `tasks.referral_rewards`     | 双方 500 K-coin 首开盒奖励。         |
| `tasks.referral_commissions` | 好友开盒 10% 分红记录。               |
| `tasks.share_events`         | Telegram 分享、复制链接、群分享记录。      |

设计要点：

邀请关系不可随意变更。
双方 500 K-coin 只在被邀请人完成首次开盒后发放。
10% 分红建议按 K-coin 奖励或开盒返利计算，不建议按 Telegram Stars 直接分账，避免支付和合规复杂度。
任务领取必须以 `task_claims` 唯一约束防重。

### 6.9 `album`：图鉴、里程碑、排行榜

| 表                           | 责任                         |
| --------------------------- | -------------------------- |
| `album.books`               | 图鉴册：全系列、阵营、稀有度、活动册。        |
| `album.book_items`          | 每个图鉴册包含哪些藏品模板。             |
| `album.user_discoveries`    | 用户首次获得某藏品模板的记录。            |
| `album.milestones`          | 收集数量里程碑奖励。                 |
| `album.milestone_claims`    | 用户领取记录。                    |
| `album.weekly_leaderboards` | 周榜定义。                      |
| `album.leaderboard_entries` | 周榜条目：排名、完成度、稀有数、Mint 数、分数。 |
| `album.score_rules`         | 榜单积分规则。                    |

设计要点：

图鉴进度最好基于 `user_discoveries`，而不是当前库存。用户分解或出售后，不应丢失已收集记录。
排行榜建议每日或定时快照，避免实时计算太重。
图鉴奖励领取要防重复，按里程碑唯一。

| `onchain.nft_items`            | 链上 NFT Item 地址、index、owner、关联 item_instance。 |

链上 NFT 是藏品的外部映射，数据库库存仍要记录。

链上同步必须能处理延迟、失败、重复回调。
如果 NFT 已转出钱包，要更新链上持有状态，但游戏内是否仍可用要有明确规则。

## 8. 关键业务流程

### 8.1 Telegram 登录流程

1. 前端启动后读取 Telegram `initData`。
2. 前端调用 `/api/auth/telegram`。
3. 后端验证 `initData`。
4. 后端创建或更新 `core.users`。
5. 后端发行短期 session。
6. 前端之后所有 API 都带 session。
7. API 只认 session 中的 `user_id`。

---

### 8.2 Stars 支付开盒流程

1. 用户选择盲盒和次数。
2. 前端调用 `/api/boxes/create-open-order`。
3. 后端调用 `gacha_create_order` 创建订单，生成 invoice payload。
4. 用户通过 Telegram Stars 支付。
5. Telegram webhook 发送 `pre_checkout_query`。
6. 后端校验订单状态、库存、活动状态，并确认或拒绝。
7. Telegram webhook 发送 `successful_payment`。
8. 后端落库 `payments.star_payments`。
9. 后端调用 `gacha_process_paid_order`。
10. RPC 在一个事务中完成：扣库存、抽卡、更新保底、创建 item instance、返还 100 K-coin、写 ledger、写 draw_results。
11. 前端查询 `/api/boxes/result` 并展示结果。

---

### 8.3 购买藏品流程

1. 前端展示市场挂单。
2. 用户点击购买。
3. `/api/market/buy` 校验 session 和参数。
4. RPC 锁定 listing、listing_items、买家余额、卖家余额。
5. 校验挂单仍有效、买家 K-coin 足够、藏品仍被锁定。
6. 扣买家 K-coin。
7. 卖家收到扣除手续费后的 K-coin。
8. 平台手续费入账。
9. item instance owner 改为买家。
10. listing 状态更新为 sold 或部分成交。
11. 所有 ledger、order、event 一起提交。

---

### 8.4 出售藏品流程

1. 前端读取 `sellable-items`。
2. 用户选择藏品、数量、价格。
3. 前端展示建议区间和预计到手。
4. 用户确认出售。
5. RPC 校验这些 item instance 属于用户、状态为 available、未锁定。
6. RPC 创建 listing，并将 item instance 状态改为 listed / locked。
7. 挂单进入市场。

---

### 8.5 合成流程

1. 用户选择 3 份相同藏品。
2. 前端调用 `/api/inventory/evolve`。
3. RPC 锁定 3 个 item instance 和用户 K-coin。
4. 校验是否满足同模板、同形态、可合成。
5. 扣 K-coin。
6. 根据成功率计算结果。
7. 成功：消耗 3 个旧实例，创建下一形态实例。
8. 失败：保留等级最高的主藏品，其余 2 个消耗，K-coin 不返还。
9. 写入 `inventory.evolution_attempts`、`item_instance_events`、`currency_ledger`。

---

### 8.6 分解流程

1. 用户选中重复藏品。
2. RPC 校验不是挂售、不是 Mint 中、不是锁定状态。
3. 根据 `decompose_rules` 计算 Fgems。
4. 藏品状态改为 decomposed。
5. 增加 Fgems ledger。
6. 返回奖励结果。

---

### 8.7 邀请任务流程

1. 邀请人生成 referral link。
2. 被邀请人通过 `/start payload` 或 TMA deep link 进入。
3. 后端记录 `tasks.referrals`，状态为 pending。
4. 被邀请人完成首次开盒后，RPC 触发 `referral_process_first_open`。
5. 双方各得 500 K-coin。
6. 后续被邀请人开盒时，邀请人获得 10% 分红记录。
7. 分红写 `referral_commissions` 和 `currency_ledger`。

---

## 9. 数据库必须一次性做好的约束

| 类型     | 约束                                         |
| ------ | ------------------------------------------ |
| 用户     | `telegram_user_id` 唯一。                     |
| 支付     | `telegram_payment_charge_id` 唯一，防重复发货。     |
| 开盒订单   | `idempotency_key` 唯一，防重复点击。                |
| 保底     | `user_id + box_id` 唯一。                     |
| 余额     | `user_id + currency` 唯一，余额不可为负。            |
| Ledger | 不允许 update / delete，只能 insert 冲正流水。        |
| 藏品     | 每个 item instance 同一时间只能有一个 active lock。    |
| 市场     | active listing 中同一 item instance 只能出现一次。   |
| 任务     | `user_id + task_id + period_key` 只能领取一次。   |
| 签到     | `user_id + campaign_id + day_index` 唯一。    |
| 邀请     | `invitee_id` 唯一，不能多头绑定邀请人。                 |
| Mint   | 同一 item instance 不能重复创建 active mint queue。 |
| 后台     | 所有配置变更必须写 `admin_audit_logs`。              |

## 10. Supabase RLS 和权限建议

Supabase 文档建议暴露 schema 的表启用 RLS，RLS 是 Postgres 的行级安全能力；Supabase 也支持通过数据库函数和 JavaScript `rpc()` 调用 Postgres 函数。([Supabase][6])

建议权限策略：

| 数据类型       | 前端是否可直连 Supabase | 建议                             |
| ---------- | ---------------: | ------------------------------ |
| 公开藏品配置     |       可以，只读 view | 只暴露 active 数据。                 |
| 盲盒列表       |       可以，只读 view | 不暴露后台草稿和隐藏概率版本。                |
| 市场挂单列表     |       可以，只读 view | 也可全部走 API，便于风控。                |
| 用户资产       |            不建议直连 | 走 `/api/me/assets`。            |
| 用户库存       |            不建议直连 | 走 `/api/inventory/list`。       |
| 交易、开盒、任务领取 |             禁止直连 | 只能 API → RPC。                  |
| 支付记录       |             禁止直连 | 只能 API 和后台。                    |
| 后台配置       |           禁止前端访问 | 只允许 admin app + admin session。 |

service role key 只放 Vercel 环境变量。
前端永远不出现 Supabase service role、Bot token、TON mint key。
如果未来需要前端订阅 Realtime，只给只读表或用户自己的通知表开放 RLS。

## 12. 商业化必须预留的后台能力

| 后台能力          | 原因                             |
| ------------- | ------------------------------ |
| 概率池版本管理       | 防止运营改概率后无法审计历史订单。              |
| 盲盒上下架         | 支持未开始、进行中、暂停、售罄、结束。            |
| 活动 banner 配置  | 支持交易页顶部活动、限时折扣、月卡等。            |
| 手续费配置         | 市场手续费可能变化，需要按订单保存快照。           |
| 用户风控          | 高频开盒、异常交易、刷邀请要能限制。             |
| 支付争议处理        | Telegram Stars 需要支付支持、退款和争议记录。 |
| Ledger 对账     | 商业化后必须能解释每一笔资产变化。              |
| 数据回滚方案        | 配置错误时要能下架活动，不要直接删数据。           |
| NFT Mint 队列管理 | 链上失败、重试、补偿、人工处理。               |
| 管理员审计         | 任何概率、奖励、库存、余额调整都要留痕。           |

## 13. 推荐开发顺序

第一阶段：基础闭环
完成 Telegram 登录、资产栏、藏品 catalog、库存表、K-coin / Fgems ledger、基础开盒、开盒结果展示。

第二阶段：交易闭环
完成购买页、出售页、出售管理、挂单锁库存、购买转移、手续费、市场价格统计。

第三阶段：成长系统
完成升级、合成、分解、图鉴、图鉴奖励、排行榜。

第四阶段：任务和增长
完成邀请、7 日签到、任务列表、奖励领取、分红统计。

第五阶段：支付和链上
接入 Telegram Stars 正式支付、支付 webhook、退款记录、TON Connect、钱包同步、NFT Mint 队列。

第六阶段：后台和商业运营
上线 admin、概率版本、活动配置、风控、对账、监控、审计。
