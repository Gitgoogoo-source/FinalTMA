你要让 AI 帮你创建数据库表，**不能只告诉它“我要做一个抽卡项目”**。你需要把业务规则、前端页面、接口、数据流、权限、交易规则全部讲清楚。否则 AI 很容易只设计出“看起来对”，但实际业务跑不通的数据库。

你应该准备下面这些资料。

---

# 1. 项目整体说明

先告诉 AI 你的项目是什么。

例如：

```text
这是一个 Telegram Mini App 开盲盒抽卡小游戏。

用户可以：
1. Telegram 登录
2. 查看资产余额
3. 开蛋 / 开盲盒
4. 获得藏品
5. 查看自己的藏品库存
6. 升级藏品
7. 3 个相同藏品合成 1 个高级藏品
8. 出售藏品
9. 购买其他用户出售的藏品
10. 完成任务领取奖励
11. 邀请好友获得奖励
12. 使用 Telegram Stars 支付
13. 可选择把藏品 Mint 成链上 NFT
```

这个部分的作用是：让 AI 先理解你的项目边界，知道数据库要服务哪些业务。

# 3. 每个功能的完整业务流程

数据库不是孤立设计的。你要告诉 AI 每个功能的完整链路。

例如“开盒功能”要这样描述：

```text
开盒完整流程：

前端点击开蛋按钮
→ 请求 POST /api/boxes/open
→ 后端验证 Telegram session
→ 检查用户余额是否足够
→ 创建开盒订单
→ 扣除用户余额
→ 根据当前概率版本抽取藏品
→ 写入用户库存 user_inventory
→ 写入开盒结果 box_open_results
→ 写入资产流水 currency_ledger
→ 返回开盒结果给前端
```

例如“出售功能”：

```text
出售完整流程：

用户选择一个库存藏品
→ 点击出售
→ 输入价格
→ 后端校验藏品是否属于该用户
→ 校验藏品是否处于 normal 状态
→ 校验是否在锁定期
→ 创建市场挂单 market_listings
→ 把库存藏品状态改成 listed
→ 返回出售成功
```

例如“购买功能”：

```text
购买完整流程：

买家点击购买
→ 校验买家余额
→ 校验挂单是否 still active
→ 锁定该挂单
→ 扣买家余额
→ 给卖家增加可用余额或待结算余额
→ 平台收取手续费
→ 转移库存 owner_user_id
→ 关闭挂单
→ 写入交易记录 market_orders
→ 写入买家、卖家、平台三条流水
```

这个部分非常重要。
如果 AI 不知道业务链路，它设计的表很容易缺少状态字段、幂等字段、锁定字段、流水字段。

# 9. 业务规则

你要把规则写清楚，不要让 AI 猜。

例如：

```text
业务规则：

1. 用户只有登录后才能开盒、购买、出售、升级、合成。
2. 用户出售藏品时，库存状态必须是 normal。
3. 出售中的藏品不能升级、合成、分解、Mint。
4. Stars 支付获得的藏品有 21 天锁定期。
5. 锁定期内不能出售、不能转移。
6. 3 个相同藏品可以合成 1 个高级藏品。
7. 合成有成功率。
8. 合成成功后，消耗 3 个材料藏品，生成 1 个新藏品。
9. 合成失败后，是否销毁材料需要明确。
10. 升级消耗 Fgems。
11. 分解藏品获得 Fgems。
12. 市场交易平台抽取 5% 手续费。
13. 买家支付全额，卖家到账 95%。
14. 邀请奖励需要冻结，防止退款作弊。
15. Telegram Stars 支付必须防止重复发货。
```

你要明确：

```text
什么情况下允许操作？
什么情况下禁止操作？
失败后数据怎么处理？
是否需要退款？
是否需要冻结？
是否需要写流水？
```

# 10. 货币和资产规则

你的项目涉及 Star、Fgems、K-coin、TON，必须讲清楚。

例如：

```text
货币规则：

1. Star
- 用于购买盲盒
- 用于市场交易
- 不允许用户直接提现
- 平台收取 5% 手续费

2. Fgems
- 游戏内积分
- 可通过开盒返还、任务、分解获得
- 用于升级、合成

3. K-coin
- 游戏内普通积分
- 用于基础任务奖励

4. TON
- 可用于链上 Mint 或钱包绑定
- 不作为普通虚拟商品主支付方式
```

还要告诉 AI：

```text
余额是否分为 available 和 locked？
是否有 pending 余额？
退款时如何扣回？
手续费是否单独记账？
```

建议资产表至少支持：

```text
available_balance
locked_balance
total_earned
total_spent
```

流水表必须支持：

```text
before_amount
change_amount
after_amount
reason
reference_type
reference_id
```

否则后期对账会非常痛苦。

# 11. 权限规则 / RLS 规则

如果你用 Supabase，一定要告诉 AI RLS 怎么设计。

例如：

```text
RLS 规则：

1. 用户只能读取自己的 user_balances。
2. 用户只能读取自己的 user_inventory。
3. 用户不能直接修改自己的余额。
4. 用户不能直接插入资产流水。
5. 用户不能直接修改市场订单。
6. 所有核心写操作必须通过后端 API 或 RPC。
7. 管理员可以查看后台数据。
8. 普通用户不能查看其他用户的隐私信息。
9. 市场挂单 active 状态可以公开读取。
10. 藏品 catalog 可以公开读取。
```

这是 Supabase 项目最容易出错的地方。

建议你明确告诉 AI：

```text
前端不要直接写核心表。
核心业务写操作全部走 RPC 或服务器接口。
```

# 19. 给 AI 的数据库开发任务应该这样下

不要这样说：

```text
帮我设计数据库表。
```

应该这样说：

```text
请根据我提供的项目资料，为这个 Telegram Mini App 抽卡游戏设计 Supabase Postgres 数据库。

要求：

1. 输出完整 schema 设计。
2. 每张表说明用途。
3. 每张表列出字段、类型、是否必填、默认值、约束。
4. 标明主键、外键、唯一索引、普通索引。
5. 设计 RLS 策略。
6. 设计 RPC 函数边界。
7. 标明哪些业务必须使用事务。
8. 标明哪些字段用于幂等。
9. 标明哪些表只允许服务端写入。
10. 标明哪些表允许前端读取。
11. 输出 Supabase migration SQL。
12. 输出 seed 数据结构。
13. 输出测试用例清单。
14. 检查是否存在余额错乱、库存重复、重复发货、越权访问、概率被篡改等风险。
```

---

# 20. 最重要的结论

你需要给 AI 的不是“数据库字段”，而是：

```text
业务流程 + 页面需求 + API 需求 + 权限规则 + 交易规则 + 状态规则 + 事务规则 + 幂等规则
```

数据库开发最容易出错的地方不是少一两个字段，而是：

```text
1. 余额没有流水，后期无法对账
2. 库存状态不完整，导致出售中还能升级或合成
3. 支付没有幂等，导致重复发货
4. 交易没有事务，导致钱扣了但藏品没转移
5. RLS 太宽，用户可以改自己的余额
6. 概率配置没有版本，后期无法审计
7. 管理员操作没有日志，出问题无法追踪
8. 邀请奖励没有冻结，退款后无法追回
```

你可以把目标定为：**让 AI 不是单纯建表，而是按照完整业务闭环设计数据库。**


# 21.你不要只让 AI “帮我创建数据库表”。
你应该让 AI 先做这三步：

第一步：根据业务，先输出数据库 ERD / 表关系说明。
第二步：确认表结构没有遗漏后，再输出 migration SQL。
第三步：最后再输出 RLS、RPC、索引、测试数据。

不要一开始就让 AI 直接写 SQL。

正确流程应该是：

业务资料
→ AI 设计表结构
→ 你检查业务是否遗漏
→ AI 修改表结构
→ AI 输出 migration SQL
→ AI 输出 RLS
→ AI 输出 RPC
→ AI 输出 seed 数据
→ AI 输出测试 SQL
