你要让 AI 帮你创建数据库表，**不能只告诉它“我要做一个抽卡项目”**。你需要把业务规则、前端页面、接口、数据流、权限、交易规则全部讲清楚。否则 AI 很容易只设计出“看起来对”，但实际业务跑不通的数据库。

你应该准备下面这些资料。

# 1. 项目整体说明

这是一个 Telegram Mini App 开盲盒抽卡小游戏，未来会商业化运营、投放广告，并按百万月活用户规模设计。

用户在 Telegram 内打开应用，通过 Telegram 登录进入游戏。前端只负责展示页面、发起请求、展示 loading / error / success 状态；用户身份、资产余额、开盒结果、库存归属、支付发货、任务奖励、交易结算、链上状态，都必须以后端和数据库为准。

本次工作只重新设计数据库，不改前端代码，也不改 Vercel Functions API 代码。所以新的数据库设计要尽量服务现有页面和接口返回结构，不能为了数据库“看起来规整”就要求前后端大改。

数据库重构要围绕这些业务能力设计表、约束、RLS、RPC、事务、幂等、审计和风控记录：

```text
用户可以：
1. 使用 Telegram 登录 / 进入 TMA，并由后端验证 Telegram initData 后建立应用 session。
2. 查看头像、用户资料、K-coin、Fgems 等内部资产余额。
3. 连接 TON 钱包，用于展示公开地址、同步链上 NFT 和发起 Mint 流程。
4. 选择不同档次盲盒，查看奖励池、概率、库存和活动状态。
5. 使用 Telegram Stars 支付开盒，支持单抽和 10 连抽，10 连抽有折扣。
6. 支付开盒后获得藏品，并按现有规则返还 K-coin 积分。
7. 查看开盒结果、藏品库存、藏品详情、等级、战力、稀有度、系列、固定星级形态 form_index。
8. 按每个盲盒独立的保底规则累计和触发保底。
9. 消耗 Fgems 升级藏品，升级改变等级和战力，不改变生成时固定的 form_index。
10. 消耗 3 个相同源藏品和 K-coin 进行进化，进化有失败概率，失败后只返还主藏品。
11. 分解重复藏品获得 Fgems 积分。
12. 在市场出售自己的藏品，设置数量和价格，挂单后库存进入出售中 / 锁定中状态。
13. 使用 K-coin 购买其他用户出售中的藏品，成交后完成买家扣款、卖家入账、平台手续费和库存转移。
14. 管理自己的挂单，支持改价、下架和查看预计到账。
15. 查看图鉴收集进度、系列图鉴、图鉴里程碑奖励和排行榜。
16. 每日签到、完成任务、领取奖励，任务状态和奖励发放由后端判断。
17. 邀请好友，好友完成首次有效开盒后双方获得 K-coin 奖励，邀请人获得好友开盒积分分红。
18. 可选择把 off-chain 藏品 Mint 成链上 NFT，并跟踪 Mint 队列、链上 NFT 状态和交易哈希。
```

数据库设计必须坚持这些边界：

```text
1. 前端不可信，不能让前端决定 user_id、余额、中奖结果、任务完成、支付成功、钱包归属或最终手续费。
2. K-coin 和 Fgems 是应用内资产，所有变化必须写 ledger；余额表只能作为查询快照。
3. Telegram Stars 是外部支付单位，不作为顶部可用余额的业务真相；发货必须以支付订单、webhook 事件和发货状态为准。
4. 开盒概率、保底、库存、交易手续费、任务奖励、邀请奖励、分红比例都必须由后端和数据库控制。
5. 核心写操作必须通过事务 / RPC 完成，必须支持幂等、并发锁定、唯一约束和可追溯业务事件。
6. 用户相关数据必须有明确 owner、状态字段、时间字段、必要索引、唯一约束和 RLS。
7. TON 钱包只能保存公开地址、验证结果和链上同步状态，不能保存私钥、助记词或服务端密钥。
8. 本项目不做后台管理系统，但数据库仍要支持运营配置、审计日志和风控记录，方便通过受控脚本、受控 RPC 或人工审核流程管理配置。
```

这个部分的作用是：让 AI 先理解你的项目边界，知道数据库要服务哪些业务。

# 2. 功能模块清单

本项目是 Telegram Mini App 开盲盒抽卡小游戏，数据库要按真实业务边界拆模块。后面让 AI 设计表、RPC、RLS、索引和测试时，必须覆盖下面这些模块。

功能模块：

1. 用户与登录模块
- Telegram 登录。
- Telegram `initData` 后端验签结果。
- 应用用户资料：头像、昵称、Telegram 用户信息快照。
- 用户状态：正常、封禁、限制、删除。
- 应用 session：token hash、过期时间、撤销时间。
- 用户邀请码入口信息，但邀请关系本身放到邀请模块。

2. 资产与账本模块
- 游戏内资产：K-coin、Fgems。
- 用户余额快照：可用余额、锁定余额、累计收入、累计支出。
- 不可变资产流水：充值、开盒返还、任务奖励、邀请奖励、分红、购买扣款、出售到账、手续费、升级消耗、进化消耗、分解奖励、退款冲正、风控冻结 / 解冻。
- 资产变动幂等键。
- Telegram Stars 不是用户站内余额，只作为支付订单和支付事件记录。

3. Telegram Stars 支付模块
- Stars / XTR 支付订单。
- Telegram invoice payload。
- pre-checkout 校验记录。
- successful payment webhook 事件。
- Telegram charge id 幂等。
- 支付发货状态：未发货、发货中、已发货、失败、退款、撤销。
- 支付退款、争议和补偿记录。
- 不同业务支付类型：开盒支付、VIP / 月卡支付，后续其他 Stars 商品预留。

4. 盲盒与开盒模块
- 盲盒基础配置：三个档次、价格、单抽、10 连抽、10 连抽折扣。
- 盲盒状态：未开始、进行中、已结束、已售罄、下架。
- 奖励池和概率版本。
- 每个盲盒当前启用的概率版本。
- 开盒订单。
- 开盒结果。
- 用户每个盲盒独立保底计数。
- 支付开盒后返还 100 K-coin 的记录。
- 盲盒库存、活动时间、上下架记录。

5. 藏品图鉴配置模块
- 藏品基础信息：名称、编号、图片、描述。
- 藏品稀有度。
- 藏品系列。
- 藏品阵营 / 类型。
- 藏品固定星级形态 `form_index`。
- 藏品初始等级、初始战力、成长参数。
- 藏品可开出范围。
- 藏品进化链路配置。
- 藏品分解奖励配置。

6. 用户库存模块
- 用户拥有的藏品实例。
- 藏品归属用户。
- 藏品等级。
- 藏品战力。
- 固定星级形态 `form_index`。
- 藏品来源：开盒、市场购买、进化、任务奖励、图鉴奖励、VIP 权益、运营发放。
- 藏品状态：正常、出售中、锁定中、已消耗、已分解、已 Mint、链上转出。
- 锁定原因：市场挂单、进化材料、Mint 中、风控冻结。
- 藏品归属变更事件。

7. 成长玩法模块
- 升级配置：按稀有度、固定星级、当前等级配置消耗和成长。
- 升级记录：消耗 Fgems，提升等级和战力。
- 进化配置：3 个相同源藏品 + K-coin 进化成同系列下一阶藏品。
- 进化记录：成功、失败、材料消耗、失败返还主藏品。
- 分解配置：不同藏品、稀有度、星级对应的 Fgems 返还。
- 分解记录：消耗库存，发放 Fgems。
- 成长过程必须写资产流水和库存事件。

8. 市场交易模块
- 市场购买列表。
- 出售挂单。
- 用户出售管理列表。
- 挂单价格、数量、币种、手续费快照。
- 挂单状态：出售中、已成交、已下架、已过期、已锁定。
- 改价记录。
- 下架记录。
- 购买订单。
- 买家扣款、卖家到账、平台手续费。
- 市场成交价、参考价、市场深度、价格健康状态。
- 交易并发锁和幂等键。

9. 图鉴与排行榜模块
- 用户图鉴收集进度。
- 按全系列、阵营、稀有度拆分的图鉴进度。
- 图鉴里程碑奖励配置。
- 图鉴奖励领取记录。
- 钱包同步后的链上 NFT 收集状态。
- 每周图鉴排行榜。
- 排行榜分数、排名、稀有藏品数量、Mint 数量。
- 排行榜快照和重算记录。

10. 任务与签到模块

- 任务统计：累计完成、累计奖励、当前周期进度。

11. 邀请与分红模块

# 3. 每个功能的完整业务流程

数据库不是孤立设计的。你要告诉 AI 每个功能的完整链路。

例如“开盒功能”要这样描述：

```text
开盒完整流程：

前端点击开蛋按钮
→ 请求 POST /api/boxes/open
→ 后端验证 Telegram session
→ 检查用户余额是否足够
→ 创建开盒订单
→ 扣除用户余额
→ 根据当前概率版本抽取藏品
→ 写入用户库存 user_inventory
→ 写入开盒结果 box_open_results
→ 写入资产流水 currency_ledger
→ 返回开盒结果给前端
```

例如“出售功能”：

```text
出售完整流程：

用户选择一个库存藏品
→ 点击出售
→ 输入价格
→ 后端校验藏品是否属于该用户
→ 校验藏品是否处于 normal 状态
→ 校验是否在锁定期
→ 创建市场挂单 market_listings
→ 把库存藏品状态改成 listed
→ 返回出售成功
```

例如“购买功能”：

```text
购买完整流程：

买家点击购买
→ 校验买家余额
→ 校验挂单是否 still active
→ 锁定该挂单
→ 扣买家余额
→ 给卖家增加可用余额或待结算余额
→ 平台收取手续费
→ 转移库存 owner_user_id
→ 关闭挂单
→ 写入交易记录 market_orders
→ 写入买家、卖家、平台三条流水
```

这个部分非常重要。
如果 AI 不知道业务链路，它设计的表很容易缺少状态字段、幂等字段、锁定字段、流水字段。

# 9. 业务规则

你要把规则写清楚，不要让 AI 猜。

例如：

```text

2. 用户出售藏品时，库存状态必须是 normal。
3. 出售中的藏品不能升级、合成、分解、Mint。
4. Stars 支付获得的藏品有 21 天锁定期。
5. 锁定期内不能出售、不能转移。
6. 3 个相同藏品可以合成 1 个高级藏品。
7. 合成有成功率。
8. 合成成功后，消耗 3 个材料藏品，生成 1 个新藏品。

11. 分解藏品获得 Fgems。
12. 市场交易平台抽取 5% 手续费。
13. 买家支付全额，卖家到账 95%。
14. 邀请奖励需要冻结，防止退款作弊。
15. Telegram Stars 支付必须防止重复发货。
```

你要明确：

```text

是否需要退款？
是否需要冻结？

```

# 19. 给 AI 的数据库开发任务应该这样下

不要这样说：

```text
帮我设计数据库表。
```

应该这样说：

```text
请根据我提供的项目资料，为这个 Telegram Mini App 抽卡游戏设计 Supabase Postgres 数据库。

要求：

1. 输出完整 schema 设计。
2. 每张表说明用途。
3. 每张表列出字段、类型、是否必填、默认值、约束。
4. 标明主键、外键、唯一索引、普通索引。
5. 设计 RLS 策略。
6. 设计 RPC 函数边界。
7. 标明哪些业务必须使用事务。
8. 标明哪些字段用于幂等。
9. 标明哪些表只允许服务端写入。
10. 标明哪些表允许前端读取。
11. 输出 Supabase migration SQL。
12. 输出 seed 数据结构。
13. 输出测试用例清单。
14. 检查是否存在余额错乱、库存重复、重复发货、越权访问、概率被篡改等风险。
```

# 20. 最重要的结论

你需要给 AI 的不是“数据库字段”，而是：

```text
业务流程 + 页面需求 + API 需求 + 权限规则 + 交易规则 + 状态规则 + 事务规则 + 幂等规则
```

数据库开发最容易出错的地方不是少一两个字段，而是：

```text
1. 余额没有流水，后期无法对账
2. 库存状态不完整，导致出售中还能升级或合成
3. 支付没有幂等，导致重复发货
4. 交易没有事务，导致钱扣了但藏品没转移
5. RLS 太宽，用户可以改自己的余额
6. 概率配置没有版本，后期无法审计
7. 管理员操作没有日志，出问题无法追踪
8. 邀请奖励没有冻结，退款后无法追回
```


# 21.你不要只让 AI “帮我创建数据库表”。
你应该让 AI 先做这三步：

第一步：根据业务，先输出数据库 ERD / 表关系说明。
第二步：确认表结构没有遗漏后，再输出 migration SQL。
第三步：最后再输出 RLS、RPC、索引、测试数据。

不要一开始就让 AI 直接写 SQL。

正确流程应该是：

业务资料
→ AI 设计表结构
→ 你检查业务是否遗漏
→ AI 修改表结构
→ AI 输出 migration SQL
→ AI 输出 RLS
→ AI 输出 RPC
→ AI 输出 seed 数据
→ AI 输出测试 SQL
