你要让 AI 帮你创建数据库表，**不能只告诉它“我要做一个抽卡项目”**。你需要把业务规则、前端页面、接口、数据流、权限、交易规则全部讲清楚。否则 AI 很容易只设计出“看起来对”，但实际业务跑不通的数据库。

你应该准备下面这些资料。

# 1. 项目整体说明

这是一个 Telegram Mini App 开盲盒抽卡小游戏，未来会商业化运营、投放广告，并按百万月活用户规模设计。

用户在 Telegram 内打开应用，通过 Telegram 登录进入游戏。前端只负责展示页面、发起请求、展示 loading / error / success 状态；用户身份、资产余额、开盒结果、库存归属、支付发货、任务奖励、交易结算、链上状态，都必须以后端和数据库为准。

本次工作只重新设计数据库，不改前端代码，也不改 Vercel Functions API 代码。所以新的数据库设计要尽量服务现有页面和接口返回结构，不能为了数据库“看起来规整”就要求前后端大改。

数据库重构要围绕这些业务能力设计表、约束、RLS、RPC、事务、幂等、审计和风控记录：

```text
用户可以：
1. 使用 Telegram 登录 / 进入 TMA，并由后端验证 Telegram initData 后建立应用 session。
2. 查看头像、用户资料、K-coin、Fgems 等内部资产余额。
3. 连接 TON 钱包，用于展示公开地址、同步链上 NFT 和发起 Mint 流程。
4. 选择不同档次盲盒，查看奖励池、概率、库存和活动状态。
5. 使用 Telegram Stars 支付开盒，支持单抽和 10 连抽，10 连抽有折扣。
6. 支付开盒后获得藏品，并按现有规则返还 K-coin 积分。
7. 查看开盒结果、藏品库存、藏品详情、等级、战力、稀有度、系列、固定星级形态 form_index。
8. 按每个盲盒独立的保底规则累计和触发保底。
9. 消耗 Fgems 升级藏品，升级改变等级和战力，不改变生成时固定的 form_index。
10. 消耗 3 个相同源藏品和 K-coin 进行进化，进化有失败概率，失败后只返还主藏品。
11. 分解重复藏品获得 Fgems 积分。
12. 在市场出售自己的藏品，设置数量和价格，挂单后库存进入出售中 / 锁定中状态。
13. 使用 K-coin 购买其他用户出售中的藏品，成交后完成买家扣款、卖家入账、平台手续费和库存转移。
14. 管理自己的挂单，支持改价、下架和查看预计到账。
15. 查看图鉴收集进度、系列图鉴、图鉴里程碑奖励和排行榜。
16. 每日签到、完成任务、领取奖励，任务状态和奖励发放由后端判断。
17. 邀请好友，好友完成首次有效开盒后双方获得 K-coin 奖励，邀请人获得好友开盒积分分红。
18. 可选择把 off-chain 藏品 Mint 成链上 NFT，并跟踪 Mint 队列、链上 NFT 状态和交易哈希。
```

数据库设计必须坚持这些边界：

```text
1. 前端不可信，不能让前端决定 user_id、余额、中奖结果、任务完成、支付成功、钱包归属或最终手续费。
2. K-coin 和 Fgems 是应用内资产，所有变化必须写 ledger；余额表只能作为查询快照。
3. Telegram Stars 是外部支付单位，不作为顶部可用余额的业务真相；发货必须以支付订单、webhook 事件和发货状态为准。
4. 开盒概率、保底、库存、交易手续费、任务奖励、邀请奖励、分红比例都必须由后端和数据库控制。
5. 核心写操作必须通过事务 / RPC 完成，必须支持幂等、并发锁定、唯一约束和可追溯业务事件。
6. 用户相关数据必须有明确 owner、状态字段、时间字段、必要索引、唯一约束和 RLS。
7. TON 钱包只能保存公开地址、验证结果和链上同步状态，不能保存私钥、助记词或服务端密钥。
8. 本项目不做后台管理系统，但数据库仍要支持运营配置、审计日志和风控记录，方便通过受控脚本、受控 RPC 或人工审核流程管理配置。
```

这个部分的作用是：让 AI 先理解你的项目边界，知道数据库要服务哪些业务。

# 3. 每个功能的完整业务流程

数据库不是孤立设计的。下面按当前真实代码整理业务链路。前端路径来自 `apps/web/src/api/endpoints.ts`，服务器文件在 `api/` 目录，核心写操作基本走 `api` schema 下的 Supabase RPC。数据库重构时要服务这些现有返回结构，不要为了重构数据库要求前端和 Vercel Functions 大改。

## 3.1 用户与登录模块

Telegram 登录完整流程：

```text
前端拿到 Telegram WebApp initData
→ 请求 POST /auth/telegram
→ 后端校验请求体和频率限制
→ 后端用 Bot Token 验证 initData 签名、auth_date 和 init_data_hash
→ 如果当前 Cookie 中已有同一 init_data_hash 的有效 session，复用旧 session
→ 否则调用 api.auth_upsert_telegram_user
→ 写入或更新 core.users 的 Telegram 用户快照、invite_code、last_auth_at
→ 后端生成 opaque session token，只把明文 token 写入 HttpOnly Cookie
→ 调用 api.auth_create_session，只把 session_token_hash 写入 core.app_sessions
→ 记录 ip_hash、user_agent hash、device_id、platform、telegram_auth_date、init_data_hash
→ 如果 start_param 带邀请码，只记录风险信号或后续邀请绑定线索，不直接发奖励
→ 返回 user、session 过期时间、isNewUser 给前端
```

Session 刷新和退出流程：

```text
POST /auth/refresh
→ 后端从 Cookie 或 Bearer token 取 session
→ 查询 core.app_sessions 和 core.users
→ 用户必须存在，默认必须 active
→ 在最大生命周期内延长 expires_at，并更新 last_seen_at / platform
→ 重新设置 HttpOnly Cookie

POST /auth/logout
→ 后端校验 session
→ 更新 core.app_sessions.revoked_at
→ 清掉浏览器 Cookie
```

首屏用户数据流程：

```text
GET /me/bootstrap
→ 后端校验 session
→ 调用 api.get_user_bootstrap(p_user_id)
→ 返回用户资料、资产、入口状态等首屏数据

GET /me/assets
→ 后端校验 session
→ 调用 api.get_user_asset_balances(p_user_id)
→ 读取 economy.user_balances
→ 只返回 KCOIN、FGEMS 的 available / locked
```

当前代码状态提示：`api/me/profile.ts` 和 `api/me/notifications.ts` 当前是空文件。数据库可以保留 `core.user_profiles`、`core.notifications`，但不要把这两个接口当成已经接好的真实服务器逻辑。

## 3.2 资产与账本模块

资产查询完整流程：

```text
前端顶部资产栏请求 GET /me/assets 或首屏 GET /me/bootstrap
→ 后端从已验证 session 取 user_id
→ 调用 api.get_user_asset_balances
→ 读取 economy.user_balances
→ 返回 KCOIN、FGEMS 余额快照
```

资产变动完整流程：

```text
业务 RPC 触发资产变化
→ 例如开盒返还、任务奖励、邀请奖励、市场购买、升级、进化、分解、VIP 日福利
→ RPC 内部调用 api._credit_balance / api._debit_balance / economy_* RPC
→ 锁定对应用户的 economy.user_balances 行
→ 计算 available_before / available_after / locked_before / locked_after
→ 写入 economy.currency_ledger，不更新或删除旧流水
→ 更新 economy.user_balances 快照
→ 使用 idempotency_key 或 source_type + source_id 防止重复入账 / 扣账
→ API 把新的余额变化返回给前端
```

资产设计边界：

```text
KCOIN 和 FGEMS 是站内资产
→ 所有变化必须写 economy.currency_ledger
→ economy.user_balances 只是查询快照
→ Telegram Stars / XTR 只进 payments.star_orders、payments.star_payments，不作为用户顶部余额
→ 前端不能提交 balance、reward_amount、price_after_fee 这类事实字段
```

## 3.3 Telegram Stars 支付模块

支付能力查询流程：

```text
GET /telegram/payment-support
→ 后端返回当前 Stars 支付支持状态和前端可展示的提示
→ 这个接口不能返回 Bot Token、webhook secret、service role key 等私密配置
→ 真正发货仍然必须等 successful_payment webhook 落库后，由数据库事务处理
```

Stars 开票完整流程：

```text
业务接口创建 Stars 订单
→ 开盒走 POST /boxes/create-open-order，调用 api.gacha_create_order_checked
→ VIP 月卡走 POST /vip/create-order，调用 api.vip_create_order_checked
→ 写 payments.star_orders，business_type 区分 gacha_open / vip_monthly
→ 写业务订单，例如 gacha.draw_orders 或 vip.vip_orders
→ 生成不可预测 telegram_invoice_payload
→ 调 Telegram Bot API createInvoiceLink
→ 成功后写 payments.star_invoices
→ 调 api.payment_mark_order_invoice_created，把订单状态推进到 invoice_created
→ 返回 invoice_link / invoice_payload / star_order_id 给前端
```

pre-checkout 完整流程：

```text
Telegram 发送 pre_checkout_query 到 POST /telegram/webhook
→ 后端校验 X-Telegram-Bot-Api-Secret-Token
→ 先调用 api.payment_record_telegram_webhook_received 记录 payments.telegram_webhook_events
→ 调用 api.payment_mark_precheckout_checked
→ 校验 invoice_payload 能找到 payments.star_orders
→ 校验 Telegram 用户、currency='XTR'、total_amount、订单状态、过期时间
→ 如果是开盒，还要校验 gacha.draw_orders、盲盒 active、活动时间和库存
→ 如果是 VIP，还要校验 vip.vip_orders、vip.vip_plans、价格和 payload
→ 只把订单标记为 precheckout_checked
→ 调 Telegram answerPreCheckoutQuery
→ 不发货、不写库存、不写 currency_ledger
```

successful payment 完整流程：

```text
Telegram 发送 successful_payment 到 POST /telegram/webhook
→ 后端再次校验 webhook secret
→ 记录 payments.telegram_webhook_events
→ 调 api.payment_record_successful_payment
→ 用 update_id 防重复 webhook
→ 用 payments.star_payments.telegram_payment_charge_id 防重复支付
→ 校验 invoice_payload、Telegram 用户、currency、金额
→ 写 payments.star_payments
→ 把 payments.star_orders.status 标记为 paid
→ 根据 payments.star_orders.business_type 选择发货 RPC
→ gacha_open 调 api.gacha_process_paid_order
→ vip_monthly 调 api.vip_process_paid_order
→ 发货成功后 payments.star_orders.status 变为 fulfilled
```

当前代码状态提示：前端已有 `fetchVipStatus()` 和 `createVipOrder()`，路径是 `/vip/status`、`/vip/create-order`；服务器也已经有 `api/vip/status.ts`、`api/vip/create-order.ts`。这两个接口直接依赖 `api.vip_get_status` 和 `api.vip_create_order_checked`。如果数据库重构后缺少 `vip` schema 或这些 RPC，服务器会返回 `VIP_DATABASE_NOT_READY`。

月卡状态流程：

```text

→ 读取 active vip.vip_plans
→ 读取用户 active 且未过期的 vip.vip_subscriptions
→ 读取当天 vip.vip_daily_claims

```

月卡购买流程：

```text
前端点击订阅 / 续费 VIP 月卡
→ POST /vip/create-order 带 plan_id、expected_price_xtr、idempotencyKey
→ 后端校验 session、风险状态、Stars 支付开关
→ 调 api.vip_create_order_checked
→ RPC 校验用户 active、plan active、价格未变化、幂等键未冲突
→ 写 vip.vip_orders
→ 写 payments.star_orders，business_type='vip_monthly'
→ 生成 invoice_payload
→ 后端创建 Telegram invoice link
→ webhook successful_payment 后调用 api.vip_process_paid_order
→ 新用户创建 vip.vip_subscriptions，老用户续费则延长 current_period_end
```

月卡每日福利流程：

```text

→ 数据库已有 api.vip_claim_daily_benefit，可作为后续接 API 的真实事务入口
```

## 3.13 TON 钱包与 Mint 模块

钱包状态和普通连接流程：

```text
GET /wallet/status
→ 后端校验 session
→ 读取 core.user_wallets 中当前 connected 或最近钱包
→ 返回 connected / verified / disconnected / revoked 状态

POST /wallet/connect
→ 后端校验 session 和幂等键
→ 使用 ops.idempotency_keys 做请求幂等
→ upsert core.user_wallets，保存公开地址、network、wallet_app_name、is_primary、status='connected'
→ 不保存私钥、助记词、服务端密钥
```

TON proof 验签流程：

```text
POST /wallet/challenge
→ 后端生成随机 challenge
→ 写 core.wallet_proofs，status='pending'，expires_at，proof_nonce，request_id
→ 返回 ton_proof_payload

POST /wallet/proof
→ 前端提交 TON Connect account + proof
→ 后端校验 challenge 匹配、proof 未过期、proof_hash 未重复
→ 更新 core.wallet_proofs.used_at、address、domain、proof_signature、proof_hash
→ 服务端验证 TON proof 签名和 domain
→ 调 api.wallet_save_verified_address 保存 verified 钱包
→ core.wallet_proofs.status='verified'，写 wallet_public_key、verified_at
→ proof 重放或冲突写 ops.risk_events
```

断开钱包和同步 NFT 流程：

```text
POST /wallet/disconnect
→ 后端校验 session 和幂等键
→ 更新 core.user_wallets.status='disconnected'、disconnected_at、is_primary=false

POST /wallet/sync-nfts
→ 后端校验 session、钱包必须 connected 且 verified
→ 校验同步功能开关
→ 根据 idempotencyKey 或最近同步任务复用 onchain.wallet_sync_jobs
→ 创建 processing 同步任务
→ 服务端调用 TON NFT provider 查询钱包 NFT
→ 只保留本游戏 active onchain.nft_collections
→ upsert onchain.wallet_nft_snapshots
→ 更新 onchain.nft_items owner_address / owner_user_id 等已知 NFT 绑定信息
→ 更新 core.user_wallets.last_sync_at
→ onchain.wallet_sync_jobs 标记 success / failed

GET /wallet/nfts
→ 调 api.wallet_list_nft_snapshots
→ 返回钱包 NFT 快照分页
```

Mint 请求流程：

```text
前端点击 Mint
→ POST /wallet/mint，带 itemInstanceId、targetAddress、chain、idempotencyKey
→ 后端拒绝前端传 collectionId、collectionAddress、metadata、imageUrl、metadataUrl、priority 等关键事实字段
→ 校验 Mint 功能开关和用户风险状态
→ 使用 ops.idempotency_keys 做请求幂等
→ 调 api.wallet_prepare_mint_request
→ RPC 返回 verified wallet、active collection、item、active_lock、template、form、media
→ 后端校验钱包网络、targetAddress、collection active、metadata 配置、item status='available'、未 minted、未在队列、未锁定、template 可 Mint
→ 后端用 catalog 和 collection 配置生成 metadata snapshot
→ 调 api.wallet_enqueue_mint
→ RPC 锁定 inventory.item_instances，创建 onchain.mint_queue，状态 queued
→ 调 api.wallet_save_mint_metadata_snapshot 保存 metadata snapshot
→ 返回 mintQueueId、status、metadataUrl
```

Mint worker 和链上同步流程：

```text
GET /wallet/mint-status
→ 调 api.wallet_get_mint_status
→ 返回 onchain.mint_queue 分页和状态统计

cron /retry-mint-queue
→ 校验 CRON_SECRET、功能开关
→ 领取 queued / retrying 任务，状态改 processing
→ 服务端调用 Mint provider
→ 写 onchain.transactions，记录 tx_hash、query_id、provider、raw_response
→ 队列进入 submitted / confirming / retrying / manual_review

cron /sync-onchain-transactions
→ 查询 pending Mint 交易状态
→ 成功后调 api.onchain_mark_mint_success
→ 写 onchain.nft_items，绑定 item_address、owner_address、minted_tx_hash
→ 更新 inventory.item_instances 为 minted，释放或消费 Mint 锁
→ 失败后调 api.onchain_mark_mint_failed 或进入 manual_review
```

当前代码状态提示：`api/wallet/transactions.ts` 当前是空文件。数据库有 `onchain.transactions`，但这个读取接口不能当成已接好。

# 9. 业务规则

你要把规则写清楚，不要让 AI 猜。

例如：

```text

5. 锁定期内不能出售、不能转移。
6. 3 个相同藏品可以合成 1 个高级藏品。
```

# 19. 给 AI 的数据库开发任务应该这样下

不要这样说：

```text

```

应该这样说：

```text
请根据我提供的项目资料，为这个 Telegram Mini App 抽卡游戏设计 Supabase Postgres 数据库。

要求：

1. 输出完整 schema 设计。

3. 每张表列出字段、类型、是否必填、默认值、约束。
4. 标明主键、外键、唯一索引、普通索引。
5. 设计 RLS 策略。
6. 设计 RPC 函数边界。
7. 标明哪些业务必须使用事务。
8. 标明哪些字段用于幂等。
9. 标明哪些表只允许服务端写入。
10. 标明哪些表允许前端读取。
11. 输出 Supabase migration SQL。
12. 输出 seed 数据结构。
13. 输出测试用例清单。
```

# 20. 最重要的结论

你需要给 AI 的不是“数据库字段”，而是：

```text
业务流程 + 页面需求 + API 需求 + 权限规则 + 交易规则 + 状态规则 + 事务规则 + 幂等规则
```

数据库开发最容易出错的地方不是少一两个字段，而是：

```text
1. 余额没有流水，后期无法对账
2. 库存状态不完整，导致出售中还能升级或合成
3. 支付没有幂等，导致重复发货
4. 交易没有事务，导致钱扣了但藏品没转移
5. RLS 太宽，用户可以改自己的余额
6. 概率配置没有版本，后期无法审计
7. 管理员操作没有日志，出问题无法追踪
8. 邀请奖励没有冻结，退款后无法追回
```


# 21.你不要只让 AI “帮我创建数据库表”。
你应该让 AI 先做这三步：

第一步：根据业务，先输出数据库 ERD / 表关系说明。
第二步：确认表结构没有遗漏后，再输出 migration SQL。
第三步：最后再输出 RLS、RPC、索引、测试数据。

不要一开始就让 AI 直接写 SQL。

正确流程应该是：

业务资料
→ AI 设计表结构
→ 你检查业务是否遗漏
→ AI 修改表结构
→ AI 输出 migration SQL
→ AI 输出 RLS
→ AI 输出 RPC
→ AI 输出 seed 数据
→ AI 输出测试 SQL
