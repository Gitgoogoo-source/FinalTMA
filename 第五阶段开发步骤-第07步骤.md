## 第 7 步：successful_payment 后触发开盒发货事务

### 目标

将正式支付成功事件接入现有 `api.gacha_process_paid_order`，在数据库事务中完成发货、抽卡、库存、保底、K-coin 返还、任务 / 邀请进度更新。

### 需要先检查的现有文件 / 表

| 类型 | 路径 / 表                                                                                                                                                     |
| ---- | ------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| API  | `api/telegram/webhook.ts`、`api/boxes/result.ts`                                                                                                              |
| RPC  | `api.gacha_process_paid_order`、`api.referral_process_first_open`、`api.referral_create_commission`、`api.task_record_progress`                               |
| DB   | `payments.star_orders`、`star_payments`、`gacha.draw_orders`、`draw_results`、`inventory.item_instances`、`economy.currency_ledger`、`gacha.user_pity_states` |

### 需要新增或修改的内容

| 文件 / 表 / RPC                    | 责任                                                                             |
| ---------------------------------- | -------------------------------------------------------------------------------- |
| `api/telegram/webhook.ts`          | successful_payment 后调用 paid RPC。                                             |
| `api.gacha_process_paid_order`     | 确认接受 `star_order_id` / `payment_charge_id` / `invoice_payload`，并保证幂等。 |
| `payments.star_orders`             | 状态从 `paid` → `fulfilling` → `fulfilled` 或 `failed`。                         |
| `payments.telegram_webhook_events` | 记录发货结果。                                                                   |
| `ops.risk_events`                  | 记录异常发货、金额不一致、重复发货尝试。                                         |

### 后端开发工作

发货处理步骤：

1. 将 `star_orders` 状态置为 `fulfilling`。
2. 调用 `api.gacha_process_paid_order`。
3. RPC 内部锁定 `draw_orders`、`star_orders`、相关库存 / 余额行。
4. 生成 `draw_results`。
5. 创建 `inventory.item_instances`。
6. 更新 `gacha.user_pity_states`。
7. 返还 K-coin，写 `economy.currency_ledger`。
8. 更新 `draw_orders.status='completed'`。
9. 更新 `star_orders.status='fulfilled'`。
10. 如果失败，记录错误，保持可重试状态。

### 数据库开发工作

`gacha_process_paid_order` 必须检查：

- draw_order 存在。
- star_order 存在。
- star_payment 存在。
- payment charge id 未被其他订单使用。
- draw_order 未 completed。
- draw_results 不重复生成。
- ledger idempotency key 唯一。
- inventory item instance 不重复生成。
- 库存足够。
- drop pool version 与订单一致。

### 前端开发工作

`useDrawResult` 可继续复用。
前端应区分：

- `pending_payment`
- `paid_waiting_fulfillment`
- `fulfilling`
- `completed`
- `failed`
- `expired`
- `refunded`

`DrawResultModal` 在 `fulfilling` 时显示“支付已成功，发货处理中”。
支付成功但发货失败时显示“支付已成功，奖励补发中”，不要让用户重复支付。

### 安全和幂等要求

发货 RPC 必须可重复调用。
重复调用只能返回已有结果。
发货中断后可以由后台重试。
任何发货失败不能丢失 `star_payment`。
K-coin 返还和 item_instances 必须与 draw_results 一致。

### 测试要求

支付成功正常发货。
重复 webhook 只发货一次。
支付成功但 RPC 中途失败后可重试。
K-coin 返还正确。
保底正确更新。
库存不足时进入异常流程。
draw_results 数量等于 draw_count。
ledger 和 balance 对账一致。

### 验收标准

- [ ] 支付成功后触发 `gacha_process_paid_order`。
- [ ] 每个 paid order 只发货一次。
- [ ] `draw_results` 数量正确。
- [ ] `item_instances` 正确创建。
- [ ] K-coin 返还正确。
- [ ] `currency_ledger` 有完整流水。
- [ ] 保底状态正确更新。
- [ ] 前端刷新后仍能查询结果。
- [ ] 发货失败可后台重试。
