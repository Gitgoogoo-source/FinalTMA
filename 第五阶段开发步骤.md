### 4.5 第 5 步：实现 Telegram webhook 的 pre_checkout_query

#### 目标

实现 `api/telegram/webhook.ts` 对 `pre_checkout_query` 的接收、secret 校验、订单校验、事件落库和确认响应。

#### 需要先检查的现有文件 / 表

| 类型   | 路径 / 表                                                                                            |
| ------ | ---------------------------------------------------------------------------------------------------- |
| API    | `api/telegram/webhook.ts`                                                                            |
| Server | `packages/server/src/payments/telegramStars.ts`                                                      |
| DB     | `payments.telegram_webhook_events`、`payments.star_orders`、`gacha.draw_orders`、`gacha.blind_boxes` |
| ENV    | `TELEGRAM_WEBHOOK_SECRET`、`TELEGRAM_BOT_TOKEN`                                                      |
| RPC    | 可新增 `api.payment_mark_precheckout_checked` 或通过 service role 表更新                             |

#### 需要新增或修改的内容

| 文件 / 表                                       | 责任                                                                |
| ----------------------------------------------- | ------------------------------------------------------------------- |
| `api/telegram/webhook.ts`                       | 处理 Telegram webhook 入口。                                        |
| `packages/server/src/payments/telegramStars.ts` | 校验 update、解析 pre_checkout_query、调用 answerPreCheckoutQuery。 |
| `payments.telegram_webhook_events`              | 先落库原始事件，再处理。                                            |
| `payments.star_orders`                          | 状态更新为 `precheckout_checked` 或当前兼容状态 `precheckout_ok`。  |

#### 后端开发工作

pre_checkout 处理步骤：

1. 校验 HTTP method。
2. 校验 webhook secret header。
3. 解析 Telegram update。
4. 识别 `pre_checkout_query`。
5. 将原始 update 写入 `payments.telegram_webhook_events`。
6. 根据 invoice payload 查询 `payments.star_orders`。
7. 校验订单存在、未过期、状态可支付、金额一致、currency 为 XTR。
8. 校验关联 `gacha.draw_orders` 状态可支付。
9. 校验盲盒仍可用、库存仍足够。
10. 调用 Telegram Bot API 确认 pre_checkout。
11. 更新 `star_orders.precheckout_at` 和状态。
12. 标记 webhook event processed。

#### 数据库开发工作

`telegram_webhook_events.update_id` 已有 unique，继续使用。
增加 event_type 索引。
增加 `process_status` + `created_at` 索引。
`star_orders.telegram_invoice_payload` 已唯一，作为订单查找入口。
需要记录 pre_checkout 校验失败原因。

#### 前端开发工作

无直接 UI 变更。
前端只会看到支付窗口内的失败信息或后续订单状态。
可在支付等待 Sheet 中展示“支付校验中”。

#### 安全和幂等要求

pre_checkout 不发货。
重复 pre_checkout 只重复确认订单可支付，不创建新订单。
金额不一致必须拒绝。
订单过期必须拒绝。
盲盒不可用或库存不足必须拒绝。
webhook 事件必须先落库。
secret 校验失败不落入业务处理链路，可写 risk event。

#### 测试要求

有效 pre_checkout。

- 创建支付订单。
- 支付等待。
- 成功后展示结果。
- 连接钱包。
- 验证钱包。
- 创建 Mint。
- 查看 Mint 状态。
- 刷新恢复状态。

#### 安全和幂等要求

所有回滚只关闭新请求，不删除历史数据。
已支付订单必须仍能补发。
已入队 Mint 必须有明确处理策略，不能静默丢弃。
回滚脚本不能直接修改 ledger。
