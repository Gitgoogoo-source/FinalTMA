# 第二阶段开发总目标

第二阶段完成后，应该形成完整交易闭环：

```txt
出售页：
用户选择自己的 available 藏品
  ↓
设置数量和价格
  ↓
后端 RPC 校验并创建挂单
  ↓
item_instances.status = listed
  ↓
inventory_locks 写入 market_listing 锁
  ↓
market.listings / listing_items 写入 active/reserved 状态

购买页：
用户看到 active 挂单
  ↓
查看商品详情、参考价、市场深度、价格健康
  ↓
使用 K-coin 购买
  ↓
后端 RPC 同事务完成：
买家扣 K-coin
卖家到账
平台收手续费
藏品 owner 转移
listing_items 标记 sold
listing 状态更新 sold / partially_sold
ledger / order / event / fee_settlement 全部写入

出售管理页：
卖家查看自己的 active 挂单
  ↓
改价或下架
  ↓
改价写 event
  ↓
下架释放库存锁，藏品回到 available
```

核心规则不变：**前端只展示和发起请求；交易、扣款、转移、手续费、库存锁全部由 API + Supabase RPC 完成。**

# 阶段 1：明确第二阶段数据模型边界

## 要完成什么

在真正开发前，明确交易市场卖的不是“抽象藏品数量”，而是具体的 `inventory.item_instances.id`。

这是你数据库当前结构的正确用法，因为 `market.listing_items.item_instance_id` 已经指向具体库存实例，`inventory.inventory_locks.item_instance_id` 也能锁定具体藏品。

## 大致怎么操作

确定第二阶段采用以下规则：

1. 出售时，用户必须选择具体的 `item_instance_id`。
2. 如果用户出售数量为 3，实际就是选择 3 个具体藏品实例。
3. 每个挂单可以包含多个相同 `template_id + form_id` 的 item instance。
4. 购买时，建议先支持“购买 1 个挂单单位”，即从 `listing_items` 中取 1 个 `reserved` item 转给买家。
5. 如果一个挂单包含多件商品：
   - 买走一件后 `remaining_count - 1`。
   - 还有剩余时，`listing.status = partially_sold`。
   - 全部卖完时，`listing.status = sold`。

## 验收标准

你的第二阶段 PR 描述或开发文档中明确写出：

```txt
市场交易对象 = inventory.item_instances 的具体实例
出售数量 = 被选择的 item_instance_id 数量
购买转移 = owner_user_id 从卖家改为买家
挂单锁定 = inventory_locks + item_instances.status=listed
```

| ----------- | --------------------------------------------------------- |
| 卖家用户    | 至少拥有 3 个 `available` 状态的可交易藏品。              |
| 买家用户    | 至少拥有足够 K-coin，例如 10,000 KCOIN。                  |
| 藏品模板    | `catalog.collectible_templates.tradeable = true`。        |

测试数据可以通过以下方式准备：

1. 本地 seed 增加测试卖家和买家。
2. 或通过 Supabase Studio 手动给第二个用户补 K-coin 和库存。
3. 或在测试 SQL 里为每个 test case 临时创建用户、余额和藏品。

## 验收标准

```txt

```

# 阶段 3：检查和加固交易数据库结构

## 要完成什么

你已经有市场表，但第二阶段上线前需要确认约束、索引、RLS、状态流转都足够安全。

重点不是重建表，而是 **加固现有表**。

## 大致怎么操作

检查以下内容。

### 3.1 检查重复挂单约束

确保同一个 `item_instance_id` 在同一时间只能出现在一个未完成的 active / reserved 挂单中。

建议加固方向：

```txt
market.listing_items 中，同一个 item_instance_id 不能同时存在 active listing 下的 reserved 记录。
inventory.inventory_locks 中，同一个 item_instance_id 不能同时有多个 active lock。
```

因为 PostgreSQL partial unique index 不能直接跨表判断 listing 是否 active，所以更可靠的方式是：

```txt
1. RPC 创建挂单时先锁 item_instances 行。
2. 检查 inventory_locks 是否已有 active lock。
3. 检查 item_instances.status 是否 available。
4. 成功后插入 inventory_locks active。
5. 成功后 item_instances.status 改 listed。
```

### 3.2 检查交易相关索引

Supabase performance advisor 显示市场相关外键有不少未加索引。第二阶段最重要的查询会用到：

```txt
market.listings.status + created_at
market.listings.seller_user_id + status
market.listings.template_id + form_id + status
market.listing_items.listing_id + status
market.listing_items.item_instance_id
market.orders.buyer_user_id + created_at
market.orders.seller_user_id + created_at
market.order_items.item_instance_id
market.listing_events.listing_id + created_at
market.price_snapshots.template_id + form_id + snapshot_at
market.depth_snapshots.template_id + form_id + snapshot_at
inventory.item_instances.owner_user_id + status
inventory.inventory_locks.item_instance_id + status
economy.currency_ledger.user_id + currency_code + created_at
```

### 3.3 检查市场 RLS

你的业务写操作走 Vercel service role + RPC，因此市场表不需要直接暴露给前端写入。

建议规则：

```txt
market.* 表：前端不可直接 insert/update/delete
公开市场列表：可以通过 API 返回，或只开放只读 view
用户自己的挂单：通过 API 返回
核心写操作：只能 API -> RPC
```

### 3.4 检查手续费规则

确认 `economy.fee_rules` 有：

```txt
fee_type = market_sell
currency_code = KCOIN
fee_bps = 500 或你设定的费率
active = true
```

手续费应该在创建挂单和成交订单时保存快照。即使后续平台调整手续费，历史订单也不能改变。

## 验收标准

- 同一个藏品不能重复挂售。
- 已挂售藏品不能再被升级、合成、分解、Mint。
- 买卖相关查询不会全表扫描核心大表。
- 直接用前端 anon key 不能修改 `market.*` 表。
- service role + RPC 可以完成交易写操作。
- 手续费规则可被 RPC 正确读取。

# 阶段 5：实现或核查市场 RPC

## 要完成什么

第二阶段最关键的是 RPC。API 只是入口，真正业务必须在数据库事务中完成。

虽然当前迁移里已有 `0015_create_rpc_market`，但需要逐个验证 RPC 是否已经完整支持第二阶段。如果没有，就补充新的 migration。

## 大致怎么操作

建议最终具备以下 RPC。

---

## 5.1 `market_list_listings`

### 负责什么

返回购买页市场列表。

### 应完成内容

- 只返回 `active` 或 `partially_sold` 且 `remaining_count > 0` 的挂单。
- 支持价格、稀有度、类型、系列、模板筛选。
- 支持排序。
- 返回藏品图片、名称、稀有度、价格、剩余数量。
- 可以返回 `is_own_listing`，前端用于禁用“购买自己挂单”。

### 验收标准

- 市场为空时返回空数组。
- 创建挂单后能查到。
- 下架或售罄后不再出现在购买页。
- 筛选条件能正确生效。

---

## 5.2 `market_get_listing_detail`

### 负责什么

返回商品详情页 / 弹窗信息。

### 应完成内容

- 查询 listing 基础信息。
- 查询藏品模板、形态、图片、稀有度。
- 查询卖家展示信息。
- 查询最近价格快照。
- 查询市场深度。
- 返回价格健康状态。
- 判断当前用户是否能购买。

### 验收标准

详情数据包含：

```txt

template_id
form_id
item_name
rarity
image_url
seller
unit_price_kcoin
remaining_count
floor_price_kcoin
avg_price_kcoin

price_health
market_depth
can_buy
disabled_reason
```

## 5.3 `market_list_sellable_items`

### 负责什么

返回出售页可出售库存。

当前 `api/inventory/list.ts` 明确第一阶段暂不支持复杂筛选，比如 keyword、rarities、types、series_ids 等。
第二阶段不建议强行改坏第一阶段库存接口，建议单独做 `market/sellable-items`。

### 应完成内容

只返回：

```txt
owner_user_id = 当前用户
status = available
catalog.collectible_templates.tradeable = true
没有 active inventory_locks
nft_mint_status 不在 minting / queued
```

支持筛选：

```txt
rarity
type
series
price suggestion
keyword
```

返回时最好按 `template_id + form_id` 聚合，同时保留具体 `item_instance_ids`：

```txt
template_id
form_id
name
rarity
image_url
owned_count
available_count
item_instance_ids
suggested_price
min_price
max_price
```

### 验收标准

- 已挂售的藏品不会出现在可出售列表。
- 已锁定、Mint 中、分解、消耗的藏品不会出现。
- 可出售列表能支持出售页选择数量。

## 5.4 `market_create_listing`

### 负责什么

确认出售，创建挂单并锁定库存。

### 应完成内容

RPC 必须在一个事务中完成：

1. 校验 `idempotency_key`。
2. 校验 `item_instance_ids` 数量大于 0。
3. 锁定这些 `inventory.item_instances` 行。
4. 校验这些 item 都属于当前用户。
5. 校验状态都是 `available`。
6. 校验模板都可交易。
7. 校验所有 item 属于同一个 `template_id + form_id + rarity_code`。
8. 检查没有 active `inventory_locks`。
9. 读取 active `market_sell` 手续费规则。
10. 计算：
    - `item_count`
    - `remaining_count`
    - `unit_price_kcoin`
    - `fee_bps`
    - `expected_net_amount`
    - `price_health`

11. 插入 `market.listings`。
12. 插入多条 `market.listing_items`，状态 `reserved`。
13. 插入多条 `inventory.inventory_locks`，`lock_type = market_listing`，`status = active`。
14. 将这些 `inventory.item_instances.status` 改为 `listed`。
15. 写入 `inventory.item_instance_events`，事件 `listed`。
16. 写入 `market.listing_events`，事件 `created`。
17. 返回创建结果。

### 验收标准

创建挂单后：

```txt
market.listings 有 1 条 active 记录
market.listing_items 有 N 条 reserved 记录
inventory.inventory_locks 有 N 条 active market_listing 锁
inventory.item_instances 对应 N 件 status = listed
market.listing_events 有 created 事件
出售页不再显示这些 item 为可出售
出售管理页能看到这个挂单
购买页能看到这个挂单
```

异常验收：

```txt
重复提交同一个 idempotency_key 不会创建重复挂单
传入别人的 item_instance_id 会失败
传入 listed / locked / decomposed item 会失败
传入不可交易模板会失败
传入不同 template/form 的多个 item 会失败
价格小于等于 0 会失败
```

---

## 5.5 `market_buy_listing`

### 负责什么

购买挂单，完成 K-coin 扣款、手续费、藏品转移。

### 应完成内容

RPC 必须在一个事务中完成：

1. 校验 `idempotency_key`。
2. 锁定 `market.listings` 行。
3. 校验 listing 状态为 `active` 或 `partially_sold`。
4. 校验 `remaining_count >= purchase_quantity`。
5. 禁止买家购买自己的挂单。
6. 校验前端传入的 `expected_unit_price_kcoin` 与数据库当前价格一致；不一致则返回“价格已变化”。
7. 从 `listing_items` 中锁定指定数量的 `reserved` item。
8. 锁定买家的 `economy.user_balances` KCOIN 行。
9. 校验买家 K-coin 足够。
10. 计算：
    - `total_price_kcoin`
    - `fee_amount_kcoin`
    - `seller_net_amount_kcoin`

11. 写买家 debit ledger。
12. 更新买家余额。
13. 写卖家 credit ledger。
14. 更新卖家余额。
15. 写平台手续费 `market.fee_settlements`。
16. 更新 item owner：
    - `owner_user_id = buyer_user_id`
    - `status = available`
    - `source_type = market`

17. 对应 `inventory.inventory_locks.status = consumed`。
18. 对应 `market.listing_items.status = sold`。
19. 创建 `market.orders`。
20. 创建 `market.order_items`。
21. 写 `inventory.item_instance_events`：
    - 卖家侧 `sold`
    - 买家侧 `bought`

22. 更新 `market.listings.remaining_count`。
23. 如果剩余为 0，listing 改 `sold`。
24. 如果还有剩余，listing 改 `partially_sold`。
25. 写 `market.listing_events`。
26. 可选：更新交易任务进度。

### 验收标准

购买成功后：

```txt
buyer KCOIN 减少 total_price
seller KCOIN 增加 seller_net_amount
fee_settlements 写入 fee_amount
currency_ledger 至少有 buyer debit 和 seller credit
market.orders 有 completed 订单
market.order_items 绑定购买到的 item
item_instances.owner_user_id 从 seller 改为 buyer
item_instances.status = available
listing_items.status = sold
inventory_locks.status = consumed
listing.remaining_count 正确减少
listing.status 正确变成 partially_sold 或 sold
购买页库存显示减少或挂单消失
买家藏品页能看到新买到的藏品
```

异常验收：

```txt
买家余额不足失败，且不产生任何半成品数据
购买自己挂单失败
重复 idempotency_key 不重复扣款
两个买家同时买最后一件，只能一个成功
价格已变化时购买失败
已下架挂单不可购买
已售罄挂单不可购买
```

## 5.6 `market_update_listing_price`

### 负责什么

出售管理页改价。

### 应完成内容

1. 校验 listing 属于当前用户。
2. 校验 listing 状态为 `active` 或 `partially_sold`。
3. 校验新价格大于 0。
4. 锁定 listing。
5. 重新计算：
   - `expected_net_amount`
   - `price_health`

6. 更新 `unit_price_kcoin`。
7. 更新 `last_price_changed_at`。
8. 写 `market.listing_events`，事件 `price_changed`。

### 验收标准

- 改价后购买页显示新价格。
- 商品详情显示新价格和新的价格健康状态。
- 已售罄、已下架挂单不可改价。
- 非卖家不可改价。
- 改价不会影响已经完成的历史订单。

---

## 5.7 `market_cancel_listing`

### 负责什么

出售管理页下架。

### 应完成内容

1. 校验 listing 属于当前用户。
2. 校验 listing 状态为 `active` 或 `partially_sold`。
3. 锁定 listing。
4. 找出 remaining 的 `listing_items.status = reserved`。
5. 将这些 `listing_items.status` 改为 `cancelled`。
6. 将对应 `inventory.inventory_locks.status` 改为 `released`。
7. 将对应 `item_instances.status` 改回 `available`。
8. 将 listing 状态改为 `cancelled`。
9. 写 `inventory.item_instance_events`，事件 `delisted`。
10. 写 `market.listing_events`，事件 `cancelled`。

### 验收标准

下架后：

```txt
market.listings.status = cancelled
remaining listing_items.status = cancelled
sold listing_items 仍保持 sold
inventory_locks.status = released
未售出的 item_instances.status = available
出售页重新能看到未售出的 item
购买页不再显示该挂单
出售管理页显示已下架或不再显示 active
```

---

## 5.8 `market_get_my_listing_stats`

### 负责什么

出售管理页顶部统计。

### 应完成内容

返回：

```txt
active_listing_count
active_item_count
total_listing_value_kcoin
expected_net_amount_kcoin
sold_24h_count
sold_24h_value_kcoin
```

### 验收标准

- 创建挂单后统计增加。
- 改价后总价值变化。
- 下架后 active 统计减少。
- 成交后 active 数量和预计到账正确变化。

---

## 5.9 `market_refresh_price_stats`

### 负责什么

刷新市场参考价、地板价、均价、市场深度。

### 应完成内容

对每个 `template_id + form_id` 计算：

```txt
floor_price_kcoin = 当前 active 挂单最低价
avg_price_kcoin = 近期成交均价
last_sale_price_kcoin = 最近成交价
active_listing_count = 当前 active 挂单数量
sale_count_24h = 24 小时成交数量
volume_24h_kcoin = 24 小时成交额
```

并写入：

```txt

```

### 验收标准

- 有挂单时能生成地板价。
- 有成交时能生成最近成交价和 24h 成交量。
- 没有数据时前端显示 `unknown`，而不是报错。

# 阶段 13：市场价格统计和价格健康

## 要完成什么

完成商品详情里的：

```txt
市场参考价
市场深度
价格健康状态
```

## 大致怎么操作

### 13.1 价格快照

通过 RPC 或 cron 定时刷新：

```txt
market.price_snapshots
```

计算维度：

```txt
template_id + form_id
```

计算字段：

```txt
floor_price_kcoin
avg_price_kcoin
last_sale_price_kcoin
active_listing_count
sale_count_24h
volume_24h_kcoin
snapshot_at
```

### 13.2 市场深度

写入：

```txt
market.depth_snapshots
```

按价格桶聚合：

```txt
price_bucket_kcoin
listing_count
item_count
```

价格桶可以先简单处理：

```txt
0-99
100-499
500-999
1000-4999
5000+
```

或按固定步长，例如每 100 K-coin 一个桶。

### 13.3 价格健康状态

优先使用：

```txt
market.price_health_rules
```

如果没有规则，可以使用默认逻辑：

```txt
没有 floor_price：unknown
价格 < floor_price * 0.5：too_low
价格 > floor_price * 2：too_high
其他：healthy
```

### 13.4 刷新入口

建议两个入口都做：

```txt
api/cron/refresh-market-stats.ts
scripts/rebuild-market-stats.ts
```

其中脚本用于本地和运营手动刷新，cron 用于线上定时刷新。

## 验收标准

- 创建挂单后，刷新统计能生成 floor price。
- 成交后，刷新统计能生成 last sale price。
- 商品详情能看到参考价。
- 没有参考价时显示“暂无参考”，不报错。
- 改价后 price_health 能重新计算。
