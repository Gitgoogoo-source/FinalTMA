## 第 4 步：完善 Stars 开盒订单创建与 invoice 创建

### 目标

把当前 `api/boxes/create-open-order.ts` 从“创建内部订单 + dev mode 可直接发货”升级为“创建正式 Telegram Stars invoice 的开盒支付订单”。

### 需要先检查的现有文件 / 表

| 类型       | 路径 / 表                                                             |
| ---------- | --------------------------------------------------------------------- |
| API        | `api/boxes/create-open-order.ts`                                      |
| Server     | `packages/server/src/payments/telegramStars.ts`                       |
| Validation | `packages/validation/src/box.schemas.ts`                              |
| DB         | `payments.star_orders`、`payments.star_invoices`、`gacha.draw_orders` |
| RPC        | `api.gacha_create_order`                                              |

### 需要新增或修改的内容

| 文件 / 表                                       | 责任                                                                                  |
| ----------------------------------------------- | ------------------------------------------------------------------------------------- |
| `api/boxes/create-open-order.ts`                | 保留现有 RPC 创建订单逻辑；新增正式环境 invoice 创建逻辑。                            |
| `packages/server/src/payments/telegramStars.ts` | 封装 Telegram Bot API invoice 创建、payload 构造、金额校验、响应落库。                |
| `payments.star_invoices`                        | 保存 invoice request / response / invoice_link / payload。                            |
| `gacha.draw_orders`                             | 关联 `payment_star_order_id`，保持 `invoice_payload` 与 `star_order` 一致。           |
| 前端 `box.api.ts`                               | 扩展 response：invoice link / invoice open mode / expires_at / payment_order_status。 |

### 后端开发工作

创建订单时流程：

1. `requireSession` 获取真实 `user_id`。
2. 校验 box、draw count、expected price。
3. 调用 `api.gacha_create_order`。
4. 根据返回的 `star_order_id`、`invoice_payload`、`xtr_amount` 创建 Telegram invoice。
5. 把 invoice 请求和响应写入 `payments.star_invoices`。
6. 返回前端：`draw_order_id`、`star_order_id`、`invoice_payload`、`xtr_amount`、`invoice_link` 或前端可打开 invoice 的参数、`expires_at`。
7. 生产环境不得调用 `gacha_process_dev_paid_order`。

### 数据库开发工作

确认 `star_orders.business_type='gacha_open'`，`business_id=draw_order_id`。
确认 `star_orders.xtr_amount` 与 `draw_orders.total_price_stars` 一致。
确认 `star_invoices.payload` 与 `star_orders.telegram_invoice_payload` 一致。
为 `star_invoices.star_order_id`、`payload`、`status` 建索引。
为过期订单添加查询索引。

### 前端开发工作

`createOpenOrder` response 需要保留：

- `orderId`
- `starOrderId`
- `invoicePayload`
- `invoiceLink`
- `xtrAmount`
- `drawCount`
- `expiresAt`
- `paymentStatus`

`BoxPage` 在创建订单成功后，不能只显示等待 Sheet，需要尝试打开 Telegram Stars invoice。
若 Telegram invoice 打开失败，显示“支付未打开，可重试支付”而不是“支付成功”。

### 安全和幂等要求

同一 `idempotency_key` 重复请求必须返回同一订单，不创建第二个订单。
前端 `expected_price_stars` 只用于发现价格变化，最终价格以后端 / DB 为准。
invoice payload 必须不可预测，且与订单强绑定。
生产环境不允许 dev paid mode。

### 测试要求

创建单抽订单。
创建十连订单。
重复提交同一 idempotency key。
价格变化时拒绝或返回冲突。
盲盒暂停 / 售罄时拒绝。
dev mode 与正式 mode 分支测试。
invoice 创建失败时订单进入可重试或 failed 状态，不发货。

### 验收标准

- [ ] 用户可以创建 Stars 开盒订单。
- [ ] 订单包含唯一 invoice payload。
- [ ] 单抽和十连金额正确。
- [ ] 十连 9 折金额以后端计算为准。
- [ ] 订单创建失败不会产生半完成发货。
- [ ] 未支付订单不会生成 `draw_results`。
- [ ] 生产环境不会调用 dev paid RPC。
