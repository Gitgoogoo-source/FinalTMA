# 藏品库存功能说明

## 1. 这份文档写什么

这份文档只说明当前项目里已经能从代码确认的“藏品库存”功能。

这里说的“藏品库存”，主要是用户在“藏品”页面看到和操作自己藏品的能力，包括：

- 查看自己拥有的藏品。
- 按稀有度、类型、状态筛选。
- 查看单个藏品详情。
- 对藏品做出售、下架、升级、进化、分解、Mint。
- 查看 Mint 队列状态。

本说明只根据当前项目代码和数据库迁移文件整理，不包含没有在代码中确认的功能。

## 2. 一句话说明

藏品库存是用户管理自己藏品的页面。

用户可以在这里看藏品、筛选藏品、查看详情，并对符合条件的藏品进行出售、下架、升级、进化、分解和 Mint。

所有真正会改变库存、资产或链上状态的操作，都不是前端自己决定结果，而是提交给后端，由后端和数据库事务判断。

## 3. 页面入口

前端底部导航里有“藏品”入口。

进入后加载 `CollectionPage` 页面。

页面加载时会先读取用户库存摘要。读取成功后，页面显示藏品；读取中显示“藏品加载中”；读取失败时显示“库存读取失败”和重试按钮；如果用户还没有藏品，会提示“还没有藏品”，并给出“去开盒”的入口。

## 4. 藏品页面展示什么

藏品页主要分成两块：

1. 上方是当前选中的藏品详情区。
2. 下方是藏品网格区。

详情区展示：

- 藏品图片。
- 藏品名称。
- 稀有度。
- 形态。
- 等级。
- 战力。
- 当前状态。
- 可执行操作按钮。

网格区展示：

- 用户拥有的藏品。
- 同款同形态藏品会合并成一组。
- 每组显示代表藏品。
- 如果同一组有多件，点击后可以展开看这一组下面的具体藏品。

## 5. 藏品分组规则

库存摘要会把藏品按“同一个藏品模板 + 同一个形态”分组。

也就是说，同一个角色、同一种形态的多件藏品，会先合成一组展示。

分组里会统计：

- 用户一共拥有几件。
- 有几件可用。
- 有几件挂售中。
- 有几件锁定中。
- 有几件正在 Mint。
- 有几件已经 Mint。
- 这一组里最高等级是多少。
- 这一组里最高战力是多少。
- 这一组最近获得时间。

页面默认显示每组的代表藏品。代表藏品优先选可用、未锁定、未 Mint 中的藏品；如果没有这种藏品，再按状态、等级、战力和获得时间挑一个。

用户点击某个数量大于一的藏品组后，页面会展开这一组，显示具体每一件藏品。展开后还可以返回全部。

## 6. 筛选和排序

藏品网格支持筛选和排序。

可以筛选：

- 稀有度。
- 类型。
- 状态。

状态筛选包含：

- 可用。
- 出售中。
- 锁定。
- Mint 中。
- 已 Mint。

可以排序：

- 最新获得。
- 稀有度从高到低。
- 稀有度从低到高。
- 等级从高到低。
- 等级从低到高。
- 战力从高到低。
- 战力从低到高。
- 名称顺序。

页面有重置筛选按钮，点击后恢复默认列表。

## 7. 藏品状态说明

代码里能看到的藏品状态包括：

| 状态 | 大白话说明 |
| --- | --- |
| 可用 | 藏品当前可以正常操作 |
| 锁定中 | 藏品被某个业务锁住，暂时不能操作 |
| 挂售中 | 藏品正在市场出售 |
| 已消耗 | 藏品已经被进化等流程消耗 |
| 已分解 | 藏品已经被分解 |
| Mint 中 | 藏品已经进入 Mint 流程 |
| 已 Mint | 藏品已经完成 Mint |
| 已转出 | 藏品已经转出 |
| 已销毁 | 藏品已经销毁 |

当前藏品页主要展示用户仍然拥有、仍需要展示的藏品，比如可用、挂售中、锁定中、Mint 中、已 Mint。

已经被消耗、分解、转出、销毁的藏品，不是正常库存列表里的主要展示对象。

## 8. 藏品详情

用户选中某个藏品后，页面会展示这个藏品的关键信息。

详情接口会返回更多内容，包括：

- 藏品基础信息。
- 稀有度。
- 系列。
- 阵营。
- 形态。
- 等级。
- 战力。
- 图片。
- 是否可交易。
- 是否可升级。
- 是否可进化。
- 是否可分解。
- 是否可 Mint。
- 当前是否有库存锁。
- 当前是否在市场挂售。
- 当前链上状态。
- 升级预览。
- 进化预览。
- 分解预览。
- 同款数量。
- 可用同款数量。

用户只能查看属于自己的藏品。后端会校验藏品归属。

## 9. 库存锁

项目有库存锁表，用来避免同一件藏品同时被多个业务操作。

库存锁会用于这些场景：

- 市场挂单。
- 市场订单。
- 进化处理中。
- 分解处理中。
- 升级处理中。
- Mint 处理中。
- 系统锁定。

如果一件藏品有有效锁，页面会提示它当前受限制。后端也会拒绝不应该执行的操作。

简单说：同一件藏品不能一边卖、一边升级、一边分解、一边 Mint。

## 10. 出售功能

藏品详情里有“出售”入口。

可以出售的前提：

- 藏品属于当前用户。
- 藏品状态是可用。
- 藏品没有被锁定。
- 藏品本身允许交易。
- 藏品不在 Mint 流程中。

用户在弹窗里填写出售单价后提交。

页面会展示平台手续费和预计到账，但最终手续费和到账金额以后端返回为准。

出售成功后：

- 后端创建市场挂单。
- 对应藏品会被锁定。
- 藏品在库存中会表现为挂售中。
- 页面提示上架成功。

当前藏品详情里的出售入口一次只处理当前选中的这一件藏品。交易页的出售页另有批量选择和数量设置能力。

## 11. 下架功能

如果当前藏品已经挂售，详情区会显示“下架”入口。

用户确认下架后：

- 后端查找这件藏品对应的当前挂单。
- 只有挂单卖家可以下架。
- 后端取消挂单。
- 释放未售出藏品的库存锁。
- 页面提示下架成功。

如果藏品没有可下架挂单，后端会返回失败。

## 12. 升级功能

藏品详情里有“升级”入口。

可以升级的前提：

- 藏品属于当前用户。
- 藏品状态是可用。
- 藏品没有挂售。
- 藏品没有锁定。
- 藏品不在 Mint 流程中。
- 藏品本身允许升级。
- 还没有达到最高等级。
- 有可用升级规则。
- FGEMS 余额足够。

升级会消耗 FGEMS。

升级成功后：

- 藏品等级提升。
- 藏品战力提升。
- 写入升级记录。
- 写入库存事件。
- 写入资产流水。
- 页面展示升级成功结果。

这里有一个重要点：升级不会改变藏品形态。

代码里多次说明，形态里的星级是藏品生成时固定下来的形态，不是升级阶段。升级只改变等级和战力。

## 13. 进化功能

藏品详情里有“进化”入口。

可以进化的前提：

- 藏品属于当前用户。
- 藏品状态是可用。
- 藏品没有挂售。
- 藏品没有锁定。
- 藏品不在 Mint 流程中。
- 藏品本身允许进化。
- 用户有足够的同款同形态可用藏品。
- 有可用进化规则。
- KCOIN 余额足够。

当前进化规则要求选择三件同款同形态的源藏品。

进化面板会展示：

- 当前藏品。
- 目标藏品。
- 同款可用数量。
- 已选择材料。
- KCOIN 消耗。
- 成功率。
- 失败返还说明。

进化结果分两种：

成功时：

- 选中的源藏品会被消耗。
- 后端创建新的目标藏品。
- 新藏品归当前用户。
- 写入进化记录。
- 写入库存事件。
- 写入资产流水。
- 页面展示进化成功。

失败时：

- 后端会返还主藏品。
- 其他材料会被消耗。
- KCOIN 消耗以后端结果为准。
- 页面展示进化失败。

主藏品不是前端随便决定。后端按规则选出主藏品，代码里按等级、战力和获得时间来确定。

## 14. 分解功能

藏品详情里有“分解”入口。

可以分解的前提：

- 藏品属于当前用户。
- 藏品状态是可用。
- 藏品没有挂售。
- 藏品没有锁定。
- 藏品不在 Mint 流程中。
- 藏品本身允许分解。
- 用户至少还有同款同形态的重复藏品。
- 有可用分解规则。

当前藏品页的分解弹窗一次分解当前选中的这一件藏品。

分解成功后：

- 被分解的藏品状态变为已分解。
- 藏品不再归当前用户持有。
- 用户获得 FGEMS。
- 写入分解记录。
- 写入库存事件。
- 写入资产流水。
- 页面展示分解成功结果。

分解是不可恢复操作。页面也有对应提醒。

## 15. Mint 功能

藏品详情里有 Mint 入口。

可以 Mint 的前提：

- 藏品属于当前用户。
- 藏品状态是可用。
- 藏品没有挂售。
- 藏品没有锁定。
- 藏品本身允许 Mint。
- 藏品没有 Mint 成功过。
- 藏品没有正在 Mint 的队列。
- 用户已经连接并验证 TON 钱包。
- 钱包网络和当前 NFT Collection 网络一致。

用户点击 Mint 时，页面会先刷新藏品详情和钱包状态，确认没有阻塞原因。

确认后：

- 后端生成 Mint 请求。
- 藏品进入 Mint 队列。
- 后端锁定藏品。
- 藏品状态变为 Mint 中。
- 页面打开 Mint 队列。

Mint 队列里会展示：

- 进行中的数量。
- 成功数量。
- 需要处理的数量。
- 已同步 NFT 数量。
- 每条队列的状态。
- 已同步的钱包 NFT。

库存页面只负责把藏品提交到 Mint 队列和展示队列状态。链上交易确认和重试由后端定时任务和链上同步流程处理。

当前仓库里 `contracts/src` 下的合约源码文件是空文件，所以这份文档不写“合约已经实现完成”。

- 创建。
- 获得。
- 升级。
- 进化成功。
- 进化失败返还。
- 被消耗。
- 分解。
- 上架。
- 下架。
- 售出。
- 买入。

但当前前端藏品页没有看到调用这个接口的页面入口，所以不能写成“用户已经能在页面查看完整库存流水”。

## 17. 和资产的关系

库存功能会影响两种项目内资产：

- KCOIN。
- FGEMS。

升级消耗 FGEMS。

进化消耗 KCOIN。

分解获得 FGEMS。

出售和购买走交易市场，使用 KCOIN。

这些资产变化都必须由后端和数据库处理。前端看到的消耗、奖励、预计到账，只能作为展示和确认，不是最终结果。

## 18. 和交易市场的关系

库存里的出售入口会调用市场挂单能力。

出售后，这件藏品会被市场挂单锁住。

挂售中的藏品不能升级、进化、分解或 Mint。

下架后，服务端会释放未售出藏品的锁，藏品才能重新变成可操作状态。

## 19. 和图鉴的关系

进化成功时，代码会把新获得的目标藏品写入图鉴发现记录。

这说明库存成长会影响图鉴收集进度。

普通库存列表本身不等于图鉴。库存看的是用户当前拥有的实例；图鉴看的是用户是否曾经发现过某类藏品。

## 20. 前端和后端的职责边界

前端负责：

- 展示库存。
- 展示筛选和排序。
- 展示藏品详情。
- 打开操作弹窗。
- 提交用户操作。
- 展示操作结果。
- 在失败时给用户提示。

后端负责：

- 校验用户登录。
- 校验藏品归属。
- 校验藏品状态。
- 校验库存锁。
- 校验资产余额。
- 校验成长、分解、交易、Mint 规则。
- 执行数据库事务。
- 写库存事件和资产流水。
- 返回最终结果。

数据库负责：

- 保存每件藏品实例。
- 保存库存锁。
- 保存升级规则和记录。
- 保存进化规则和记录。
- 保存分解规则和记录。
- 保存库存事件。
- 保存 Mint 队列关联状态。

## 21. 安全和防重复处理

库存写操作都有几类保护：

- 必须有登录会话。
- 必须是藏品拥有者。
- 写操作经过后端接口。
- 数据库函数会再次检查归属和状态。
- 同一件藏品有锁时不能重复操作。
- 前端看到的预览如果过期，后端会拒绝并要求刷新。
- 写操作带防重复机制，避免重复提交造成重复扣款或重复发放。
- 升级、进化、分解、出售、下架等操作都有风控检查。

这类保护的目的很简单：用户不能操作别人的藏品，也不能让同一件藏品同时进入多个业务流程。

## 23. 总结

藏品库存不是单纯的“我有哪些卡片”。

它是整个项目的核心业务中心之一，连接了开盒产出、资产消耗、交易市场、成长系统、图鉴、钱包和 NFT Mint。

从当前代码看，库存功能已经覆盖：

- 查看库存。
- 分组展示。
- 筛选排序。
- 查看详情。
- 出售。
- 下架。
- 升级。
- 进化。
- 分解。
- Mint 入队。
- Mint 队列查看。

所有关键结果以后端和数据库为准，前端只负责展示和提交动作。
