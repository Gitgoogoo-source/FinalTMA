## 第 3 步：设计支付、钱包、onchain 数据库迁移

### 目标

在不破坏已有表的前提下，补齐第五阶段所需状态、索引、约束、审计字段和查询能力。

| payments  | `payments.star_orders`、`star_invoices`、`star_payments`、`telegram_webhook_events`、`star_refunds`、`payment_disputes`                    |

### 需要新增或修改的内容

不建议重建已有表。建议使用 migration 扩展：

| 表                                                        | 建议处理                                                                                                |
| --------------------------------------------------------- | ------------------------------------------------------------------------------------------------------- |
| `payments.star_orders`                                    | 扩展状态机：增加 `precheckout_checked` 或明确映射现有 `precheckout_ok`；增加 `fulfilling`、`disputed`。 |
| `payments.telegram_webhook_events`                        | 增加处理耗时、retry_count、next_retry_at、request_headers_hash、webhook_secret_verified 等字段。        |
| `payments.star_invoices`                                  | 增加 invoice 打开方式、invoice_link、bot_api_method、expires_at、last_opened_at。                       |
| `core.user_wallets`                                       | 增加状态表达或明确用 `verified_at` 派生 `verified`。                                                    |
| `core.wallet_proofs`                                      | 增加 proof nonce、request_id、used_at、wallet_public_key、proof_hash，防重放。                          |
| `core.wallet_connect_sessions`                            | 建议新增，用于 challenge 生命周期；也可复用 `wallet_proofs` pending 行。                                |
| `onchain.mint_queue`                                      | 扩展状态：`submitted`、`confirming`、`retrying`、`manual_review`。                                      |
| `onchain.transactions`                                    | 增加 transaction_type、external_api_provider、last_checked_at、check_count、raw_response。              |
| `onchain.wallet_sync_jobs`                                | 增加 idempotency_key、retry_count、next_retry_at、cursor。                                              |
| `ops.reconciliation_runs` / `economy.reconciliation_runs` | 当前在 `economy`，阶段五支付对账可以复用。                                                              |
| 关键索引                                                  | 补齐 performance advisor 中与支付 / 钱包 / onchain / gacha 相关的外键索引。                             |

### 后端开发工作

所有 API 调用必须基于新的状态机。
RPC 返回状态必须能被前端识别。
所有 webhook / mint worker 更新状态时需要带 request_id、source、错误原因。

### 数据库开发工作

添加状态枚举或 check constraint 扩展。
添加唯一约束：charge id、invoice payload、active mint queue、wallet proof challenge。
添加 partial index：active mint queue、pending webhook、pending payment、processing mint。
添加 RLS policy：用户只能看自己的钱包、Mint 状态、链上同步状态；后台只走 admin API。
确保 service role 能调用 RPC，但前端 anon role 不能直接写核心表。

### 前端开发工作

前端不直接读取表；只适配 API 返回的新状态。
更新状态文案映射：支付中、已支付、发货中、已完成、失败、过期、退款、争议。

### 安全和幂等要求

`telegram_payment_charge_id` 保持唯一。
`telegram_invoice_payload` 保持唯一。
`mint_queue` 同一 `item_instance_id` 不能有多个 active queue。
`wallet_proofs.challenge` 保持唯一且过期后不可复用。
所有支付 / Mint 状态更新必须可重复执行但不会重复发货 / 重复 Mint。

### 测试要求

迁移测试：旧数据仍可查询。
约束测试：重复 charge id 插入失败。
Mint 队列测试：同一藏品不能创建两个 active queue。
RLS 测试：用户不能读取他人钱包 / Mint 队列。
Performance 测试：关键列表查询使用索引。

### 验收标准

- [ ] 现有 payments 表未被重建。
- [ ] 支付状态机覆盖 created、invoice_created、precheckout_checked、paid、fulfilling、fulfilled、failed、expired、refunded、disputed。
- [ ] `telegram_payment_charge_id` 只能处理一次。
- [ ] 同一藏品只能有一个 active Mint 队列。
- [ ] 用户不能直接写 payments / onchain 核心表。
- [ ] Supabase performance advisor 中支付、钱包、onchain 相关关键索引已处理或记录延后原因。
