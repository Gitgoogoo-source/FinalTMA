## 第 8 步：前端 Stars 支付打开、等待和结果恢复

### 目标

让用户点击单抽 / 十连后，前端创建订单、打开 Telegram Stars 支付、显示支付状态、支持刷新后恢复订单结果。

### 需要先检查的现有文件 / 表

| 类型       | 路径                                                                        |
| ---------- | --------------------------------------------------------------------------- |
| 页面       | `apps/web/src/features/box/pages/BoxPage.tsx`                               |
| API client | `apps/web/src/features/box/box.api.ts`                                      |
| Hook       | `apps/web/src/features/box/hooks/useCreateOpenOrder.ts`、`useDrawResult.ts` |
| Component  | `PaymentPendingSheet.tsx`、`DrawResultModal.tsx`                            |
| Endpoints  | `apps/web/src/api/endpoints.ts`                                             |

### 需要新增或修改的内容

| 文件                                                             | 责任                                                                     |
| ---------------------------------------------------------------- | ------------------------------------------------------------------------ |
| `apps/web/src/features/box/hooks/useStarsPayment.ts`（建议新增） | 打开 Telegram invoice、处理支付窗口关闭、写入本地 pending order id。     |
| `PaymentPendingSheet.tsx`                                        | 增加支付中、支付取消、支付已打开、支付已成功待发货、发货失败补发中状态。 |
| `box.api.ts`                                                     | 增加支付订单状态字段 normalize。                                         |
| `queryKeys`                                                      | 增加 payment order / pending draw order query key。                      |
| `API_ENDPOINTS`                                                  | 增加支付订单状态查询 endpoint，或复用 `/boxes/result`。                  |

### 后端开发工作

提供订单状态查询 API。可以复用 `api/boxes/result.ts`，但建议新增：

| 建议 API                      | 责任                                                                     |
| ----------------------------- | ------------------------------------------------------------------------ |
| `api/boxes/payment-status.ts` | 返回 star_order、draw_order、payment、fulfillment 状态，不返回抽卡结果。 |

### 数据库开发工作

无需直接变更。
但订单状态查询 RPC 应避免暴露敏感 charge id、raw update、Bot 信息。

### 前端开发工作

用户流程：

1. 点击单抽 / 十连。
2. 调用 `createOpenOrder`。
3. 拿到 invoice link / payload。
4. 打开 Telegram Stars 支付。
5. 支付窗口关闭后不直接认为成功。
6. 展示等待 Sheet。
7. 轮询订单状态或点击“查看结果”。
8. completed 后展示 `DrawResultModal`。
9. 刷新页面后从本地 pending order id 或服务端最近订单恢复。

### 安全和幂等要求

前端不能因支付窗口返回成功就发货。
前端不能把 invoice payload 当作支付成功凭证。
重复点击必须依赖 idempotency key 和按钮锁。
刷新恢复只查服务端状态，不相信 localStorage 中的状态。

### 测试要求

支付窗口打开成功。
支付窗口取消。
支付成功但 webhook 延迟。
支付成功但发货处理中。
用户刷新页面后恢复 pending order。
订单过期。
重复点击单抽按钮。
十连价格展示和后端返回一致。

### 验收标准

- [ ] 点击单抽 / 十连可以打开 Telegram Stars 支付。
- [ ] 用户取消支付不会发货。
- [ ] 支付窗口关闭后前端不会自行判断成功。
- [ ] 支付成功后等待 webhook 发货。
- [ ] 发货完成后展示抽卡结果。
- [ ] 页面刷新后仍能恢复订单状态。
- [ ] 支付失败 / 过期 / 发货失败有明确提示。
