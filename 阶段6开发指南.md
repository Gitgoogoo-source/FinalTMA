以下指南基于我能读取到的 GitHub 项目代码和 Supabase 项目数据做规划。当前项目的基础业务 schema 已经比较完整：Supabase 里已经有 `core / economy / catalog / gacha / inventory / market / payments / tasks / album / onchain / ops` 等 schema，并且核心表都启用了 RLS；`public` schema 为空，业务数据主要集中在这些专用 schema 中。当前 `ops.admin_users`、`ops.admin_audit_logs`、`ops.risk_events`、`economy.reconciliation_runs`、`catalog.banner_campaigns`、`market.price_snapshots`、`market.depth_snapshots`、`market.price_health_rules` 等商业运营关键表仍为空。  

GitHub 代码层面，项目已经有 `apps/admin`、`api/admin`、`ops:*` 脚本、`requireAdmin` 权限体系、支付 / Mint / 钱包 / 监控 / Feature Flag 的初版后台，但当前 admin 页面只有 **监控、支付、Mint、钱包、开关** 5 个 tab，还没有概率版本、盲盒配置、活动配置、风控、对账中心、审计、用户运营、市场运营、任务运营等完整商业后台。

# 2. 每个功能的具体开发步骤

## 2.1 Admin 初始化与权限后台

### 完整业务闭环

管理员打开 Admin
→ 使用 Telegram session 或独立 admin session 登录
→ API 调用 `requireAdmin`
→ 查询 `ops.admin_users`
→ 查询 `ops.admin_user_roles` 和 `ops.admin_roles`
→ 校验 permissions
→ 返回后台菜单权限
→ 管理员执行写操作
→ `X-Admin-Confirm` 二次确认
→ `X-Idempotency-Key` 防重复
→ RPC 执行业务变更
→ 写入 `ops.admin_audit_logs`
→ 前端刷新数据
→ 操作失败时展示 requestId 和错误原因

当前代码里已经有 `requireAdmin`：先校验普通 app session，再查询 `ops.admin_users` 和角色权限，最后检查 permissions；也已经支持 `SUPER_ADMIN` 和通配权限。

### 开发步骤

1. 补齐 `scripts/create-admin.ts`。
   1. 先读取 `package.json`，确认现有入口是 `pnpm ops:create-admin`，不要新增第二套初始化命令。
   2. 打开 `scripts/create-admin.ts`，确认文件当前为空，再按项目现有脚本风格组织 `main()`、参数解析、JSON 输出和 `process.exitCode`。
   3. 复用后端已有 Supabase Admin Client 初始化方式，禁止在脚本里重新硬编码 Supabase URL、service role key 或数据库连接。
   4. 读取 `ADMIN_BOOTSTRAP_TELEGRAM_USER_IDS`，按英文逗号拆分，逐个 trim，过滤空值。
   5. 校验每个 Telegram user id 是正整数；非法值直接让脚本失败，并输出失败项。
   6. 查询 `ops.admin_roles` 中 `code = 'SUPER_ADMIN'` 的角色；不存在时脚本失败，提示先补 seed / migration，不要静默创建未知权限。
   7. 对每个 Telegram user id 查询 `core.users.telegram_user_id`，如果已存在 `core_user_id` 就绑定；如果不存在，则允许先以 `telegram_user_id` 创建 `ops.admin_users`，因为当前 `requireAdmin` 已支持通过 session 的 Telegram user id 回查管理员。
   8. 使用 `telegram_user_id` 做唯一键 upsert `ops.admin_users`，写入 `status = 'active'`、`display_name`、`metadata.source = 'bootstrap_script'`、`metadata.bootstrapped_at`。
   9. 对每个 admin user upsert `ops.admin_user_roles(admin_user_id, role_id)`，已存在时跳过，不重复插入。
   10. 写入 `ops.admin_audit_logs`，action 建议为 `admin.bootstrap_super_admin`，`admin_user_id` 可为新建管理员或 null，`after_state` 记录新增 / 已存在 / 绑定角色结果。
   11. 增加 `--dry-run` 模式：只打印将创建 / 将跳过 / 将授权的管理员，不写数据库。
   12. 增加 `--telegram-user-id` 单人模式，方便 staging 单独初始化测试管理员。
   13. 增加 `--role-code` 参数，默认 `SUPER_ADMIN`，但只允许传已存在角色 code。
   14. 输出 JSON 汇总：`created_count`、`updated_count`、`role_granted_count`、`skipped_count`、`errors`。
   15. 本地验证：执行 dry-run，再执行真实初始化，再用 Supabase 查询 `ops.admin_users`、`ops.admin_user_roles`、`ops.admin_audit_logs`。

2. 支持通过 `ADMIN_BOOTSTRAP_TELEGRAM_USER_IDS` 初始化超级管理员。
   1. 检查 `.env.example` 是否已有该变量说明；如果缺少，补充变量用途、格式和示例，但不要提交真实 Telegram user id。
   2. 检查 Vercel Production / Preview 是否配置该变量；未配置时，生产环境脚本必须失败并输出明确错误。
   3. 明确变量只用于 bootstrap，不作为运行时权限来源。
   4. 脚本执行后，后台权限以后端 `requireAdmin` 查询 `ops.admin_users` 为准，不再读取该环境变量判断管理员。
   5. 在文档中补充 bootstrap 完成后可清空该变量，避免长期保留超级管理员入口。

3. 在 `apps/admin/src/App.tsx` 增加登录态检查和权限菜单渲染，不同角色只显示有权限的页面。
   1. 先检查当前 `App.tsx` 的 `NAV_ITEMS`，确认现有 tab 只有 `monitoring / payments / mint / wallets / flags`。
   2. 新增 `AdminSessionProvider` 或 `useAdminMe` hook，不在组件里直接读取 cookie。
   3. 新增 `/api/admin/me`，返回 `adminId`、`roleCode`、`permissions`、`isSuperAdmin`、`serverTime`，不要返回 session token。
   4. `/api/admin/me` 必须复用 `requireAdmin`，不要另写一套管理员鉴权。
   5. 定义 `AdminNavItem.requiredPermissions`，例如 `payments:read`、`mint:read`、`feature_flags:read`。
   6. 菜单渲染前根据 `permissions` 和 `isSuperAdmin` 过滤，不满足权限的页面不显示。
   7. 直接访问无权限页面时显示 403 空状态，不只隐藏菜单。
   8. 增加 loading、session expired、forbidden、network error 四种状态。
   9. API 失败时展示 `requestId`，保持现有 `AdminApiError` 处理风格。
   10. 增加退出 / 重新进入 Telegram Mini App 的提示，不在前端伪造管理员身份。

4. 新增 Admin 页面：`AdminUsersPage`、`RolesPage`、`PermissionMatrixPage`。
   1. 在 `apps/admin/src/admin.types.ts` 增加 `AdminUser`、`AdminRole`、`AdminPermissionMatrix`、`AdminUsersResponse`、`AdminRolesResponse` 类型。
   2. 扩展 `AdminTab`，新增 `admins`、`roles`、`permissions`。
   3. 在 `App.tsx` 增加导航项，绑定对应权限：管理员用户 `admin:read`，角色配置 `roles:read`，权限矩阵 `permissions:read`。
   4. 创建 `apps/admin/src/pages/AdminUsersPage.tsx`，展示管理员列表、状态、角色、最后登录时间、创建时间、操作按钮。
   5. 创建 `RolesPage.tsx`，展示角色 code、名称、权限 JSON、绑定人数。
   6. 创建 `PermissionMatrixPage.tsx`，按业务域展示权限：payments、mint、onchain、feature_flags、audit、risk、gacha、market、users、reports。
   7. 每个页面都包含 loading、empty、error、refresh 状态，复用 `StatusBadge`、`formatDate`、`shortId`。
   8. 所有写按钮先只接入确认弹窗和 API 占位，不在前端直接改本地状态作为真相。

5. 新增 API：`/api/admin/admin-users`、`/api/admin/roles`、`/api/admin/grant-role`、`/api/admin/revoke-role`。
   1. 每个 API 文件都使用 `withApiHandler`，限制 method。
   2. 读接口使用 `requireAdmin(req, { permissions: ['admin:read', 'roles:read'], requireAll: false })`。
   3. 写接口使用 `requireAdmin(req, { permissions: ['admin:write', 'roles:write'], requireAll: false })`。
   4. 写接口必须调用 `parseJsonBody`，限制 body 大小。
   5. 写接口必须调用 `requireAdminConfirmation(req, body)`。
   6. 写接口必须调用 `readBodyIdempotencyKey(req, body)`。
   7. 所有 UUID 用 `normalizeRequiredUuid`，文本用 `normalizeRequiredText`，布尔用 `normalizeBoolean`。
   8. 读接口只查询必要字段，禁止返回 session、token、secret、完整 metadata 中的敏感项。
   9. 写接口不直接 update 多张表，统一调用 admin RPC。
   10. 错误使用 `ApiError` 或 `mapAdminRpcError`，不要把 Supabase 原始错误直接暴露给前端。

6. 新增 RPC：`admin_create_user`、`admin_update_user_status`、`admin_grant_role`、`admin_revoke_role`。
   1. 先检查 Supabase 是否已存在同名 RPC，存在则扩展，不重复创建。
   2. 所有 RPC 放在 `api` schema，参数使用 `p_` 前缀。
   3. RPC 使用 `SECURITY DEFINER` 时必须固定 `search_path`。
   4. RPC 内部校验 `p_admin_user_id` 是否 active admin。
   5. `admin_create_user` 校验 `telegram_user_id`、`email`、`core_user_id` 至少一个可绑定身份。
   6. `admin_update_user_status` 只允许 `active / disabled / locked`。
   7. `admin_grant_role` 校验 role 存在、admin user 存在、目标用户不是 disabled。
   8. `admin_revoke_role` 防止撤销系统最后一个 `SUPER_ADMIN`。
   9. 所有 RPC 接入 `ops.idempotency_keys`，同 key 同请求返回同结果，不同请求返回冲突。
   10. 所有 RPC 返回 `audit_log_id`、`idempotent`、`admin_user_id`、`server_time`。

7. 所有管理员变更必须写 `ops.admin_audit_logs`。
   1. 为每个 admin RPC 增加 before_state 查询。
   2. 业务变更完成后查询 after_state。
   3. 调用现有 `admin_write_audit_log` 或在同事务内 insert 审计日志。
   4. `action` 使用稳定命名：`admin.create_user`、`admin.update_status`、`admin.grant_role`、`admin.revoke_role`。
   5. `target_schema = 'ops'`，`target_table` 对应 `admin_users` 或 `admin_user_roles`。
   6. `reason` 必填，前端不填时 API 拒绝。
   7. `ip_hash` 和 `user_agent` 从 `buildAdminRpcContext` 传入，不记录明文 IP。
   8. 写测试确认审计日志不可通过普通前端路径修改。

8. 增加后台危险操作确认：暂停支付、修改概率、补偿资产、封禁用户、退款、释放锁定库存。
   1. 统一前端确认组件，替代散落的 `window.prompt` / `window.confirm`。
   2. 确认组件必须收集 reason。
   3. 确认组件必须要求输入目标 key / order id / user id 的短码，避免误点。
   4. 所有危险请求自动加 `X-Admin-Confirm: true`。
   5. 所有危险请求自动生成 `X-Idempotency-Key`，key 包含 action、target id、timestamp 或 uuid。
   6. API 端仍然强制校验 confirmation 和 idempotency，不能只依赖前端。
   7. 高风险动作预留二人审批字段，不在 Phase 6 初版也要保证数据库扩展空间。
   8. 为每类危险动作补充权限：`payments:write`、`gacha:write`、`users:ban`、`inventory:write`、`risk:write`。
### 测试

1. 非管理员访问 Admin，必须返回 403。
2. `inactive / suspended` 管理员访问后台，必须返回 403。
3. `SUPER_ADMIN` 能访问所有页面。
4. `SUPPORT` 只能看用户、支付、工单，不能改概率。
5. `OPS` 能改活动和概率，但不能处理退款和封禁。
6. 角色授予、移除必须写审计日志。
7. 重复点击同一个授权操作，幂等结果一致。
8. `ADMIN_BOOTSTRAP_TELEGRAM_USER_IDS` 不配置时，生产环境禁止启动 Admin 写操作。

## 2.2 审计日志中心

### 完整业务闭环

Admin 发起写操作
→ API 校验管理员权限
→ API 要求 reason 和二次确认
→ RPC 读取修改前数据
→ 执行业务变更
→ RPC 写入 `ops.admin_audit_logs.before_state / after_state`
→ 返回 `audit_log_id`
→ Admin 页面展示操作结果
→ 审计中心可按管理员、动作、表、目标 ID、时间范围查询
→ 需要回溯时导出审计记录

当前数据库已经有 `ops.admin_audit_logs` 表，包含 `admin_user_id / action / target_schema / target_table / target_id / before_state / after_state / ip_hash / user_agent / reason / created_at` 等字段；但当前表为空，说明还没有形成完整运营留痕。 

### 开发步骤

1. 新增 `AuditLogsPage`。
   1. 在 `apps/admin/src/admin.types.ts` 新增 `AdminAuditLog`、`AuditLogsResponse`、`AuditLogFilters` 类型。
   2. 在 `admin.api.ts` 新增 `fetchAuditLogs(params)`，沿用现有 `adminRequest` 和 `toQueryString`。
   3. 在 `App.tsx` 新增 `audit` tab，权限为 `audit:read` 或 `admin:read`。
   4. 创建 `apps/admin/src/pages/AuditLogsPage.tsx`。
   5. 页面顶部增加筛选栏：管理员、action、target_schema、target_table、target_id、from、to、q。
   6. 列表字段展示：时间、管理员、action、目标、reason、request id、before / after 摘要。
   7. 详情弹窗展示 before_state 和 after_state 的 JSON diff，但默认折叠大字段。
   8. 增加空状态、loading、error、刷新、分页。
   9. 所有错误展示稳定错误码和 requestId，不展示内部 SQL 错误。

2. 新增 API：`/api/admin/audit-logs`。
   1. 创建 `api/admin/audit-logs.ts`。
   2. 使用 `withApiHandler`，只允许 `GET`。
   3. 使用 `requireAdmin(req, { permissions: ['audit:read', 'admin:read'], requireAll: false })`。
   4. 复用 `parseAdminLimit`、`parseOffsetCursor`、`buildNextCursor`。
   5. 支持筛选参数：`adminUserId`、`action`、`targetSchema`、`targetTable`、`targetId`、`from`、`to`、`q`。
   6. 查询 `ops.admin_audit_logs`，按 `created_at desc, id desc` 排序。
   7. 关联 `ops.admin_users` 只返回 `id / display_name / telegram_user_id / email` 的安全字段。
   8. 对 `before_state / after_state` 做大小限制或摘要，详情接口再返回完整 JSON。
   9. 返回结构保持 `{ items, summary, nextCursor, serverTime }`。

3. 所有 admin 写接口统一使用 `admin_write_audit_log`。
   1. 先盘点现有 admin 写接口：`feature-flags.ts`、`retry-mint.ts`，确认它们都通过 RPC 返回 `audit_log_id`。
   2. 新增写接口时必须调用 `callAdminWriteRpc`，禁止 API 层直接写核心运营表。
   3. 每个新增 RPC 内统一调用 `api.admin_write_audit_log`。
   4. 如果某个旧 RPC 只改数据不写审计，先补 RPC，再接页面。
   5. 审计 action 命名统一为 `domain.verb_target`，例如 `feature_flag.update`、`payment.retry_fulfillment`。
   6. 为审计 action 建一份常量文档，避免同义 action 混用。

4. 每个后台写操作必须传：`reason`、`target_schema`、`target_table`、`target_id`、`before_state`、`after_state`。
   1. 前端危险操作弹窗统一要求 reason，最短长度建议 5 个字符。
   2. API 入参校验 reason，空 reason 返回 `VALIDATION_FAILED`。
   3. RPC 执行业务前读取 before_state。
   4. RPC 执行业务后读取 after_state。
   5. `target_id` 必须是被操作主记录 id；批量操作时 `after_state` 里记录 ids。
   6. 对补偿、退款、封禁等动作，`after_state` 必须包含影响资产 / 状态的摘要。
   7. 对敏感字段执行脱敏，不写 token、cookie、私钥、完整 initData。

5. 审计列表支持筛选：管理员、动作、目标表、目标 ID、时间范围、风险等级。
   1. 数据库现有 `admin_audit_logs` 没有风险等级字段，先在 API 层按 action 映射 `risk_level`。
   2. 后续如需持久化风险等级，通过 migration 增加 `risk_level`，不要临时塞进 reason。
   3. 前端筛选项使用 query string，刷新页面后筛选条件可复现。
   4. target id 支持完整 UUID 精确查询。
   5. action 支持 ilike 模糊搜索。
   6. 时间范围统一按 UTC 传给 API，前端只负责本地展示。

6. 支持导出 CSV，但导出接口必须有 `audit:export` 权限。
   1. 新增 `/api/admin/audit-logs/export`，只允许 `GET` 或 `POST` 中一种，避免重复接口。
   2. 使用 `requireAdmin(req, { permissions: 'audit:export' })`。
   3. 导出必须复用审计列表的筛选条件。
   4. 限制最大导出行数，超过时返回错误并提示缩小范围。
   5. CSV 中只导出脱敏字段，不导出完整 before_state / after_state 中的敏感内容。
   6. 导出动作本身也写一条 `audit.export` 审计日志。

7. 高风险操作增加二人复核状态：`pending_approval → approved → executed / rejected`。
   1. 先通过 migration 新增 `ops.admin_approval_requests`，不要把审批状态塞进 audit log。
   2. 字段至少包含：requester、approver、action、target、payload、status、reason、review_reason、created_at、reviewed_at、executed_at。
   3. 高风险 RPC 增加 `requires_approval` 分支。
   4. pending 状态只记录申请，不执行业务写入。
   5. 审批人不能等于申请人。
   6. approved 后再调用原业务 RPC 执行，并关联 approval id。
   7. rejected 只写审核结果和审计，不改业务表。

8. 不允许直接 update / delete 审计日志。需要更正时只能追加 correction 记录。
   1. 检查 RLS / 权限，禁止普通 authenticated 修改 `ops.admin_audit_logs`。
   2. 禁止新增 update / delete 审计日志 API。
   3. 如需更正，新增 `admin.append_audit_correction` RPC。
   4. correction 记录 `action = 'audit.correction'`，`target_id` 指向原审计日志 id。
   5. correction 只追加说明，不覆盖原 before / after。
   6. 在 AuditLogsPage 详情里展示 correction 链路。
### 测试

1. 修改 feature flag 后必须生成一条审计日志。当前 feature flag 写接口已有确认、幂等和 RPC 调用模式，可作为其他后台写接口的模板。
2. 未填写 reason 的危险操作必须拒绝。
3. 重复提交同一个 idempotency key 不得重复写业务变更。
4. 审计日志查询不能泄露敏感密钥、Bot Token、私钥、session secret。
5. 普通 SUPPORT 角色不能导出全量审计日志。
6. 审计日志创建后不可被普通 API 修改。

## 2.3 概率版本 / 奖励池后台

### 完整业务闭环

运营创建概率池草稿
→ 选择盲盒
→ 添加奖励项、稀有度、权重、库存、是否保底 eligible
→ 系统计算概率和总权重
→ 校验概率、库存、藏品状态
→ 草稿保存
→ 预览「可能获得」和保底规则
→ 管理员发布版本
→ 旧 active 版本进入 archived / inactive
→ 新版本写入 `gacha.drop_pool_versions`
→ 奖励项写入 `gacha.drop_pool_items`
→ 保底规则写入 `gacha.pity_rules`
→ 审计日志记录 before / after
→ 前端开盒读取 active 版本
→ 历史订单继续关联历史版本，不能被新版本影响

当前 Supabase 已有 `gacha.drop_pool_versions`、`gacha.drop_pool_items`、`gacha.pity_rules`，并且已有版本和奖励项数据；数据库函数里也已经有 `refresh_drop_pool_total_weight` 触发器。 

### 开发步骤

1. 在 Admin 增加 `GachaPoolsPage`。
   1. 先检查 `apps/admin/src/App.tsx` 的现有 tab，确认没有 gacha / pools 页面后再新增。
   2. 在 `admin.types.ts` 增加 `BlindBoxAdminItem`、`DropPoolVersion`、`DropPoolItem`、`PityRule`、`DropPoolValidationResult` 类型。
   3. 在 `admin.api.ts` 新增读取盲盒、读取池版本、读取奖励项、保存草稿、校验、发布、归档的函数。
   4. 在 `AdminTab` 增加 `gacha-pools`。
   5. 在导航中新增“概率池”菜单，权限使用 `gacha:read`。
   6. 页面左侧展示盲盒列表，右侧展示该盲盒的版本列表和当前 active 版本。
   7. 草稿编辑区按奖励项展示藏品、稀有度、权重、概率、库存、保底 eligible。
   8. 页面必须显示当前 active version，防止运营误改历史版本。
   9. 所有保存 / 发布按钮都显示 loading、错误码、requestId。

2. 新增 API：`/api/admin/gacha/boxes`、`/api/admin/gacha/drop-pool-versions`、`/api/admin/gacha/drop-pool-items`、`/api/admin/gacha/pity-rules`、`/api/admin/gacha/publish-drop-pool`。
   1. 新建 `api/admin/gacha/` 目录，避免把所有 gacha 后台接口塞进一个大文件。
   2. 每个文件使用 `withApiHandler` 和 `requireAdmin`。
   3. 读接口权限为 `gacha:read` 或 `admin:read`。
   4. 写接口权限为 `gacha:write` 或 `admin:write`。
   5. 写接口统一要求 `X-Admin-Confirm`、`X-Idempotency-Key`、`reason`。
   6. 入参 UUID 使用 `normalizeRequiredUuid`，文本状态使用白名单校验。
   7. 读接口只返回运营需要字段，不返回内部随机审计 secret。
   8. `publish-drop-pool` 只调用 RPC，不在 API 层自己改 `active` 状态。
   9. 每个接口加 rate limit：读 `admin.read`，写 `admin.write`。

3. 新增 RPC：`admin_create_drop_pool_draft`、`admin_update_drop_pool_item`、`admin_validate_drop_pool`、`admin_publish_drop_pool_version`、`admin_archive_drop_pool_version`。
   1. 先用 Supabase 查询确认同名 RPC 不存在，避免重复创建。
   2. 所有 RPC 放在 `api` schema，使用 `p_` 参数前缀。
   3. 所有写 RPC 接收 `p_admin_user_id`、`p_reason`、`p_idempotency_key`、`p_request_context`。
   4. `admin_create_drop_pool_draft` 从某个 active / archived 版本 clone，或从空白创建。
   5. `admin_update_drop_pool_item` 只允许修改 draft / validating 版本。
   6. `admin_validate_drop_pool` 只做校验，不改变线上 active 版本。
   7. `admin_publish_drop_pool_version` 在同一事务中归档旧 active、发布新 active、写审计。
   8. `admin_archive_drop_pool_version` 禁止归档仍被未完成订单引用的 active 版本，除非先发布替代版本。
   9. RPC 返回 `validation_errors`、`warnings`、`audit_log_id`、`idempotent`。

4. 概率池状态建议：`draft / validating / scheduled / active / archived / disabled`。
   1. 检查 `gacha.drop_pool_versions.status` 当前约束；不支持这些值时先写 migration。
   2. migration 中用 check constraint 或 enum 约束状态值，不靠前端限制。
   3. draft：允许编辑奖励项。
   4. validating：正在校验，不允许并发编辑。
   5. scheduled：已校验，等待上线时间。
   6. active：线上抽卡使用，只读不可编辑。
   7. archived：历史版本，只读可追溯。
   8. disabled：异常下线版本，不能被开盒使用。

5. 发布前校验。
   1. 校验奖励池至少有 1 个奖励项。
   2. 校验总权重大于 0。
   3. 校验每个奖励项 `drop_weight > 0`。
   4. 校验概率换算后的 bps 总和在允许误差内。
   5. 校验每个奖励项关联的 `collectible_template` 存在且 active / 可投放。
   6. 校验每个奖励项关联的 form 存在且属于该 template。
   7. 校验库存字段不为负；有限库存不得小于已锁定 / 已发放量。
   8. 校验至少存在一个保底 eligible 奖励，且满足 `pity_rules` 的稀有度要求。
   9. 校验同一盲盒同一时间只有一个 active / scheduled 覆盖窗口。
   10. 校验盲盒本身不是 archived / disabled。
   11. 校验发布人拥有 `gacha:write`。

6. 发布时生成 `config_snapshot`，保存完整概率、库存、保底、发布人、发布时间。
   1. 在发布 RPC 内构造 snapshot，不允许前端传完整 snapshot 当作事实。
   2. snapshot 包含 box_id、pool_version_id、items、weight、computed_probability_bps、pity_rules、stock、published_by、published_at。
   3. snapshot 中记录校验结果 hash，方便后续审计。
   4. 将 snapshot 写入 `gacha.drop_pool_versions.metadata` 或专用字段；若字段不存在，通过 migration 增加。
   5. `draw_audit` 必须能追溯到该 pool version 和 snapshot。

7. 已发布版本禁止直接编辑，只允许 clone 成新版本。
   1. API 层发现版本 status 为 active / archived 时拒绝 update。
   2. RPC 层再次校验 status，防止绕过 API。
   3. UI 上 active / archived 版本只显示“克隆为草稿”和“查看详情”。
   4. clone 时复制奖励项、保底规则、展示配置，但新版本 status 必须是 draft。
   5. clone 记录 `source_version_id`，方便追踪。

8. 前端「可能获得」接口只读 active / scheduled 版本。
   1. 检查现有 web 开盒页读取奖励池的 API，确认不依赖后台草稿。
   2. 后台草稿接口与用户前端接口分离。
   3. 用户前端只返回 active 或已到时间的 scheduled 版本。
   4. 不向前端返回内部权重计算细节以外的管理字段。
   5. 发布后刷新前端 query cache，确保「可能获得」展示新版本。

9. 增加概率版本对比视图：展示新旧概率差异、稀有度概率变化、库存变化。
   1. 在 `GachaPoolsPage` 增加“对比 active”按钮。
   2. API 返回两个版本的 items 和 pity rules。
   3. 前端计算展示差异即可，但发布校验仍以后端为准。
   4. 差异维度包括新增奖励、删除奖励、权重变化、概率变化、库存变化、保底 eligible 变化。
   5. 高风险变化高亮：Legendary 概率降低、保底次数增加、库存减少。
   6. 发布确认弹窗中展示差异摘要，并要求输入 reason。
### 测试

1. 发布空奖励池必须失败。
2. 发布总权重为 0 必须失败。

8. 高并发发布同一个盲盒版本，只能成功一个。
9. 所有发布、归档、回滚都必须写审计日志。

## 2.5 支付运营与发货补偿

### 完整业务闭环

Telegram webhook 到达
→ 原始 webhook 写入 `payments.telegram_webhook_events`
→ 校验 webhook secret
→ 校验 invoice payload
→ 写 `payments.star_payments`
→ 调用 `gacha_process_paid_order`
→ 更新 `payments.star_orders.status = fulfilled`
→ 生成 `gacha.draw_results` 和库存
→ 写 ledger
→ 前端展示结果
→ 如果失败，订单进入异常状态
→ Admin 支付页面发现异常
→ 管理员点击重试 / 退款 / 标记争议
→ API 校验 `payments:write`
→ RPC 幂等重试发货或记录退款 / 争议
→ 写审计日志
→ 用户收到通知

当前 admin 支付页已经能读取 `star_orders`、`star_payments`、`telegram_webhook_events`、`star_refunds`、`payment_disputes`，但主要是查询；Supabase 中已有 `star_orders = 7`、`star_payments = 6`，但 `telegram_webhook_events = 0`，说明 webhook 原始事件落库还没有形成生产级闭环。  

### 开发步骤

1. 确认 `/api/telegram/webhook` 对所有事件先写 `payments.telegram_webhook_events`，再处理业务。
   1. 打开现有 `/api/telegram/webhook`，梳理 `pre_checkout_query`、`successful_payment`、未知 update 的处理顺序。
   2. 确认 webhook secret 校验发生在业务发货前。
   3. 在任何业务处理前插入原始事件落库逻辑。
   4. 落库字段包含 `update_id`、`event_type`、`telegram_user_id`、`invoice_payload`、`process_status = 'received'`、`webhook_secret_verified`、`status_context`。

   1. `create-refund-record` 只能写 `payments.star_refunds` 和订单状态，不伪造外部退款成功。

   2. 重试发货不重复插入 draw_results。
   3. 补偿资产必须通过 ledger 或 item event，不直接改余额。
   4. 每次补偿写 `ops.admin_audit_logs`。
   5. 每次补偿写 `ops.risk_events` 或关联风险处理记录，方便对账追踪。
   6. 并发点击同一重试按钮时只有一次实际发货成功。
### 测试

1. 重复 webhook 只发货一次。
2. `telegram_payment_charge_id` 重复时不得重复生成库存。
3. paid 订单重试发货后，`draw_results` 数量等于订单 draw_count。
4. 已 fulfilled 订单再次点击重试，返回 idempotent。
5. 支付失败订单不能生成库存。
6. 退款记录不会自动回滚已发放藏品，必须有明确补偿策略。
7. 争议订单进入 `disputed` 后，用户相关操作可被风控限制。
8. webhook secret 不通过时必须记录异常但不发货。
9. 管理员手动重试必须写审计。

## 2.6 对账中心

### 完整业务闭环

定时任务或管理员手动点击对账
→ 调用对账 API
→ 校验 `ops:read / ops:write` 权限或 Cron Secret
→ 运行对账 job
→ 写入 `economy.reconciliation_runs`
→ 发现问题写入 `ops.risk_events`
→ Admin 对账页面展示 run 和 findings
→ 管理员查看异常详情
→ 选择重试、补偿、忽略、升级处理
→ 处理动作写审计
→ 异常状态关闭

当前代码已经有 `runPhase5Reconciliation`，支持 `payment_fulfillment / mint_queue / wallet_sync / ledger_balance` 四类检查，并且会创建 `reconciliation_runs`、写 `risk_events`。  

### 开发步骤

   2. 在 `admin.api.ts` 增加 `fetchReconciliationRuns`、`runReconciliationNow`、`fetchReconciliationFindings`、`resolveReconciliationFinding`。
   3. 在 `AdminTab` 增加 `reconciliation`，菜单权限为 `ops:read` 或 `risk:read`。
   4. 页面顶部展示最近一次对账状态、异常总数、critical 数、risk event 数。

   1. 新增 `/api/cron/reconciliation`。
   2. Cron API 校验 `CRON_SECRET`。
   3. 根据 query 或 body 选择 runTypes。
   4. Vercel Cron 配置分频触发不同 runTypes。
   5. 每次 cron 写 `economy.reconciliation_runs` 或 job run 记录。
   6. 失败时创建 risk_event 或 app_event。
   7. 同一 run type 增加任务锁，避免并发执行。

   1. 前端“立即对账”弹窗提供 dry-run switch。
   2. dry-run 默认 true，降低误操作风险。

## 2.7 风控中心

### 完整业务闭环

用户触发行为：开盒、交易、邀请、支付、钱包、Mint
→ API 或对账 job 记录风险信号
→ 写入 `ops.risk_events`
→ 根据规则累计风险分
→ 严重风险写入 `core.user_flags`
→ 业务 API 读取 user_flags 并限制功能
→ Admin 风控页面展示风险事件
→ 管理员处理：忽略、限制、解封、补偿、升级客服
→ 处理动作写审计
→ 用户状态和业务权限更新

当前数据库已经有 `ops.risk_events` 和 `core.user_flags`，但 `risk_events` 为空，说明风控还没有投入运行。 

### 开发步骤

1. 新增 `RiskCenterPage`。
   1. 在 `admin.types.ts` 增加 `RiskEvent`、`RiskUserProfile`、`UserFlag`、`RiskEventsResponse` 类型。
   2. 在 `admin.api.ts` 增加 `fetchRiskEvents`、`fetchRiskUserProfile`、`resolveRiskEvent`、`applyUserFlag`、`clearUserFlag`。
   3. 在 `AdminTab` 增加 `risk`，菜单权限为 `risk:read`。
   4. 页面展示风险事件列表，字段包括 severity、status、event_type、user_id、source_type、source_id、created_at。
   5. 支持筛选 severity、status、event_type、userId、sourceId、时间范围。
   6. 详情区展示风险上下文、关联订单、钱包、市场挂单、支付、对账 run。
   7. 处理按钮包括：开始处理、忽略、标记已修复、升级、应用用户限制、解除用户限制。

   5. `user-profile` 聚合 `core.users`、`core.user_flags`、`payments.star_orders`、`market.orders`、`tasks.referrals`、`core.user_wallets` 的摘要。
   6. `resolve` 只更新风险事件状态，不删除记录。
   7. `apply-user-flag` 调用 RPC 写 `core.user_flags`，不直接 update。

   1. 先检查 Supabase 是否已有同名函数。
   2. `risk_record_event` 给业务 API 和对账 job 复用，保证风险事件写入格式一致。
   3. `admin_resolve_risk_event` 校验风险事件存在、状态可流转。
   4. `admin_apply_user_flag` 校验 flag 类型合法、过期时间合法、目标用户存在。
   5. `admin_clear_user_flag` 只允许清除 active flag，并记录解除原因。
   6. 所有 admin RPC 写 audit log。
   7. 所有 RPC 接入幂等 key。

4. 风险类型建议。
   1. 在文档或 seed 中定义稳定 risk event type 常量。
   2. 支付类：`payment_duplicate_webhook`、`payment_paid_not_fulfilled`、`payment_disputed`。
   3. 抽卡类：`gacha_high_frequency`、`gacha_stock_mismatch`、`gacha_fulfillment_mismatch`。
   4. 市场类：`market_self_trade`、`market_price_manipulation`、`market_abnormal_cancel_rate`。
   5. 邀请类：`referral_abuse`、`referral_self_loop`、`referral_multi_account`。
   6. 钱包类：`multi_account_wallet`、`wallet_proof_replay`、`wallet_sync_stuck`。
   7. Mint 类：`mint_retry_exceeded`、`mint_confirmed_queue_not_minted`。
   8. 账本类：`ledger_balance_mismatch`、`negative_balance_detected`。

5. 风控动作建议。
   1. 在 `core.user_flags` 中定义 flag key 白名单。
   2. 开盒限制：`gacha_blocked`。
   3. 市场购买限制：`market_buy_blocked`。
   4. 市场出售限制：`market_sell_blocked`。
   5. 任务奖励限制：`task_reward_blocked`。
   6. Mint 限制：`mint_blocked`。
   7. 资产冻结：`kcoin_frozen`、`fgems_frozen`。
   8. 客服只读：`support_review_required`。
   9. 每个 flag 支持 reason、expires_at、created_by_admin_id。

6. 业务 API 必须在关键写操作前读取 user_flags。
   1. 开盒订单创建前检查 `gacha_blocked`。
   2. 市场购买前检查 `market_buy_blocked` 和资产冻结。
   3. 市场出售前检查 `market_sell_blocked`。
   4. 任务领取前检查 `task_reward_blocked`。
   5. 合成 / 分解前检查库存和资产冻结。
   6. Mint 请求前检查 `mint_blocked`。
   7. 检查逻辑放后端 API 或 RPC，不放前端。
   8. 被拦截时返回稳定错误码 `RISK_REJECTED`，并带 requestId。

7. 增加风险评分：low / medium / high / critical。
   1. 沿用 `ops.risk_events.severity` 的 check constraint。
   2. `risk_record_event` 根据事件类型给默认 severity。
   3. 对同一用户近期多次事件增加 score_delta。
   4. 页面按 severity 排序，critical 置顶。
   5. 超过阈值自动建议 user flag，但是否自动执行要由配置控制。

8. 增加用户维度画像：设备、IP hash、钱包地址、交易对象、邀请关系、支付历史、订单失败率。
   1. 用户画像 API 只返回 hash 或摘要，不返回明文 IP。
   2. 展示关联钱包数、钱包被其他用户绑定次数。
   3. 展示近期支付成功 / 失败 / 争议数量。
   4. 展示市场交易对象 Top N。
   5. 展示邀请人、被邀请人数、首开盒转化。
   6. 展示最近风险事件时间线。
   7. 数据量大时分页加载，不一次性扫全表。

9. 所有封禁 / 解封 / 限制必须写审计。
   1. apply flag 写 audit action：`risk.apply_user_flag`。
   2. clear flag 写 audit action：`risk.clear_user_flag`。
   3. resolve event 写 audit action：`risk.resolve_event`。
   4. before_state 记录原 flag / event 状态。
   5. after_state 记录新 flag / event 状态。
   6. reason 必填。
   7. SUPPORT 角色默认只能读，不能执行 flag 写操作。
### 测试

1. 被标记 `gacha_blocked` 的用户不能创建开盒订单。
2. 被标记 `market_blocked` 的用户不能购买和出售。
3. 同一用户短时间高频创建订单，生成 risk_event。
4. 自买自卖或同设备账号互刷，生成 risk_event。
5. 邀请关系异常循环，生成 risk_event。
6. 管理员关闭风险事件后，状态变为 fixed / ignored，并写审计。
7. SUPPORT 只能查看风险，不可封禁。
8. RISK 角色可以处理风险，但不能改概率。

## 2.8 商业监控与告警

业务 API / Cron / webhook / RPC 产生日志和指标

4. 增加阈值配置。
   1. 在 `ops.system_settings` 中增加监控阈值 key，例如 `monitoring.thresholds`。
   2. 默认支付失败率 > 5% 为 warning / critical。

   6. ledger mismatch > 0 为 critical。
   7. 库存为负为 critical。
   8. K-coin 净发行超阈值为 warning。
   9. 阈值修改走 Admin RPC 和审计日志。

5. 接入 Sentry / Axiom / Logtail。

   2. 告警状态包括 open、acknowledged、resolved、ignored。
   3. ack 操作要求 reason，但不要求业务修复。
   4. resolved 要求填写处理结果。
   5. 告警状态变更写审计。
   6. Dashboard 只展示 open / acknowledged 未关闭告警。

7. 告警处理动作必须能跳转到相关订单、用户、风险事件、对账 run。
   1. alert detail 存 `source_type` 和 `source_id`。
   2. source_type 为 star_order 时跳支付详情。
   3. source_type 为 mint_queue 时跳 Mint 队列详情。
   4. source_type 为 risk_event 时跳风控中心。
   5. source_type 为 reconciliation_run 时跳对账中心。
   6. source_type 为 user 时跳用户详情。
   7. 前端为未知 source_type 显示纯文本，不报错。
### 测试

## 2.9 市场运营后台

### 完整业务闭环

定时任务刷新市场数据
→ 调用 `market_refresh_price_stats`
→ 写入 `market.price_snapshots`、`market.depth_snapshots`

→ 所有动作写审计

当前项目已有 `scripts/rebuild-market-stats.ts`，调用 `api.market_refresh_price_stats`，但 Supabase 中 `market.price_snapshots`、`market.depth_snapshots`、`market.price_health_rules` 目前都是 0。 

### 开发步骤

1. 新增 `MarketOpsPage`。
   1. 在 `admin.types.ts` 增加 `MarketOpsStats`、`MarketListingAdminItem`、`MarketPriceRule`、`MarketHealthRule` 类型。
   2. 在 `admin.api.ts` 增加市场运营相关 fetch 和 mutation 函数。
   3. 在 `AdminTab` 增加 `market-ops`，权限为 `market:read`。
   4. 页面顶部展示市场概览：活跃挂单、成交额、手续费、异常挂单。
   5. 列表展示 active / sold / cancelled / expired 挂单。
   6. 支持按稀有度、模板、价格区间、用户、状态筛选。
   7. 异常挂单独立分组：过低价、过高价、自买自卖疑似、锁异常。
   8. 所有强制操作需要 reason 和二次确认。

2. 新增 API：`/api/admin/market/stats`、`/api/admin/market/price-rules`、`/api/admin/market/health-rules`、`/api/admin/market/listings`、`/api/admin/market/force-cancel-listing`。
   1. 读接口使用 `market:read` 或 `admin:read`。
   2. 写接口使用 `market:write` 或 `admin:write`。
   3. `stats` 调用或读取 `market.price_snapshots`、`market.depth_snapshots`、`market.orders`。
   4. `listings` 支持分页和筛选，不一次性返回全市场。
   5. `price-rules` 管理 `catalog.market_price_rules` 或相关配置表。
   6. `health-rules` 管理 `market.price_health_rules`。
   7. `force-cancel-listing` 只能调用专用 RPC，不直接改 listing 状态。
   8. 写接口必须写审计和风险事件。

3. 后台展示。
   1. 总挂单数：按 `market.listings.status = active` 聚合。
   2. 总挂单价值：active listings 的价格和数量合计。
   3. 24h 成交额：按 `market.orders.created_at` 窗口聚合。
   4. 地板价：每个 template / form 的最低 active listing price。
   5. 平均价：按成交或 active 挂单分别展示，避免混淆。
   6. 最近成交价：按最新 market order 获取。
   7. 异常低价 / 高价挂单：按 price health rule 判断。
   8. 同用户异常刷单：按买卖双方、设备、钱包、IP hash 关联风控。
   9. 平台手续费收入：按 `market.fee_settlements` 或 ledger source 聚合。

4. 配置价格健康规则。
   1. 检查 `market.price_health_rules` 表字段，不够时先 migration。
   2. 支持按 rarity 设置默认规则。
   3. 支持按 template 覆盖默认规则。
   4. 支持按 form 覆盖 template 规则。
   5. 默认低于地板价 70% 为过低，高于 130% 为过高。
   6. 保存规则时校验 low_bps < 10000 < high_bps。
   7. 规则变更后触发 `market_refresh_price_stats` 或标记下次刷新。
   8. 规则变更写审计。

5. 手续费规则接入 `economy.fee_rules`，后台可改，但订单保存 fee snapshot。
   1. 新增 Fee Rules 管理区，读取 `economy.fee_rules`。
   2. 修改手续费必须通过 admin RPC。
   3. fee bps 校验范围，例如 0 到 3000，不允许负数。
   4. 改动生效时间支持立即或 scheduled。
   5. 市场成交 RPC 必须保存 fee snapshot，不能成交后再读当前规则覆盖历史订单。
   6. 修改手续费写审计。

6. 定时任务每 5～15 分钟刷新市场统计。
   1. 复用现有 `scripts/rebuild-market-stats.ts` 调用 `api.market_refresh_price_stats` 的模式。
   2. 新增 `/api/cron/rebuild-market-stats`。
   3. Cron API 校验 `CRON_SECRET`。
   4. 任务开始和结束写 `ops.app_events` 或 job run。
   5. 失败时写 risk_event 或 error app_event。
   6. Admin 提供手动刷新按钮，要求 reason 和二次确认。

7. 异常挂单处理：提醒卖家、强制下架、冻结交易、创建 risk_event。
   1. `force-cancel-listing` RPC 校验 listing active。
   2. 同事务释放 `inventory.inventory_locks`。
   3. 写 `market.listing_events`，reason 标记为 admin_force_cancel。
   4. 必要时写 `core.notifications` 通知卖家。
   5. 疑似操纵价格时写 `ops.risk_events`。
   6. 高风险用户可同步写 `core.user_flags` 限制市场操作。
   7. 已成交 listing 禁止强制下架，只能走争议 / 补偿流程。
### 测试

1. 有 active listing 时，刷新后生成 price_snapshots。
2. 有多个价格区间时，生成 depth_snapshots。
3. 低价挂单展示 price health warning。
4. 强制下架后 item lock 释放。
5. 已成交挂单不能被强制下架。
6. 修改手续费后，新订单使用新 fee rule，旧订单不变。
7. 刷新市场统计脚本失败时写 app_event 或 risk_event。

## 2.10 用户 / 客服后台

   3. ledger 中记录 source_type、source_id、idempotency_key、admin_user_id。
   4. 藏品补偿写 item event，事件类型为 `admin_granted`。
   5. 禁止通过 Admin 页面直接编辑 balance 快照。
   6. 对账 job 必须能识别 admin compensation 来源。

7. 客服工单状态：`open / pending_user / pending_ops / resolved / rejected / escalated`。
   1. 检查 `ops.support_tickets.status` 约束。
   2. 缺少状态时通过 migration 扩展。
   3. 新增 API 支持创建、分配、更新状态、添加处理结果。
   4. 工单详情展示关联支付订单、用户、风险事件、补偿请求。
   5. resolved 必须填写 resolution。
   6. rejected 必须填写拒绝原因。
   7. escalated 必须指定后续处理人或队列。
   8. 状态变更写审计。

8. 用户封禁或限制走 `core.user_flags`。
   1. 用户详情页的限制按钮复用风控中心的 `applyUserFlag`。
   2. 不新增直接 update `core.users.status` 的快捷按钮，除非有专门 RPC。

# 4. 第六阶段开发注意事项

## 4.1 不要直接改余额、库存、订单状态

生产环境中，后台也不能直接 update `user_balances`、`item_instances`、`star_orders`。所有资产变化必须通过 RPC，必须写 ledger / event / audit。否则后续对账无法解释资产来源。

## 4.2 概率池只能版本化，不能覆盖线上版本

已发布的 `drop_pool_versions` 不允许直接编辑。运营改概率必须 clone 新版本，发布新版本，旧版本归档。历史订单必须能查到当时的 pool version 和概率快照。

## 4.3 支付 webhook 必须先落库再处理

当前 `payments.telegram_webhook_events` 为空，不适合商业化运行。生产环境必须保存原始 webhook event、处理状态、重试次数、错误信息。否则支付争议和重复 webhook 无法追溯。

## 4.4 Admin 写操作必须全部有确认、幂等和审计

已有 feature flag 写接口使用了 `X-Admin-Confirm`、`X-Idempotency-Key`、RPC 和 request context，这个模式应复制到所有后台写操作。 

## 4.5 先补管理员初始化，否则 Admin 无法真正上线

Supabase 里 `ops.admin_users = 0`，同时 `scripts/create-admin.ts` 为空。第六阶段第一件事应是补齐 bootstrap admin 流程，而不是继续开发更多后台页面。 

## 4.6 补齐空脚本，避免 package scripts 只是占位

`package.json` 已经暴露 `ops:retry-payments` 和 `ops:retry-mints`，但两个脚本文件目前为空。上线前必须实现，或者从 package scripts 中移除，避免运营误以为已有自动恢复能力。  

## 4.7 处理 Supabase 性能 advisor 提示

Supabase performance advisor 已提示多张表存在未索引外键。商业化上线前，应为高频查询和外键列补索引，尤其是 gacha、inventory、market、ledger、payments、tasks、onchain 相关外键。

## 4.8 Feature Flag 要分阶段开启

不要一次性打开支付、Mint、worker、webhook fulfillment。建议顺序：

1. staging 打开 `FEATURE_STARS_PAYMENT_ENABLED`。
2. 使用真实 Telegram 测试订单。
3. 打开 `FEATURE_PAYMENT_WEBHOOK_FULFILLMENT_ENABLED`。
4. 跑对账，确认 paid / fulfilled / draw_results / ledger 一致。
5. 小流量 production 开启 Stars 支付。
6. 再打开 Mint 请求。
7. 最后打开 Mint worker。

## 4.9 所有运营配置都走 migration 或 Admin RPC

不要直接在 Supabase Dashboard 手动改生产表。必须通过 migration、seed、Admin RPC 或审核过的脚本修改，并产生审计日志。

## 4.10 上线前必须做恢复演练

至少演练：

支付成功但未发货如何补发；
发货成功但前端未展示如何查询结果；
重复 webhook 如何确认未重复发货；
市场成交异常如何回溯；
ledger mismatch 如何冻结和修复；
Mint 提交成功但链上确认超时如何重试；
错误概率版本发布后如何暂停盲盒；
管理员误操作如何定位审计日志。

## 4.11 日志不能泄露敏感信息

不要记录：

Telegram Bot Token；
Supabase service role key；
TON minter private key / mnemonic；
session secret；
原始 initData；
完整钱包 proof payload；
用户 IP 明文；
支付原始敏感字段。

可以记录 hash、requestId、orderId、userId、chargeId、txHash、risk_event_id。

## 4.12 第六阶段建议验收标准

正式上线前至少满足：

1. `ops.admin_users` 有至少 1 个 super admin 和 1 个非 super admin 测试账号。
2. 所有 admin 写操作产生 `ops.admin_audit_logs`。
3. `payments.telegram_webhook_events` 有真实 webhook 记录。
4. 对账中心可以跑 `payment_fulfillment`、`ledger_balance`、`mint_queue`、`wallet_sync`。
5. 风控中心可以展示和处理 `ops.risk_events`。
6. 概率池可以 draft、preview、publish、archive。
7. 已发布概率版本不可编辑。
8. 支付异常可以后台重试发货。
9. 市场统计可以定时刷新。
10. 关键 feature flags 可以后台紧急关闭。
11. Sentry 或外部日志已接入 production。
12. 数据库性能 advisor 的高频外键索引问题已处理。
