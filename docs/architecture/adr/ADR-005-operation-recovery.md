# ADR-005：未知结果恢复

## 决定

操作具有 `pending`、`succeeded`、`failed`、`unknown` 四个持久状态。同步业务在提交事务内形成终态；支付和 Mint 允许由外部事实推进。随机结果在第一次裁决时持久化，不得重新生成。

前端只在用户完成领域确认后生成并保存 UUID，随后立即进入 `confirming`，下一帧进入 `submitting`，但不提前宣告业务成功。网络中断只进入 `unknown`，恢复只查询原操作。默认只锁定同一 `use_case` 的再次提交；`gacha.open` 与 `wheel.spin` 从 `confirming` 开始禁用本领域操作按钮和五个底部导航，经过 `submitting`、`pending`、`unknown` 持续锁定，并在服务端成功或失败结果弹窗中保持锁定，直到用户通过弹窗规定动作处理结果。`inventory.evolve` 从 `confirming` 到 `unknown` 锁定新进化提交和五个底部导航，终态由覆盖界面的专用弹窗处理。弹窗内的结果处理和原操作查询按钮不被这些锁禁用。`me/bootstrap` 返回阻塞操作、待处理支付和 Mint，以便重新进入后继续恢复。

开盒、转盘与进化终态使用服务端 `result_acknowledged_at` 回执。未确认的成功与失败结果和非终态一起在重新进入时恢复；各自的专用轻量查询负责发现晚于首屏事务快照提交的原操作。用户处理结果前，专用 RPC 按当前身份、固定原用例与终态条件原子确认；确认前持续恢复原完整 payload，确认后不再展示。浏览器持久存储不保存或裁决随机结果及资产结果事实。

进化复用同一服务端结果回执列，但只通过进化专用 RPC 确认 `inventory.evolve` 终态。进化专用轻量查询发现晚于首屏快照提交的未确认原操作；成功、随机失败和前置拒绝均在用户执行结果弹窗动作后确认。共享操作注册中心通过独立的进化展示组件、NEW 集成函数和通用发现 Hook 扩展，不复制进化业务字段或结算规则。

操作注册中心只对 `succeeded` 且通过原命令输出 Schema 校验的结果开放对应业务视图。开盒、转盘与进化分别使用领域专用结果组件，只消费各自服务端持久结果；进化的随机失败是已完成结算，仍使用 `inventory.evolve` 的完整输出 Schema，前置拒绝则使用持久错误码与拒绝快照。其余操作继续使用通用状态弹窗，`pending`、`unknown`、Schema 不完整的成功结果不得进入成功视图；持久成功结果的 Schema 不完整时只查询原操作，在完整结果恢复前禁止确认。

已确认展示的终态操作保留 30 天；非终态操作与未确认展示的开盒、转盘或进化终态不得清理。
