# PokePets 项目结构一次性重构规划

## 0. 文档状态

| 项目 | 结论 |
| --- | --- |
| 状态 | 已裁决，可直接作为一次性重构实施基线 |
| 日期 | 2026-07-18 |
| 范围 | Telegram Mini App 前端、Vercel Functions、Supabase Postgres、TON、部署、运维与真实环境验收 |
| 本次交付 | 只产出分析与规划，不修改业务代码、数据库或部署状态 |
| 业务裁判 | `功能说明文档.md` |
| 冲突处理 | 删除或替换与功能说明冲突的旧接口和旧行为；只保留符合功能说明的兼容契约 |
| 实施方式 | 一次性重组并一次性交付全部上线能力，不发布中间结构，不保留新旧两套实现 |

## 1. 结论

当前项目必须调整结构。仓库已经具备 monorepo 外形，但尚未形成稳定的模块边界和唯一事实来源。目录混乱的根因不是文件夹数量不足，而是以下问题同时存在：

1. 业务规则分散在前端静态数据、环境变量、API 入口、共享包和数据库 SQL 中。
2. 根目录 `api/` 的 Vercel 入口承载了大量业务逻辑，API 适配层与应用层没有分离。
3. 前端、服务端和数据库使用不同的业务命名和数据模型，同一功能需要跨多个横向目录查找。
4. API 契约没有 OpenAPI 3.1 单一来源，请求、响应、错误和类型依靠手工重复维护。
5. 数据库同时存在 `migrations`、`rpc`、`rls`、`seed` 四组 SQL，其中只有 migration 链是正式部署入口。
6. 当前实现保留了功能说明已经明确排除的升级、自由改价、历史列表、刷新会话、排行榜和活动配置。
7. 藏品远征和幸运转盘尚未实现，`game` 仍是占位页。

重构后的唯一结构为：

```text
React/Vite Web
    ↓ 只依赖公开 API 契约
3 个 Vercel Function 网关
    ↓ 调用服务端业务模块
垂直业务模块
    ↓ 读取只读视图 / 调用写入 RPC
Supabase Postgres
    ↓
数据库事务、约束、幂等记录、账本与操作结果是最终事实
```

保留 React、Vite、TypeScript、pnpm、Vercel Functions、Supabase Postgres 和 TON 技术基线。不引入 Next.js、Supabase Auth、ORM、前端直连 Supabase 或第二个后端服务。

### 2.2 具体结构问题

#### 前端

- `apps/web/src/features/box/staticBoxes.ts` 在前端保存盲盒价格和折扣。
- `apps/web/src/features/box/staticRewards.ts` 在前端保存旧奖池、旧藏品和错误保底阈值。
- `localUpgradeRules.ts`、`localEvolutionPreviews.ts` 和分解组件内常量在前端复制资产规则。
- `CollectionPage.tsx`、`BoxPage.tsx`、多个 `*.api.ts` 已超过 900 行，页面、协议解析、业务转换和交互状态混在同一文件。
- `game` 目录只有占位页面，功能说明中的远征和转盘没有对应模块。
- `features/tasks` 同时承载任务、签到、邀请、佣金和奖励历史，边界与功能说明不一致。
- `features/wallet` 同时承载钱包验证、NFT 同步、NFT 列表和 Mint，保留了功能说明明确排除的玩家 NFT 列表和历史。

#### 后端

- `api/` 文件同时完成参数兼容、业务定价、风控、数据读取、RPC 调用、响应转换和错误映射。
- `api/boxes/create-open-order.ts` 从环境变量读取旧 Stars 价格，功能说明要求开盒只使用 K-coin 且价格固定为 9/40/120。
- `api/market/buy.ts` 仍以单条 `listing_id` 购买，功能说明要求按模板聚合并严格 FIFO 撮合，买家不能看到单条挂售。
- `api/inventory/evolve.ts` 仍以实例 ID 和前端 expected 值为核心，功能说明要求“模板 + 数量”，并由后端和数据库推导目标、消耗、成功率和保底。
- 登录、钱包、Mint、风控和 cron 中仍存在直接 `.insert()`、`.update()`、`.upsert()`，违反所有写入必须通过数据库 RPC 原子完成的规则。
- `packages/server/src/jobs/ledgerReconcileJob.ts` 超过 3000 行，支付、账本、Mint、钱包和市场对账耦合在单一任务中。
- `packages/server/src/env.ts` 超过 1000 行，大量已废弃业务规则被环境变量化，部署环境能够无审计地改变产品结果。

#### 数据库

- migration 链从初始表结构连续累积到 173 个文件，并包含 rollback、修复、重复应用和旧 phase 命名。
- migration 中出现 351 次 `api` schema 函数创建或替换，最终函数正文难以从目录直接判断。
- `supabase/config.toml` 暴露 `api` 与 `core` schema；目标只允许服务端访问 `api` facade。
- `supabase/rpc`、`supabase/rls` 不属于 Supabase migration 自动部署链，继续保留会形成“文件存在但线上未应用”的假象。
- 当前 schema 仍围绕藏品实例、自由市场价格、排行榜、活动和后台能力组织，与冻结功能说明不一致。

#### 部署与环境

- `vercel.json` 配置了 13 个 cron，其中排行榜、市场统计、活动关闭和 listing 过期不属于当前功能说明。
- `vercel.json` 的开发命令强制启用 mock。
- `.env.example` 中的会话有效期是 86400 秒，功能说明要求 900 秒绝对有效。
- `.env.example` 同时存在 Stars 开盒价格、自由市场价格、邀请佣金、升级开关、功能开关和测试强制结果，均会改变冻结业务结果。
- 当前 117 个 API 文件会造成 Function 数量、构建时间、冷启动和配置维护压力。

| --- | --- | --- |

## 3. 重构约束

1. `功能说明文档.md` 是所有产品规则、金额、概率、状态、页面结果和验收口径的唯一业务来源。
2. 前端立即提供安全的按钮、弹窗、动画、加载或临时选择反馈；前端不得提前确认资产或随机结果。
3. 前端不提交价格、奖励、库存、成功率、最终稀有度、手续费或余额结果，只提交动作、目标标识、数量和幂等键。
4. 服务端重新验证 Telegram 身份、会话、账号状态、权限、参数、业务状态、前置条件、频率和幂等性。
5. 所有新增、修改、删除、状态转换、计数、资产和库存写入只调用 PostgreSQL RPC。
6. 每个资产类操作由一个数据库事务共同完成校验、锁定、扣发、账本、图鉴、任务进度、幂等结果和审计事实。
7. 前端、API、服务端和数据库使用相同的业务域名称。
8. 不保留 mock、demo、placeholder、test-only 强制结果、功能开关或本地业务默认值。
9. 不保留本地功能测试代码；本地只执行格式、lint、类型、构建、契约和数据库结构检查。
10. 测试和生产使用同一 Git Commit 与同一 migration 序列，只允许域名、密钥、项目 ID、TON 网络和外部服务地址不同。

具体执行规则：

1. 所有 workspace package 只通过 `exports` 暴露入口，不允许 `../../packages/*/src/*` 深层导入。
2. `packages/config`、`packages/domain`、`packages/validation`、`packages/api-client` 合并为 `packages/contracts`。
3. `packages/ui` 合并到 `apps/web/src/shared/ui`，因为项目只有一个前端应用。
4. `packages/db-types` 合并到 `packages/server/src/platform/db/generated`，数据库类型不得进入浏览器 bundle。
5. 当前 `contracts/` 移到 `chain/ton/`，避免与 API contracts 名称冲突。
6. ESLint 使用 restricted imports 固化上述边界，TypeScript path alias 只指向公开 package 入口。

## 7. 请求与状态流

### 7.1 普通写操作

```text
用户点击
→ 前端立即打开弹窗/播放无结论动画/禁用重复点击
→ 前端生成 UUID 幂等键
→ API 校验契约
→ 服务端校验访问令牌、normal 状态、权限和频率
→ 服务端只调用一个业务 RPC
→ RPC 在单事务内锁定数据并完成业务裁决
→ RPC 写入资产账本、操作结果、任务/图鉴影响和审计事实
→ API 返回唯一 operation_id 与确定结果
→ 前端刷新受影响 query 并用真实结果回正
```

前端允许乐观展示“处理中”和用户尚未提交的选择，不允许乐观增减 K-coin、Fgems、藏品、资格、保底、库存或奖励。

### 7.2 外部支付与链上操作

Telegram Stars 与 TON 无法放入单个数据库事务，统一使用状态机：

```text
数据库原子创建操作意图和幂等结果
→ 调用 Telegram/TON
→ webhook、用户返回或 cron 只推进原 operation_id
→ 每次推进先锁定操作行并验证允许的状态转换
→ 交付 RPC 原子完成资产/权益/库存/账本
→ 重复事件返回原结果
```

支付窗口、钱包窗口和链上提交均不等于交付成功。结果未知时只查询原 `operation_id`，不创建替代操作。

## 8. API 与 Vercel Functions

### 8.1 三个网关

| Function | 职责 | 路由来源 |
| --- | --- | --- |
| `api/app.ts` | 登录交换、用户查询与全部游戏 API | `/api/*` 默认 rewrite |

### 8.2 契约单一来源

`packages/contracts/src/routes` 中的每个 route 定义以下确定内容：

- method 与公开 path；
- path/query/body/header Zod schema；
- 成功响应 schema；
- 错误 code、HTTP status 和是否可重试；
- 是否需要访问令牌、幂等键和频率限制；
- 所属网关。

同一 route 元数据同时驱动前端类型客户端、服务端运行时校验和 OpenAPI 3.1 生成。仓库不再手写第二份 endpoint 常量或前端 response parser。

下述标准只用于 C2 破坏性重建路由和 C4 全新路由。C1 完整兼容路由由 gateway adapter 按旧 schema 输出，旧字段名不进入服务端内部模型。

标准成功响应：

```json
{
  "data": {},
  "request_id": "uuid",
  "operation_id": "uuid-or-null"
}
```

标准失败响应：

```json
{
  "error": {
    "code": "STABLE_MACHINE_CODE",
    "message": "用户可见文案",
    "retryable": false
  },
  "request_id": "uuid",
  "operation_id": "uuid-or-null"
}
```

C2 和 C4 的 API 字段统一使用 `snake_case`。C1 在公开契约边界继续输出旧字段和旧别名，前端不对 C2/C4 增加 camelCase 或其他别名适配。

### 8.3 会话契约

1. 只有 `POST /api/auth/telegram` 接收 Telegram `initData`。
2. 服务端按 Telegram 官方算法验证 hash、`auth_date`、用户和机器人身份。
3. 登录成功返回 15 分钟绝对有效的短期访问令牌。
4. 前端只在运行时内存保存令牌，通过 `Authorization: Bearer` 发送。
5. 页面刷新或 Mini App 重开重新交换 Telegram 身份。
6. 会话自然过期只自动交换一次；被新登录替换或撤销时不自动恢复。
7. 不创建 refresh token，不使用 Supabase Auth，不保留 logout/refresh API。

### 9.4 库存模型

游戏内藏品不再使用实例 ID。唯一模型为：

```text
inventory.holdings(user_id, template_id, total_quantity)
inventory.reservations(user_id, template_id, operation_id, kind, quantity, status)
```

只读 view 按 reservation 聚合：

```text
总数量 = 可用 + 出售中 + 交易中 + Mint 中 + 远征中
可用数量 = total_quantity - 全部 active reservation 数量
```

同一操作使用唯一 `operation_id` 创建 reservation。成交、Mint 成功、分解和进化在同一事务内减少 holding 并关闭 reservation；下架、Mint 失败和远征领取在同一事务内释放 reservation。约束禁止负数、重复 reservation 和同一数量重复占用。

### 9.5 资产、幂等与并发

1. 资产使用 `bigint` 最小整数单位，余额和数量都有非负约束。
2. 余额表是当前快照，ledger 是不可变审计事实；每次余额变化必须同时写 ledger。
3. 幂等唯一键为 `(actor_user_id, operation_type, idempotency_key)`，同时保存规范化请求 hash。
4. 同键同 hash 返回原结果；同键不同 hash 返回 `IDEMPOTENCY_CONFLICT`。
5. 需要恢复的写操作写入 `ops.operation_results`，状态只允许通过数据库状态机前进。
6. 余额、holding、pity、日计数、listing 和支付订单使用行锁与唯一约束处理并发。
7. 市场买入、开盒十连、转盘十次和任务/图鉴跨域影响在单个 RPC 事务内全部成功或全部回滚。
8. 支付和 Mint 的外部结果通过原 operation 状态机恢复，不宣称跨系统原子事务。

### 9.6 数据库安全

1. Supabase PostgREST 只暴露 `api` schema。
2. Web bundle 不包含 Supabase URL、anon key、service_role 或数据库类型。
3. `anon` 与 `authenticated` 对业务 schema 无表权限、无 view 权限、无函数执行权限。
4. `service_role` 只通过服务端使用。
5. 所有 `SECURITY DEFINER` 函数固定 `search_path`，显式 schema 限定对象。
6. 所有函数先 `REVOKE EXECUTE FROM PUBLIC, anon, authenticated`，再按清单授予 `service_role`。
7. RLS 作为外围拒绝层，业务权限不依赖客户端 JWT 或 RLS 条件。

## 10. 前端结构

### 10.1 每个 feature 的固定形态

```text
feature-name/
├── api.ts                 # 只调用 contracts client
├── queries.ts             # 只读 server state
├── mutations.ts           # 幂等写操作与失效范围

文件不存在职责时不创建空文件。纯展示小组件保留在使用处；只有被两个以上 feature 使用且不含业务规则的组件才进入 `shared`。

### 10.2 状态所有权

| 状态 | 所有者 |
| --- | --- |
| 服务端数据、余额、库存、规则、操作结果 | TanStack Query |

| NEW | 当前前台生命周期内的 feature state，最终获得事实来自服务端 |

├── repository.ts          # 该域只读 view 与 RPC facade
├── errors.ts              # RPC/平台错误到稳定 API code
├── mapper.ts              # 数据库结果到 contract DTO
└── index.ts               # 公开用例
```

服务端规则：

1. HTTP route 只做契约校验、身份上下文、调用 use case 和标准响应。
2. command 只组装服务端可信参数并调用一个业务 RPC，不直接写表。
3. query 只读取 `api` schema 的只读 view/RPC，不访问浏览器传来的 user ID。
4. 当前用户 ID 只来自验证后的访问令牌。
5. 价格、概率、奖励、库存、手续费、保底、资格和账号状态从数据库读取或在 RPC 中推导。
6. 错误映射按数据库稳定 SQLSTATE/错误 code，不解析自然语言 error message。
7. 任务进度和图鉴点亮在产生业务事实的同一 RPC 内写入，不通过延迟 cron 补算。
8. 风控拒绝不改变已确定事实；banned 确认后所有正常业务 route 在进入 command 前统一拒绝。

## 14. 部署结构

1. 根 `package.json` 固定 Node.js `24.x` 与 pnpm 版本，Vercel Project Settings 同步选择 Node.js 24.x。
2. 单一 Vercel Project 同时部署 Vite 静态前端和三个 Functions。
3. 前端只调用同源 `/api`，不启用 CORS，不配置跨域凭据。
4. Supabase 测试项目先应用同一 migration 序列，真实环境验收通过后，生产项目应用相同 migration 文件。
5. Vercel 测试 deployment 与 production deployment 必须指向同一 Git Commit。
6. migration 必须先于依赖新 schema 的 Vercel deployment 生效；由于当前无生产数据，第一次上线直接建立完整 baseline。
7. Telegram webhook 只指向 `api/integrations.ts`，并验证 secret header、update ID 幂等和支付订单归属。
8. 三个 Function 都输出结构化 `request_id`；写操作额外输出 `operation_id`；job 输出 `job_run_id`。
9. Function region 与 Supabase 主区域保持一致；实施前读取真实项目区域并写入 `vercel.json`。如果区域无法确认，停止部署，不使用猜测值。
10. 实施前验证当前 Vercel 计划支持四个 cron 及每 5 分钟频率。如果不支持，停止部署并把实际计划限制交给用户裁决，不静默降低频率。

## 18. 结构验收标准

重构完成必须同时满足：

1. 根目录只保留 3 个 Vercel Function 源文件，不再有 117 个业务入口文件。

13. 仓库没有空业务文件、mock、placeholder、demo、测试强制结果或本地功能测试代码。
14. 同一 Git Commit 和 migration 序列已经在真实测试环境通过全部功能说明验收。

## 19. 技术裁决记录

### ADR-001：保留 React + Vite + TypeScript

状态：已裁决。项目是 Telegram 内嵌 SPA，不需要 SSR；继续使用 React、Vite、TypeScript 和 React Router。

### ADR-002：保留 pnpm monorepo，只保留两个共享包

状态：已裁决。共享包固定为 `contracts` 与 `server`；单应用 UI、config、domain 和 db types 不再各建空壳 package。

### ADR-003：三个 Web-standard Vercel Function 网关

状态：已裁决。使用 Node.js 24 Web API 和 URLPattern；公开路径通过 rewrite 保留，内部 Function 数量降为 3。

### ADR-004：route registry 是 API 契约源

状态：已裁决。Zod route metadata 同时生成前端类型、服务端校验和 OpenAPI 3.1，不维护第二份 endpoint/type/parser。

### ADR-005：不引入 ORM

状态：已裁决。服务端使用 Supabase service_role 读取 `api` facade 并调用 RPC；写事务由 PostgreSQL 实现，ORM 不能替代 RPC 原子裁决。

### ADR-006：使用声明式 schema + 全新 baseline migration

状态：已裁决。现有 migration 从未应用，直接重建干净基线；schema 负责可读当前态，migration 负责部署历史。

### ADR-007：数量库存 + reservation

状态：已裁决。游戏内藏品按用户和模板聚合，不使用实例 ID；所有长流程占用统一使用 reservation。

### ADR-008：短期内存访问令牌

状态：已裁决。Telegram 是唯一身份，访问令牌绝对有效 15 分钟，无 Supabase Auth、refresh token、logout 或长期 cookie。

### ADR-009：真实环境功能验收

状态：已裁决。不保留本地功能测试代码；本地只做静态门禁，功能结果在独立真实测试环境与生产真实环境验证。

### ADR-010：用户钱包提交 Mint 交易

状态：已裁决。服务端在数据库原子锁定 1 个藏品并分配唯一 NFT 编号后，签发一次性 Mint permit；用户已验证钱包提交包含该 permit 的 TON 交易并承担网络费；合约验证 permit、nonce、合集和接收钱包。服务端不代替用户钱包发送 Mint 交易，成功后只按原交易和不可变元数据快照完成库存扣除。
