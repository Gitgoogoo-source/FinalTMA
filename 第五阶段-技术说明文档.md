## 1. 第五阶段范围定义

### 1.1 Telegram Stars 支付模块

完成正式 Telegram Stars 支付接入：创建 invoice、打开支付、接收 payment webhook、根据 `successful_payment` 发货。Telegram Stars 数字商品支付使用 `XTR`，必须在收到 `successful_payment` 后才交付数字商品，`pre_checkout_query` 只用于校验订单并确认是否允许支付。

不做：前端直接判断支付成功、前端直接发货、前端保存 Bot Token。

### 1.2 支付订单与支付流水模块

基于现有 `payments.star_orders`、`star_invoices`、`star_payments`、`telegram_webhook_events`、`star_refunds`、`payment_disputes` 补齐正式状态机、异常状态、幂等索引、后台查询字段。

不做：把 Telegram Stars 当成用户可自由转移的内部余额；Stars 只作为外部支付凭证。

### 1.3 Telegram webhook 模块

实现 `api/telegram/webhook.ts`，处理 `pre_checkout_query`、`successful_payment`、退款 / 支付争议相关事件的落库、校验、幂等、错误记录。

不做：在 webhook 未落库前直接执行业务发货。

### 1.4 支付成功后开盒发货模块

使用现有 `api.gacha_process_paid_order` 作为正式发货事务入口。支付成功后必须能追溯到：`star_order`、`star_payment`、`draw_order`、`draw_results`、`item_instances`、`currency_ledger`、`user_pity_states`、`referral` 事件。

不做：前端生成抽卡结果。

### 1.5 TON Connect 钱包连接模块

前端接入 TON Connect UI，后端保存公开地址和钱包 app 信息。

不做：保存私钥、助记词、服务端读取用户钱包私钥。

### 1.6 钱包签名 / ton_proof 验证模块

实现后端 challenge 创建、ton_proof 验证、防重放、防过期、绑定钱包。

不做：仅凭前端传入地址就标记 verified。

### 1.7 NFT Mint 队列模块

基于现有 `onchain.mint_queue`、`wallet_enqueue_mint`、`onchain_mark_mint_success`、`onchain_mark_mint_failed` 实现异步 Mint 队列。

不做：用户请求直接阻塞等待链上确认。

### 1.8 链上交易同步模块

基于 `onchain.transactions` 记录 Mint 交易提交、确认、失败、重试、超时。

不做：交易失败时直接丢弃上下文。

### 1.9 链上 NFT 同步模块

实现钱包 NFT 同步任务，写入 `onchain.wallet_sync_jobs`、`wallet_nft_snapshots`，并尝试关联 `onchain.nft_items` 与游戏内 `item_instances`。

不做：前端直接写入 NFT 快照。

### 1.10 前端支付与钱包 UI 模块

改造开盒支付流程、钱包入口、Mint 状态、链上同步状态、刷新恢复状态。

不做：前端执行任何资产发放、NFT mint、支付确认逻辑。

### 1.11 后台运营管理模块

至少完成支付订单列表、异常支付列表、webhook 事件、退款 / dispute、Mint 队列、失败重试、链上交易、用户钱包查询。

不做：不受审计的后台余额修改或强制发货。

### 1.12 测试、监控、部署与回滚模块

完成支付、webhook、发货、钱包、Mint、链上同步、E2E、回滚开关、feature flag、日志、对账。

不做：未测试直接上线真实 Stars 支付。

## 4. 技术栈说明

前端: React + Vite + TypeScript
状态 / 请求: TanStack Query + Zustand
表单 / 校验: zod
后端:	Vercel Functions Node 24
数据库:	Supabase Postgres + RPC + RLS
核心事务:	Postgres RPC / SECURITY DEFINER 函数
支付:	Telegram Stars webhook，后端验签和幂等处理
钱包:	TON Connect，只保存公开地址，私钥绝不进系统
测试:	Vitest + Playwright + pgTAP / Supabase local tests
包管理:	pnpm workspace
部署:	Vercel + Supabase

## 8. Telegram Stars 支付全流程

Telegram Stars 数字商品支付应使用 `XTR`。官方流程是创建 invoice，等待 `pre_checkout_query`，调用确认接口，然后等待 `successful_payment`，保存 `telegram_payment_charge_id`，最后交付数字商品；不能在 `pre_checkout_query` 阶段发货。([Telegram][1])

### 8.1 完整流程

1. 用户点击单抽 / 十连。
2. 前端调用 `/api/boxes/create-open-order`。
3. 后端校验 session、盲盒、价格、库存、活动状态。
4. 后端调用 `api.gacha_create_order` 创建 `draw_order` 和 `star_order`。
5. 后端生成 invoice payload。
6. 后端调用 Telegram invoice 创建能力。
7. 后端写入 `payments.star_invoices`。
8. 前端打开 Telegram Stars 支付。
9. Telegram 发送 `pre_checkout_query` 到 `/api/telegram/webhook`。
10. webhook 先写入 `payments.telegram_webhook_events`。
11. 后端校验订单是否存在。
12. 后端校验金额是否一致。
13. 后端校验状态是否可支付。
14. 后端校验盲盒是否可用。
15. 后端校验库存是否可用。
16. 后端返回 pre_checkout 确认。
17. Telegram 发送 `successful_payment`。
18. webhook 事件先落库。
19. 后端根据 `telegram_payment_charge_id` 做唯一幂等。
20. 后端写入 `payments.star_payments`。
21. 后端更新 `star_orders.status='paid'`。
22. 后端调用数据库事务 `api.gacha_process_paid_order`。
23. RPC 创建 `draw_results`。
24. RPC 创建 `inventory.item_instances`。
25. RPC 更新 `gacha.user_pity_states`。
26. RPC 返还 100 K-coin。
27. RPC 写入 `economy.currency_ledger`。
28. RPC 更新 `draw_orders.status='completed'`。
29. RPC 更新 `star_orders.status='fulfilled'`。
30. 前端轮询或查询订单结果。
31. 前端展示抽卡结果和资产变化。

### 8.2 支付状态机

| 状态                  | 说明                                      | 可转移到                                   | 触发条件                                        |
| --------------------- | ----------------------------------------- | ------------------------------------------ | ----------------------------------------------- |
| `created`             | 内部支付订单已创建。                      | `invoice_created`、`expired`、`failed`     | `gacha_create_order` 成功。                     |
| `invoice_created`     | Telegram invoice 已创建。                 | `precheckout_checked`、`expired`、`failed` | invoice 创建成功。                              |
| `precheckout_checked` | Telegram pre_checkout 已校验通过。        | `paid`、`expired`、`failed`                | 收到并确认 pre_checkout_query。                 |
| `paid`                | 收到 successful_payment，支付流水已落库。 | `fulfilling`、`disputed`、`refunded`       | `star_payments` 插入成功。                      |
| `fulfilling`          | 正在执行开盒发货事务。                    | `fulfilled`、`failed`                      | webhook 调用发货 RPC。                          |
| `fulfilled`           | 发货完成。                                | `refunded`、`disputed`                     | draw_results、item_instances、ledger 全部完成。 |
| `failed`              | 支付创建、校验或发货失败。                | `fulfilling`、`refunded`、`disputed`       | 可重试失败或不可重试错误。                      |
| `expired`             | 用户未支付且订单过期。                    | 无，或重新创建订单                         | 超过 expires_at。                               |
| `refunded`            | 已退款。                                  | `disputed`                                 | 后台退款处理成功。                              |
| `disputed`            | 用户或平台争议处理中。                    | `refunded`、`fulfilled`、`failed`          | 用户提交 dispute 或后台标记。                   |

当前数据库 `payments.star_orders` 已有部分状态，但没有完整覆盖 `fulfilling`、`disputed`，且当前 precheckout 状态名为 `precheckout_ok`。建议在 migration 中统一到上述状态，或在 API 层做兼容映射。

## 9. TON Connect 与钱包验证全流程

### 9.1 完整流程

1. 前端显示钱包连接按钮。
2. 用户通过 TON Connect 连接钱包。
3. 前端拿到公开地址。
4. 前端请求 `/api/wallet/challenge`。
5. 后端创建 challenge，并记录到 `core.wallet_proofs` 或 `core.wallet_connect_sessions`。
6. 前端请求钱包签名 / ton_proof。
7. 前端提交 account + proof 到 `/api/wallet/proof`。
8. 后端验证 challenge、domain、timestamp、address、signature、防重放。
9. 后端调用 `api.wallet_save_verified_address`。
10. 后端保存 `core.user_wallets`。
11. 后端更新 `verified_at`。
12. 前端显示已连接 / 已验证。
13. 用户可以断开连接。
14. 后端记录 `disconnected_at` 和 status。
15. verified wallet 用于 Mint 和链上 NFT 同步。

### 9.2 钱包状态机

| 状态                   | 说明                                   | 可转移到                                                     | 触发条件                  |
| ---------------------- | -------------------------------------- | ------------------------------------------------------------ | ------------------------- |
| `not_connected`        | 用户未连接钱包。                       | `connecting`                                                 | 点击连接钱包。            |
| `connecting`           | TON Connect 连接中。                   | `connected_unverified`、`not_connected`                      | 用户确认或取消。          |
| `connected_unverified` | 前端拿到公开地址，但后端未验证 proof。 | `verified`、`invalid_proof`、`expired_proof`、`disconnected` | 提交 proof。              |
| `verified`             | 后端验证 proof 成功。                  | `disconnected`、`expired_proof`                              | 保存 wallet verified_at。 |
| `disconnected`         | 用户断开钱包。                         | `connecting`、`not_connected`                                | 用户点击断开。            |
| `invalid_proof`        | proof 签名、domain、address 校验失败。 | `connected_unverified`、`disconnected`                       | 用户重新请求 proof。      |
| `expired_proof`        | challenge 或 proof 已过期。            | `connected_unverified`、`disconnected`                       | 重新生成 challenge。      |

数据库当前 `core.user_wallets.status` 只包含 `connected`、`disconnected`、`revoked`，可以用 `verified_at` 派生 `connected_unverified` / `verified`，减少 migration；如果要直接持久化这些状态，需要扩展 check constraint。

## 10. NFT Mint 队列全流程

### 10.1 完整流程

1. 用户选择可 Mint 的藏品。
2. 前端调用 `/api/wallet/mint`。
3. 后端校验钱包已 verified。
4. 后端校验藏品属于用户。
5. 后端校验藏品未挂售、未分解、未锁定、未 Mint。
6. 后端校验 Collection active。
7. 后端锁定藏品。
8. 后端创建 `onchain.mint_queue`。
9. 后端写入 `inventory.item_instance_events`。
10. 前端显示 queued。
11. 后台 worker / cron 拉取队列。
12. worker 设置 processing。
13. worker 生成 metadata snapshot。
14. worker 提交链上 Mint 交易。
15. worker 写入 `onchain.transactions`。
16. worker 设置 submitted / confirming。
17. 交易同步任务查询链上交易状态。
18. Mint 成功后调用 `api.onchain_mark_mint_success`。
19. 创建 / 更新 `onchain.nft_items`。
20. 更新 `inventory.item_instances.nft_mint_status='minted'`。
21. 绑定 NFT item address。
22. Mint 失败时调用 `api.onchain_mark_mint_failed`。
23. 失败可重试，超过次数进入 manual_review。
24. 用户前端可以查看 Mint 状态。

### 10.2 Mint 状态机

| 状态            | 说明                              | 可转移到                                 | 触发条件                   |
| --------------- | --------------------------------- | ---------------------------------------- | -------------------------- |
| `queued`        | Mint 请求已入队。                 | `processing`、`cancelled`                | 用户创建 Mint 请求。       |
| `processing`    | worker 已领取并准备提交链上交易。 | `submitted`、`failed`、`retrying`        | worker 加锁处理。          |
| `submitted`     | 链上交易已提交。                  | `confirming`、`failed`                   | 获得 tx hash / query id。  |
| `confirming`    | 等待链上确认。                    | `minted`、`failed`、`retrying`           | 交易查询中。               |
| `minted`        | Mint 成功，NFT 地址已绑定。       | 无                                       | 链上确认成功。             |
| `failed`        | Mint 失败。                       | `retrying`、`manual_review`、`cancelled` | 不可立即确认或链上失败。   |
| `retrying`      | 等待下一次重试。                  | `processing`、`manual_review`            | next_attempt_at 到达。     |
| `manual_review` | 超过重试或异常状态，需人工处理。  | `retrying`、`cancelled`、`minted`        | 后台人工操作。             |
| `cancelled`     | Mint 被取消。                     | 无                                       | 仅在未提交链上交易前允许。 |

当前 `onchain.mint_queue.status` 只有 `queued`、`processing`、`minted`、`failed`、`cancelled`，需要扩展以支持 `submitted`、`confirming`、`retrying`、`manual_review`。

## 14. 回滚和故障处理方案

| 故障场景                          | 用户影响                       | 排查位置                                                                          | 临时处理                                                   | 修复方案                                                         | 是否需要补偿                           |
| --------------------------------- | ------------------------------ | --------------------------------------------------------------------------------- | ---------------------------------------------------------- | ---------------------------------------------------------------- | -------------------------------------- |
| Telegram webhook 收不到           | 用户支付后无法自动发货。       | Telegram webhook 配置、Vercel logs、`telegram_webhook_events`。                   | 暂停新支付；保留订单查询。                                 | 修复 webhook URL / secret / Bot 配置后重放或人工处理 paid 订单。 | 已支付未发货需要补发。                 |
| pre_checkout_query 校验失败       | 用户无法完成支付。             | webhook logs、`star_orders.error_message`、盲盒状态、金额。                       | 提示用户重试；暂停异常盲盒。                               | 修复价格、库存、订单过期逻辑。                                   | 未支付通常不需要补偿。                 |
| successful_payment 收到但发货失败 | 用户已支付但未收到藏品。       | `star_payments`、`star_orders.status`、`draw_orders`、RPC error。                 | 暂停新支付；后台重试发货。                                 | 修复 RPC 或数据问题，调用幂等发货。                              | 是，必须补发或退款。                   |
| 重复 webhook                      | 理论无用户影响。               | `telegram_webhook_events.update_id`、`star_payments.telegram_payment_charge_id`。 | 返回成功但不重复处理。                                     | 确保 unique 约束和幂等逻辑。                                     | 否，除非已重复发货。                   |
| 用户支付成功但前端没显示结果      | 用户误以为失败。               | `/api/boxes/result`、前端 pending order、webhook 状态。                           | 提示“支付已成功，发货处理中”；提供刷新按钮。               | 修复轮询 / 状态恢复逻辑。                                        | 若已发货不需要；若未发货需要补发。     |
| 数据库事务失败                    | 订单卡在 paid / fulfilling。   | RPC logs、Supabase logs、`star_orders.error_message`。                            | 暂停发货开关，后台重试。                                   | 修复事务、索引、约束或数据异常。                                 | 视是否支付成功；通常需要。             |
| ledger 不一致                     | 余额显示错误。                 | `currency_ledger`、`user_balances`、reconciliation runs。                         | 暂停相关资产操作。                                         | 通过冲正 ledger 修复，不直接改历史 ledger。                      | 可能需要。                             |
| Mint 卡在 processing              | 用户无法完成 NFT Mint。        | `mint_queue`、worker logs、`onchain.transactions`。                               | 暂停 Mint worker；把超时任务转 retrying 或 manual_review。 | 修复 worker 锁、链上提交或状态同步。                             | 通常不涉及资产补偿，但需解除或继续锁。 |
| 链上交易失败                      | Mint 失败。                    | TON API、`onchain.transactions`、tx hash。                                        | 进入 retrying 或 manual_review。                           | 修复交易参数、gas、Collection 配置。                             | 视是否扣费 / 锁定时间决定。            |
| TON API 不可用                    | Mint / NFT sync 延迟。         | TON API provider logs、worker logs。                                              | 暂停 Mint worker 和 wallet sync。                          | 切换 provider 或延后重试。                                       | 一般不需要，除非用户长期卡住。         |
| 用户钱包 proof 验证失败           | 用户无法绑定钱包。             | `wallet_proofs.error_message`、domain、timestamp、signature。                     | 提示重新连接。                                             | 修复 manifest、domain、proof 验证逻辑。                          | 不需要。                               |
| 错误配置导致盲盒价格异常          | 用户可能支付错误金额。         | `blind_boxes`、`box_price_rules`、`star_orders`。                                 | 立即暂停支付和相关盲盒。                                   | 修复价格配置；对异常订单退款或补偿。                             | 需要逐单判断。                         |
| 需要紧急关闭支付                  | 用户不能新开盒支付。           | `ops.feature_flags`。                                                             | 关闭 `FEATURE_STARS_PAYMENT_ENABLED`。                     | 修复后重新开启；已支付订单继续补发。                             | 已支付未发货需要。                     |
