## 第 6 步：实现 successful_payment 落库与幂等

### 目标

在收到 Telegram `successful_payment` 后，先落库支付流水，再基于 `telegram_payment_charge_id` 做唯一幂等，防止重复发货。

### 需要先检查的现有文件 / 表

| 类型   | 路径 / 表                                                                            |
| ------ | ------------------------------------------------------------------------------------ |
| API    | `api/telegram/webhook.ts`                                                            |
| Server | `packages/server/src/payments/telegramStars.ts`                                      |
| DB     | `payments.star_payments`、`payments.star_orders`、`payments.telegram_webhook_events` |
| RPC    | `api.gacha_process_paid_order`                                                       |

### 需要新增或修改的内容

| 文件 / 表                          | 责任                                            |
| ---------------------------------- | ----------------------------------------------- |
| `api/telegram/webhook.ts`          | 处理 successful_payment 分支。                  |
| `payments.star_payments`           | 保存 charge id、XTR 金额、payload、raw update。 |
| `payments.star_orders`             | 状态转为 `paid`，再进入发货。                   |
| `payments.telegram_webhook_events` | 记录处理状态、错误原因。                        |

### 后端开发工作

successful_payment 处理步骤：

1. 校验 webhook secret。
2. 原始 update 先写 `telegram_webhook_events`。
3. 解析 `successful_payment`。
4. 获取 `invoice_payload`、`telegram_payment_charge_id`、`currency`、`total_amount`。
5. 通过 invoice payload 查 `star_orders`。
6. 校验 `currency='XTR'`、金额一致、订单未 fulfilled。
7. 尝试插入 `star_payments`。
8. 如果 `telegram_payment_charge_id` 已存在，判定为重复 webhook，返回成功但不发货。
9. 更新 `star_orders.status='paid'`。
10. 进入下一步发货事务。

### 数据库开发工作

确认 `star_payments.telegram_payment_charge_id` unique。
新增或确认 `star_payments.invoice_payload` 索引。
新增或确认 `star_payments.star_order_id` 索引。
支付落库失败不能调用发货 RPC。
`star_orders.status` 需要支持 `paid`、`fulfilling`、`fulfilled`。

### 前端开发工作

无需直接处理 webhook。
支付等待 Sheet 可轮询 `/api/boxes/result` 或新增订单状态 API。
重复刷新不会影响发货。

### 安全和幂等要求

`telegram_payment_charge_id` 是支付幂等核心。
不要只依赖 invoice payload 判断支付成功。
successful_payment 中金额不一致必须进入异常状态。
重复 webhook 必须返回 200，避免 Telegram 继续重试造成噪音，但不能重复发货。

### 测试要求

成功支付落库。
重复 `telegram_payment_charge_id`。
重复 `update_id`。
金额不一致。
payload 不存在。
订单已 fulfilled 后再次收到 successful_payment。
插入 `star_payments` 成功但发货失败。

### 验收标准

- [ ] successful_payment 可以写入 `star_payments`。
- [ ] `telegram_payment_charge_id` 唯一。
- [ ] 重复 successful_payment 不重复发货。
- [ ] 金额不一致进入异常状态。
- [ ] webhook 事件处理状态可查询。
- [ ] 原始 update 可审计。
