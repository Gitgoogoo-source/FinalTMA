## 第 13 步：实现 Mint worker / cron 与链上交易状态同步

### 目标

异步处理 `onchain.mint_queue`，提交链上 Mint 交易，写入 `onchain.transactions`，查询确认状态，成功后绑定 NFT Item 地址。

### 后端开发工作

Mint worker 流程：

1. 校验 cron secret。
2. 校验 `FEATURE_MINT_WORKER_ENABLED` 和 `TON_MINT_ENABLED`。
3. 拉取 `queued` / `retrying` 且 `next_attempt_at <= now` 的任务。
4. 将任务状态置为 `processing`。
5. 读取 Collection、metadata、目标 wallet。
6. 使用服务端 minter key 提交链上交易。
7. 写入 `onchain.transactions`。
8. 将 `mint_queue` 状态改为 `submitted` / `confirming`。
9. 交易确认后调用 `api.onchain_mark_mint_success`。
10. 失败时调用 `api.onchain_mark_mint_failed`，按 retry 策略进入 retrying 或 manual_review。

### 数据库开发工作

扩展 `mint_queue.status`。
`onchain.transactions.related_type='mint_queue'`、`related_id=mint_queue.id`。
`tx_hash` unique。
`query_id` 建索引。
`mint_queue.attempt_count` 和 `max_attempts` 控制重试。
超过最大重试进入 `manual_review`。
成功后：

- `onchain.nft_items` 插入 item。
- `inventory.item_instances.nft_mint_status='minted'`。
- `inventory.item_instances.minted_nft_item_id` 写入。
- `inventory.item_instances.status='minted'` 或保持 available + minted 标记，需要产品确认。
- release / consume mint lock 的规则需要明确。

### 安全和幂等要求

Mint worker 必须 cron secret 鉴权。
worker 每次领取任务要加锁，防止并发 worker 重复提交。
链上交易提交成功但 API 超时时，不可盲目再次提交，必须先查询 `query_id` / tx 状态。
`onchain_mark_mint_success` 必须幂等。
同一个 item_instance 只能绑定一个 nft_item。

### 测试要求

queued → processing。
processing 失败 → retrying。
超过重试 → manual_review。
提交成功 → confirming。
确认成功 → minted。
重复 worker 并发抢任务。
交易 hash 已存在。
成功回写时重复调用。
TON API 不可用。
