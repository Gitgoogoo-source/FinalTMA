你不要只按“功能”审，也不要只按“文件”审。
不要从页面开始审。应该先从数据库、权限、资产、交易这些底层开始。
因为：

```txt
UI 错了，最多体验差。
资产错了，项目会亏钱。
权限错了，用户可以盗资产。
支付错了，用户可以白嫖。
交易错了，市场会乱。
```

最适合的审查方式是：

> **先按业务功能审查，再按技术层横向审查，最后按安全/资金/数据库一致性做专项审查。**

原因很简单：
代码最容易出现的问题不是单个文件语法错，而是**跨文件、跨模块、跨数据库、跨前后端的业务逻辑断裂**。

---

# 一、不要只按文件审查

比如按文件审查：

```txt
审查 apps/web/src/pages/TaskPage.tsx
审查 api/tasks/claim.ts
审查 supabase/rpc/task_claim_reward.sql
```

这种方式的优点是细，但缺点很明显：

```txt
可能只看单个文件，觉得每个文件都没问题。
但实际业务流程可能是断的。
```

例如：

```txt
前端按钮调用 /api/tasks/claim
/api/tasks/claim 调用 task_claim_reward RPC
RPC 更新 user_balances
但是前端余额刷新接口没重新拉取
```

单个文件看都没错，但整个功能体验就是错的。

所以，**文件审查适合做第二层，不适合做第一层。**

---

# 二、不要只按功能审查

只按功能审查，比如：

```txt
审查登录功能
审查开盒功能
审查交易功能
审查签到功能
```

这个方向是对的，但还不够。

因为有些问题不是某个功能内部的问题，而是全局架构问题，比如：

```txt
权限校验不统一
RLS 有漏洞
积分流水不完整
API 没做幂等
交易和开盒共用资产表但状态设计冲突
错误处理风格混乱
前端缓存导致余额显示不准
```

这些问题跨越多个功能，只靠单个功能审查容易漏掉。

所以，**功能审查是主线，但必须配合横向专项审查。**

---

# 三、最推荐的审查方式：三层审查法

你的项目建议按这三层来审：

```txt
第一层：按业务功能审查
第二层：按技术链路审查
第三层：按风险专项审查
```

---

# 第一层：按业务功能审查

这是最重要的一层。

每个功能都按“完整业务闭环”审查，而不是只看页面或接口。

例如审查“签到功能”，不要只审查签到页面，而是要审查整个链路：

```txt
前端签到按钮
→ API 请求
→ session 校验
→ 参数校验
→ RPC / 数据库事务
→ user_signins 表
→ currency_ledger 流水
→ user_balances 余额
→ 防重复领取
→ RLS 权限
→ 前端刷新余额和任务状态
→ 错误提示
```

你应该按这个格式审查每个功能：

```txt
1. 功能入口在哪里？
2. 前端调用了哪些 API？
3. API 调用了哪些 server 方法 / RPC？
4. 数据库修改了哪些表？
5. 是否有事务？
6. 是否有幂等？
7. 是否会重复发奖？
8. 是否会越权？
9. 是否会出现余额和流水不一致？
10. 前端状态是否正确刷新？
11. 失败时是否有正确提示？
12. 是否有测试覆盖？
```

---

# 第二层：按技术链路审查

功能审查完以后，再按技术层横向审查。

这个阶段不是问“某个功能能不能用”，而是问：

```txt
整个项目的技术结构是否稳定？
有没有架构层面的雷？
```

建议按这些技术链路审查：

---

## 1. 前端审查

重点看：

```txt
页面路由是否清晰
组件是否重复造轮子
状态管理是否混乱
API 调用是否统一封装
loading / error / empty 状态是否完整
用户余额、库存、任务状态是否会显示旧数据
按钮是否有防重复点击
Telegram WebApp 初始化是否稳定
移动端布局是否适配
```

尤其要查：

```txt
用户点击一次按钮，是否可能发出多次请求？
请求失败后，前端是否误显示成功？
余额是否只是前端本地加减，而没有以后端为准？
```

---

## 2. API 审查

重点看：

```txt
所有 API 是否统一使用 requireSession
是否统一校验 method
是否统一 parse body
是否统一参数校验
是否统一错误返回
是否暴露 service role key
是否有 rate limit
是否有 idempotency key
是否调用数据库 RPC，而不是前端直接写表
```


这些都涉及资产、奖励、支付或交易，属于高风险接口。

---

## 3. 数据库审查

重点看：

```txt
表结构是否满足业务闭环
字段命名是否统一
状态机是否完整
索引是否合理
唯一约束是否防止重复领取
外键是否完整
金额字段是否使用 integer / bigint，避免 float
created_at / updated_at 是否完整
ledger 是否能追溯每一次资产变化
```


核心原则：

```txt
所有资产变化必须有 ledger。
所有奖励领取必须防重复。
所有交易必须在数据库事务里完成。
所有库存状态变化必须可追溯。
```

---

## 4. RPC / 数据库事务审查

你的项目用了 Supabase RPC，这一层非常关键。

重点看：

```txt
RPC 是否 SECURITY DEFINER
是否正确设置 search_path
是否检查当前用户身份
是否防止越权操作别人的资产
是否使用事务保证原子性
是否用唯一约束防重复
是否正确处理并发
是否返回前端需要的数据
```


---

## 5. RLS 审查

这是 Supabase 项目最容易埋雷的地方。

重点看：

```txt
用户是否只能读自己的数据
用户是否不能直接修改余额
用户是否不能直接修改库存
用户是否不能伪造任务完成
用户是否不能伪造邀请关系
用户是否不能直接插入 currency_ledger
管理后台表是否只允许 admin 操作
RPC 是否绕过 RLS 时仍有内部权限校验
```

高风险表原则：

```txt
余额表：前端只读，禁止直接写
流水表：前端只读自己的，禁止直接写
库存表：前端只读自己的，禁止直接写状态
挂单表：只能操作自己的挂单
任务进度表：前端最多只读，不应直接写奖励状态
邀请奖励表：前端只读自己的，不应直接插入奖励
```

---

# 第三层：风险专项审查

这是项目最重要的一层。

因为代码最容易把“看起来能跑”的功能写出来，但忽略商业化风险。

你应该单独做以下专项审查。

---

## 1. 资产安全专项审查

重点查：

```txt
是否能刷 K-coin
是否能刷 Fgems
是否能重复领取奖励
是否能重复开盒
是否能重复购买
是否能绕过库存锁定
是否能出售不属于自己的藏品
是否能购买自己的挂单刷流水
是否能取消已成交订单
是否能通过前端参数修改价格、稀有度、奖励数量
```

一句话：

```txt
凡是影响钱、积分、库存、奖励的逻辑，都不能相信前端。
```

---

## 2. 并发专项审查

重点查这些场景：

```txt
用户连续点击两次开盒
用户同时领取两次签到奖励
两个用户同时购买同一个挂单
用户取消挂单的同时，另一个用户购买
用户领取图鉴奖励时重复请求
用户升级/合成时重复提交
```

如果没有数据库唯一约束、事务、行锁或幂等机制，就很容易出问题。

---

## 3. 支付专项审查

重点查：

```txt
订单是否有 pending / paid / failed / refunded 状态
支付回调是否验签
支付金额是否以后端订单为准
是否防止重复回调重复发货
Stars 支付后是否有退款风险处理
TON 支付是否确认链上到账
订单是否有过期时间
订单是否绑定 user_id
订单是否绑定商品内容快照
```

尤其注意：

```txt
支付成功 ≠ 前端说成功
必须以后端确认成功为准。
```

---

## 4. Telegram 登录专项审查

重点查：

```txt
initData 是否按 Telegram 官方规则验签
auth_date 是否检查过期
hash 是否正确排除后再计算
bot token 是否只在服务端
session token 是否有过期时间
是否能伪造 telegram_id
是否能用别人的 telegram_id 登录
```

这个功能一旦错，后面所有 user_id 都不可信。

---

## 5. 经济系统专项审查

重点查：

```txt
积分发放是否有来源
积分消耗是否有流水
余额 before / after 是否能对账
手续费是否正确入账
交易失败是否回滚
开盒失败是否退款
合成失败是否按规则扣除
邀请返佣是否有上限
分红是否按快照计算，而不是实时余额计算
```

推荐你检查 ledger 是否至少包含：

```txt
user_id
currency
delta
available_before
available_after
locked_before
locked_after
reason
ref_type
ref_id
idempotency_key
created_at
```

如果没有 before / after，对账会困难很多。

---

# 四、最终你应该采用的审查模式

建议你采用这个组合：

```txt
主线：按功能
补充：按技术层
收尾：按文件
```


具体做法：

```txt
第一轮：功能审查
登录、开盒、交易、成长、任务、邀请、支付逐个审。

第二轮：横向审查
数据库、RLS、RPC、API、前端状态、测试、安全、并发逐个审。

第三轮：文件审查
检查是否有废弃文件、重复代码、未使用文件、命名混乱、路径错误、环境变量错误。
```



---

# 五、最适合的结论

这个项目应该这样审：

```txt
第一优先：按业务功能闭环审查。
第二优先：按数据库 / RLS / RPC / API / 前端状态做横向审查。
第三优先：按文件做清理性审查。
```

不要一开始就按文件审。
文件审查看起来细，但容易漏掉真正的业务雷。

你这个项目最危险的不是 UI bug，而是：

```txt
资产重复发放
库存状态错乱
交易并发漏洞
支付重复发货
RLS 越权
Telegram 登录伪造
任务/签到/邀请奖励被刷
```

所以审查重点应该放在：

```txt
钱、积分、库存、奖励、权限、支付、并发、数据库事务。
```

一句话总结：

> **先按功能闭环审，再按技术层横扫，最后按文件清理。不要只看单个文件，要顺着“用户操作 → API → RPC → 数据表 → 前端刷新”的完整链路审查。**
