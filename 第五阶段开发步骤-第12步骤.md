## 第 12 步：实现 Mint 请求 API 与 Mint 队列入队

### 目标

用户选择可 Mint 的藏品后，请求后端创建 Mint 队列；后端锁定藏品，写入 `onchain.mint_queue`，返回队列状态。

### 后端开发工作

Mint 入队流程：

1. `requireSession` 获取用户。
2. 校验 feature flag。
3. 校验钱包 verified。
4. 校验 item instance 属于用户。
5. 校验藏品状态 `available`，不在市场，不在分解，不在合成，不在 active lock。
6. 校验 `nft_mint_status='not_minted'` 或允许 failed retry。
7. 校验 Collection active。
8. 调用 `api.wallet_enqueue_mint`。
9. 返回 `mint_queue_id`、status、预计处理状态。

### 数据库开发工作

`wallet_enqueue_mint` 必须在一个事务中：

- 锁定 item instance。
- 创建 inventory lock。
- 创建 mint_queue。
- 更新 item nft_mint_status。
- 写 `item_instance_events` 为 `mint_queued`。
- 防止同一 item 重复入队。

需要为 active mint queue 增加唯一约束或 partial unique index。

### 安全和幂等要求

非本人藏品不能 Mint。
挂售中藏品不能 Mint。
分解 / consumed 藏品不能 Mint。
重复 `idempotency_key` 返回同一 queue。
同一 item 不能多个 active queue。
前端不能指定任意 collection 覆盖后端默认规则，除非后端允许并校验。

### 测试要求

未连接钱包。
未验证钱包。
非本人藏品。
挂售中藏品。
已分解藏品。
重复 Mint 请求。
Collection 未配置。
正常入队。
入队后库存被锁定。

### 验收标准

- [ ] 用户可以对可 Mint 藏品创建 Mint 请求。
- [ ] Mint 请求进入 `onchain.mint_queue`。
- [ ] item instance 状态进入 minting / queued。
- [ ] active inventory lock 创建成功。
- [ ] 重复请求不会创建多个 active queue。
- [ ] 非本人 / 挂售中 / 分解 / 已 Mint 藏品不能 Mint。
- [ ] 前端能展示 queued 状态。
